1.InnoDB体系架构

1.InnoDB存储引擎
    1.如上图示,InnoDB存储引擎属于Mysql体系架构中存储引擎接口中的一种存储引擎;InnoDB是Mysql默认的存储引擎,也是真正执行SQL语句的部件：
    2.InnoDB存储引擎是第一个完整支持ACID事务的Mysql存储引擎,其特点是:行锁设计、支持MVCC、支持外键、提供一致性非锁定读（也就是默认读操作不会产生锁）、同时被设计用来最有效的利用内存和CPU;
    3.InnoDB存储数据是基于磁盘存储的,且其记录是按照页的方式进行管理;
    4.InnoDB体系架构
        从上图可以看出,InnoDB存储引擎由内存池、一些重要的后台线程和存储文件三部分组成的;下面分别对这三部分内部结构和作用进行说明;

2.1.InnoDB存储引擎内存池

InnoDB存储引擎内存池
    内存池是由多个内存块组成的,根据分工以及存储文件的不同分为缓冲池（Buffer_Pool）、重做日志缓冲（redo_log_buffer）、额外内存池（additional_pool）;内存池的主要作用是：
        1.维护所有进程/线程需要访问的多个内部数据结构
        2.缓存磁盘上的数据,方便快速读取,同时在对磁盘文件修改之前进行缓存
        3.缓存重做日志（redo log）
1.缓冲池(Buffer_Pool)
    1.缓冲池的作用：
        1.首先InnoDB作为MySQL的存储引擎,数据是基于在磁盘的,并将其中的记录按照页（page）的方式进行管理,如果数据库读取和写入操作每次都要从磁盘上读取经过磁盘IO效率（众所周知的慢）是非常低的,因此InnoDB提供了缓冲池（Buffer_Pool）通过内存的速度来弥补磁盘速度慢对数据库性能造成的影响;借助于缓冲池后它的工作逻辑是：
            1.首先将数据库文件按页（每页16K）读取到缓冲池,然后按最近最少使用（LRU）的算法来保留在缓冲池中的缓存数据;
            2.当Mysql进行读取操作时,首先将从磁盘读到的页存放在缓冲池中,下一次读取相同的页时,首先判定是否存在缓冲池中,如果有就是被命中直接读取,没有的话就从磁盘中读取;
            3.当Mysql进行修改/写入操作时,首先修改缓冲池中的页（修改后,该页即为脏页）,然后在以一定的频率刷写到磁盘上,需要说明的是这里的刷写机制不是每页在发生变更时触发的,而是通过checkpoint机制刷写到磁盘,在InnoDB中checkpoint有两种,后文会详细说明;
        2.可以看出缓冲池的大小直接影响着数据库的整体性能,可以通过配置参数innodb_buffer_pool_size来设置;而且在Mysql优化中,Buffer_Pool缓冲池的大小是必须优化项,一般建议设置innodb-buffer-pool-size的值为服务器总可用内存的80%;
        3.而且从InnoDB1.0.x版本开始,允许有多个缓冲池实例;每个页根据哈希值平均分配到不同缓冲池实例中,这样做的好处是减少数据库内部的资源竞争,增加系统的并发处理能力;
    2.缓冲池的组成：
        缓冲池中缓存的数据页类型还包括：数据页和索引页、undo页、插入缓冲、自适应哈希索引、InnoDB的锁信息、数据字典信息、 两次写等;
        1.数据页和索引页:data page和index page
            页（Page）是Innodb存储的最基本结构,也是Innodb磁盘管理的最小单位,与数据库相关的所有内容都存储在Page结构里;Page分为几种类型,而数据页和索引页就是其中最为重要的两种类型;
        2.插入缓冲:insert buffer 
            1.什么是插入缓冲？
                1.由于每张表不可能只有一个聚集索引,更多情况下一张表有多个非聚集（且不唯一）的辅助索引;这种情况下,在进行插入操作时,数据页的存放还是按照主键进行顺序存放,但是由于B+树的特性决定了非聚集索引插入的离散性,那么对于非聚集的索引叶子节点的插入就不再是顺序的,而是需要离散的访问辅助索引页,那么这种离散型的随机读取操作必然会导致了插入操作的性能下降;
                2.于是InnoDB设计了insert_buffer来优化对于非聚簇索引的插入或者更新操作;insert_buffer是一种特殊的数据结构（B+ tree）,而且它并不是缓存的一部分,而是和数据页一样都是物理页的组成部分;
                3.使用insert_buffer后,对于非聚集索引的插入或更新操作,不是每一次直接插入到索引页中,而是先判断插入的非聚集索引页是否在缓冲池中,如果在,则直接插入,如果不再,则先放入一个insert_buffer对象中;这样会给数据库一个假象：
                    数据库会认为这个非聚集的索引已经插入到叶子节点,而实际上是放在另一个位置了,然后再以一定的频率和情况进行insert_buffer和非聚簇索引叶子节点的合并merge操作,这时通常能将多个插入合并到一个操作中（因为在同一个索引页中）,于是就可以大大提高了对于非聚集索引插入的性能;
            2.insert buffer需要同时满足两个条件
                1.索引必须是非聚簇索引
                2.索引不是唯一的：
                    因为在插入缓冲时,数据库并不去查找索引页来判断插入记录的唯一性。如果去查找的话,又会涉及到离散读的情况,insert_buffer失去意义
        3.两次写:double write buffer
            1.insert_buffer带给InnoDB存储引擎的是性能上的提升,doublewrite（两次写）带给InnoDB存储引擎的是数据页的可靠性;
            2.部分失效
                当数据库正在从内存向磁盘写一个数据页时,数据库宕机,从而导致这个页只写了部分数据,这就是部分写失效,它会导致数据丢失;这时是无法通过重做日志恢复的,因为重做日志记录的是对页的物理修改,如果页本身已经损坏,重做日志也是无法恢复的;
            3.InnoDB double write双写体系架构
                从第二步解释可知道,在部分写失效的情况下,我们在应用重做日志之前,需要原始页的一个副本,而两次写就是为了解决这个问题的,如上图是其原理架构图：
                1.两次写需要额外添加两个部分：
                    1.内存中的两次写缓冲（doublewrite buffer）,大小为2MB
                    2.磁盘上共享表空间中连续的128页,大小也为2MB
                2.原理是这样的：
                    1.当刷新缓冲池脏页时,并不直接写到数据文件中,而是先拷贝至内存中的两次写缓冲区;
                    2.接着从两次写缓冲区分两次写入磁盘共享表空间中,每次写入1MB
                    3.待第2步完成后,再将两次写缓冲区写入数据文件
                这样就可以解决第二步提到的部分写失效的问题,因为在磁盘共享表空间中已有数据页副本拷贝,如果数据库在页写入数据文件的过程中宕机,在实例恢复时,可以从共享表空间中找到该页副本,将其拷贝覆盖原有的数据页,再应用重做日志就可以了;
        4.自适应哈希索引:adaptive hash index
            1.哈希（hash）是一种非常快的查找方法,在一般情况下这种查找的时间复杂度为O(1),即一般仅需要一次查找就能定位数据;而B+树的查找次数,取决于B+树的高度,在生产环境中,B+树的高度一般为3~4层,所以需要3~4次的查询;
            2.InnoDB存储引擎会根据访问的频率和模式监控对表上各索引页的查询,通过对热点页建立哈希索引来提高查询效率;自适应hash索引是指：
                如果观察到建立哈希索引可以带来速度提升,则建立哈希索引,称之为自适应哈希索引(Adaptive Hash Index, AHI)
            3.自适应哈希索引AHI是通过缓冲池的B+树页构造而来,因此建立的速度很快,而且不需要对整张表构建哈希索引;但是AHI有一个要求:对这个页的连续访问模式必须是一样的,也就是说其查询的条件（where）必须完全一样,而且必须是连续的;比如：
                对于(a,b)这样的联合索引页,其访问模式可以是下面情况：
                    where a=xxx
                    where a =xxx and b=xxx
                    访问模式一样是指查询的条件是一样的,如果使用上面这两种查询语句的查询方式则InnoDB存储引擎不会对该页构造AHI;
        5.InnoDB锁信息:lock info
            InnoDB存储引擎会在行级别上对表数据进行上锁;不过InnoDB也会在数据库内部其他很多地方使用锁,从而允许对多种不同资源提供并发访问;数据库系统使用锁是为了支持对共享资源进行并发访问,提供数据的完整性和一致性;(关于锁详见其他锁的相关文章)
        6.数据字典信息:data dictionary
            1.InnoDB有自己的表缓存,可以称为表定义缓存或者数据字典;当InnoDB打开一张表,就增加一个对应的对象到数据字典;
            2.数据字典是对数据库中的数据、库对象、表对象等的元信息的集合;在Mysql中数据字典信息内容就包括表结构、数据库名或表名、字段的数据类型、视图、索引、表字段信息、存储过程、触发器等内容;MySQL INFORMATION_SCHEMA库提供了对数据局元数据、统计信息、以及有关MySQLserver的访问信息（如数据库名或表名、字段的数据类型和访问权限等）;该库中保存的信息也可以称为Mysql的数据字典;
    3.InnoDB管理缓冲池(Buffer_Pool)的方式:LRU List、Free List和Flush List
        1.InnoDB是通过LRU算法管理存放了各种类型页的缓冲池的;它是把最频繁最近使用的页在LRU_list的前端,最少使用的页在LRU_list的尾端;当缓冲池中不能存放新读取到的页时,将释放LRU_list尾端的页;
        2.free list
            LRU_list用于管理已经读取的页;但数据库刚刚启动时,LRU列表是空的,这时页（page）都在Free_list中的,当需要从缓冲池中分页时,先从Free_List中查找是否有可用的空闲页,如果有则将该页从Free_List中删除,放入LRU_list中;否则根据LRU淘汰LRU末尾的页,将该内存空间分配给新的页;
        3.flush list
            LRU_list中被修改后的页称为脏页,也就是缓冲池中的页和磁盘的页数据产生了不一致：这时数据库会通过Checkpoint机制将脏页刷新回磁盘,而Flush_list中的页即为脏页列表;需要注意的是:脏页既存在于LRU_list中,也存在于Flush_list中;LRU_list用来管理缓冲池中页的可用性,Flush用来管理将页刷新回磁盘,二者不影响;
2.重做日志缓冲(redo_logz的redo_log_buffer)
    1.buffer pool无法保证事务的持久性和redo_log的作用
        1.首先我们知道InnoDB是支持事务的,buffer_pool的存在虽然达到了提高数据库读写性能的目的,但也带来了数据丢失的很大风险,风险的详细说明：
            1.对InnoDB的任何修改操作都会首先在buffer_pool的page上进行,然后这样的页面将被标记为dirty并被放到专门的flush_list上,后续将由master thread或专门的刷脏线程阶段性的将这些页面写入磁盘上;这样的好处是避免每次写操作都操作磁盘导致大量的随机IO,阶段性的刷脏可以将多次对页面的修改merge成一次IO操作,同时异步写入也降低了访问的时延;
            2.但是如果在dirty page还未刷写到磁盘时,Mysql因故障非正常关闭,这些放在buffer_pool中的修改操作将会丢失,如果写入操作正在进行,甚至会由于损坏数据文件导致数据库不可用,无法保证InnoDB事务的持久性;
        2.redo_log的作用
            1.因此为了避免上述问题的发生,Innodb将所有对页面的修改操作会先写入redo_log中,再写入buffer_pool中;这样即使Mysql因故障非正常关闭,在重新启动时也会根据redo_log进行恢复操作;
            2.所以redo_log存在的目的就是保证Mysql数据安全性,也就是实现了InnoDB事务的持久性;
    2.重做日志缓冲:redo_log_buffer
        1.redo日志是由redo_log_buffer和redo_log_file两部分组成的,redo_log_buffer重做日志缓冲就是放在InnoDB内存池中的,redo_log_file是放在磁盘上的;
        2.innodb是支持事务的存储引擎;因此在事务进行时,会先将修改操作记录在redo_log_buffer中,然后再事务commit时,会调用内核中的fsync将redo_log_buffer刷写到磁盘的redo_log_file中;所以磁盘的性能在一定程度上也决定了事务提交的性能;
        3.redo_log_buffer一般不需要很大,默认大小为8MB,默认会按照一定的频率将其刷写到磁盘上的redo_log_file,但在以下三种情况下也会对重做日志缓冲进行刷写操作：
            1.后台线程Master Thread每一秒将重做日志缓冲刷新到重做日志文件
            2.每个事物提交时,会将重做日志缓冲刷新到重做日志文件
            3.当重做日志缓冲剩余空间小于1/2时,重做日志缓冲刷新到重做日志文件
3.额外内存池(additional_pool)
    在InnoDB存储引擎中,对内存的管理是通过一种称为内存堆的方式进行的;在对一些数据结构本身的内存进行分配时,需要从额外的内存池中进行申请;比如：
        分配了缓冲池,但是每个缓冲池中的帧缓冲还有对应的缓冲控制对象,这些对象记录以
        一些诸如LRU、锁、等待等信息,而这个对象的内存需要从额外的内存池中申请;

4.InnoDB的关键特性
    Mysql InnoDB是通过下面这些关键特性实现了更好的新能和更高的特性：
        1.插入缓冲（insert buffer）
        2.两次写（Double write）
        3.自适应哈希索引（adaptive hash index）
        4.异步io（Async IO）
        5.刷新领接页（Flush Neighbor Page）
        前3个特性上面已经说明,其余两个是：
    1.异步IO：Async IO
        1.同步IO(sync IO) :
            是指每进行一次IO操作,此次操作结束才能继续接下来的操作;但是如果用户发需要等待出一条索引扫描的查询,那么这条SQL查询语句可能需要扫描多个索引页,也就是需要进行多次的IO操作;在每扫描一个页并等待期完成再进行下一次的扫描是没有必要的;
        2.异步IO(Async IO)
            是指用户可以在发出一个IO请求后立即再发出另一个IO请求,当全部IO请求发送完毕后,等待所有IO操作的完成,这就是AIO;
    2.刷新领接页
        当刷新一个脏页时,innodb会检测该页所在区（extent）的所有页,如果是脏页,那么一起进行刷新;这样通过AIO将多个IO写入操作合并为一个IO操作,在传统机械磁盘下有着显著优势;

2.2.Checkpoint技术

1.为了避免发生数据丢失的问题,当前事务数据库普遍采用了WAL（Write-ahead logging,预写式日志）策略,即当事务提交时,先写重做日志redo_log,再修改Buffer_Pool中的数据页;当由于宕机导致数据丢失时,通过重做日志完成数据的恢复实现事务ACID中持久性;
2.而CheckPoint技术是用来解决如下几个问题：
    1.缩短数据库恢复时间
        当数据库发生宕机时,数据库不需要重做所有的日志,因为checkpoint之前的数据页都已经刷写到磁盘上了,只需要对checkpoint之后的页进行恢复就可以大大缩短了恢复时间;
    2.缓冲池不够用时,将脏页刷新到磁盘
        当缓冲池不够用时,根据LRU算法会移除最近最少使用的页,若此页是脏页,那么需要强制执行checkpoint将脏页刷回磁盘;
    3.重做日志不可用时,刷新脏页
        1.由于事务数据库对重做日志的设计都是循环使用的,在Mysql中redo_log则是对ib_logfile0和ib_logfile1两个重做日志文件进行循环覆盖使用;
        2.而重做日志不可用时是指做日志的这部分不可以被覆盖,因为这部分日志还需要使用（可能用于恢复）,此时就必须强制执行checkpoint,将缓冲池中的数据页少刷新到当前重做日志的位置;
3.checkpoint的目的是将缓冲池中的脏页刷回到磁盘,不同之处在于刷新多少到磁盘？每次从哪里读取脏页？什么时间触发checkpoint？
    在InnoDB存储引擎内部有两种checkpoint,分别为:
    1.Sharp Checkpoint
        1.Sharp_Checkpoint发生在数据库关闭时将所有的脏页刷写到磁盘,这是默认的工作方式;相关参数是：innodb_fast_shutdown=1
        2.Sharp Checkpoint不适用于数据库运行时
            如果数据库运行时使用Sharp_Checkpoint,那么数据库的可用性会受到很大影响;而Fuzzy Checkpoint则可以在数据库运行进行刷写;
    2.Fuzzy Checkpoint
        Fuzzy_Checkpoint进行页的刷写,只刷新一部分脏页,而不是刷新所有的脏页回磁盘;InnoDB发生Fuzzy_Checkpoint的情况分为以下几种：
        1.Master Thread Checkpoint
            异步刷新,每秒或每10秒从缓冲池脏页列表刷新一定比例的页回磁盘;异步刷新时InnoDB存储引擎可以进行其他操作,用户查询线程不会受阻
        2.FLUSH_LRU_LIST Checkpoint
            1.InnoDB存储引擎需要保证LRU列表中差不多有100个空闲页可供使用;在InnoDB 1.1.x版本之前,用户查询线程会检查LRU列表是否有足够的空间操作;如果没有,则根据LRU算法溢出LRU列表尾端的页,如果这些页中有脏页,需要进行checkpoint,因此称为flush_lru_list checkpoint;
            2.InnoDB 1.2.x开始,这个检查放在了单独的进程（Page Cleaner）中进行,好处在于：
                1.减少master Thread的压力 2.减轻用户线程阻塞。
                相关设置参数：
                innodb_lru_scan_dept：控制LRU列表中可用页的数量,该值默认1024
        3.Async/Sync Flush Checkpoint
            1.指的是重做日志不可用的情况,而此时脏页是从脏页列表中选取的;额外补充说明的是：InnoDB存储引擎是通过LSN（Log Sequence Number）来标记版本,LSN是8字节的数字;内存池中的每个页有LSN,重做日志中会记录LSN,checkpoint有LSN;
            2.如下图所示,若将已经写入到重做日志redo_log中的LSN记为redo_lsn,将已经刷写到磁盘最新页的lsn记为Checkpoint_lsn,则：
                Checkpoint_age=redo_lsn-checkpoint_lsn
                async_water_mark = 75% * total_redo_file_size
                sync_water_mark = 90% * total_redo_file_size
        4.Dirty Page too much Checkpoint
            脏页太多导致InnoDB存储引擎强制Checkpoint来保证缓冲池有足够可用的页,默认为75%;相关参数：
                innodb_max_dirty_pages_pct = 75 
                #表示缓冲池中脏页的数量占75%时,强制checkpoint

2.3.InnoDB存储引擎内部的主要后台线程

InnoDB是多线程的模型,不同线程负责不同的任务;后台线程的主要作用是负责刷新内存池中的数据,保证缓冲池中的内存缓存的是最新的数据,此外将已修改的数据文件刷新到磁盘文件;同时,保证在数据库发生异常的情况下,InnoDB能恢复到正常状态;后台线程根据功能不同分为master Thread、IO Thread、Purge Thread、Page Cleaner Thread;
1.Master Thread
    1.Master_Thread是InnoDB中最核心的后台线程,主要负责将缓冲池中的数据异步刷写到磁盘上保证数据一致性,数据包括：脏页、合并插入缓存、undo页的回收等;
    2.Master thread在主循环中,分两大部分操作:每秒钟的操作和每10秒钟的操作
        1.每秒一次的操作
            1.日志缓冲刷新到磁盘：
                即使这个事务还没有提交（总是）,这就可以解释为什么再大的事务commit时都很快;
            2.合并插入缓冲
                合并插入并不是每秒都发生,InnoDB会判断当前一秒内发生的IO次数是否小于5,如果是,则系统认为当前的IO压力很小,可以执行合并插入缓冲的操作
            3.至多刷新100个InnoDB的缓冲池的脏页到磁盘(可能): 
                这个刷新100个脏页也不是每秒都在做,InnoDB引擎通过判断当前缓冲池中脏页的比例（buf_get_modified_ratio_pct）是否超过了配置文件中innodb_max_drity_pages_pct参数（默认是90,即90%）,如果超过了这个阈值,InnoDB引擎认为需要做磁盘同步操作,将100个脏页写入磁盘;
        2.每10秒一次的操作
            1.刷新100个脏页到磁盘(可能)： 
                InnoDB引擎先判断过去10秒内磁盘的IO操作是否小于200次,如果是,认为当前磁盘有足够的IO操作能力,即将100个脏页刷新到磁盘;
            2.合并至多5个插入缓冲(总是)： 
                此次的合并插入缓冲操作总会执行,不同于每秒操作时可能发生的合并操作;
            3.将日志缓冲刷新到磁盘(总是)： 
                InnoDB引擎会再次执行日志缓冲刷新到磁盘的操作,与每秒发生的操作一样
            4.删除无用的undo页(总是)： 
                当对表执行update、delete操作时,原先的行会被标记为删除,但是为了一致性读的关系,需保留这些行版本的信息,在进行10S一次的删除操作时,InnoDB引擎会判断当前事务系统中已被删除的行是否可以删除,如果可以,InnoDB会立即将其删除;InnoDB每次最多删除20个Undo页;
            5.产生一个检查点（checkpoint）
2.IO threads
    1.在InnoDB存储引擎中大量使用了异步IO来处理写IO请求以提高数据库的性能,IO_Thread的工作主要是负责这些IO请求的回调;IO Thread有4类,分别为:
        write、read、insert buffer和log IO thread
    2.线程数量可以通过参数进行调整;
        5.6以后的版本可以通过innodb_write_io_threads和innodb_read_io_threads来限制读写线程,而在5.6版本以前,只有一个参数innodb_file_io_threads来控制读写总线程数;
3.Purge threads
    1.由于InnoDB存储引擎对undo的管理采用回滚段（rollback-segment，简称Rseg）的方式来维护undo_log的并发写入和持久化;而且undo_log中记录的undo操作数量是有限的,所以undo_log需要定时清理的;
    2.purge threads的作用就是负责回收不再需要的undo页;旧版本Mysql中是在master_thread主线程中完成purge操作的,后来为了减轻master_thread的工作提高cpu使用率以及提升存储引擎的性能,于是使用专门的purge_threads线程进行purge操作;而且从innodb1.2版本开始,可以通过innodb_purge_threads选项指定多个purge线程并发回收提高undo回收速度; 
        innodb_purge_threads=N  #purge操作每10s进行一次
4.Page Cleaner Thread
    Page Cleaner Thread是InnoDB1.2.X版本中新引入的;它是用来是帮助Master_Thread刷新脏页到磁盘上的线程,以减轻Master_Thread的工作以及对于用户查询线程的阻塞,进一步提高InnoDB存储引擎的性能

2.4.InnoDB存储文件

InnoDB存储引擎中的文件包括重做日志文件、表空间文件;
1.表空间文件
    1.在InnoDB中存储数据都是按表空间进行存放的,默认为共享表空间,存储的文件即为共享表空间文件（ibdata1）,该文件初始大小为10M,并且按照参数innodb_autoextend_increment=8M的步进自动增长;设置innodb_data_file_path参数后,所有基于innoDB存储引擎的表的数据都会记录到该共享表空间;
    2.在生产环境使用Mysql时,很可能会遇到ibdata1文件很大的问题;那是因为ibdata1中存放的是数据库的表数据,如缓存、索引等,所以随着数据库越来越大,该文件会由参数innodb_autoextend_increment=8M的步进自动增长,时间长了该文件会变得很大,清理的办法只能是：
        先使用mysqldump做一个逻辑全量备份,然后停止Mysql并删除所有数据库、ib_logfile*、ibdata1*文件,当重新启动Mysql后将会创建一个新的共享表空间,然后再把全量备份恢复即可;
    3.如果开启了InnoDB的参数innodb_file_per_table=ON,则每个基于INnoDB的表使用单独的表空间,独立表空间的命名规则为：表名.ibd,也就是：
        数据和索引：放在tb_name.ibd中
        格式定义：放在tb_name.frm
        这些单独的表空间文件仅存储该表的数据、索引和插入缓冲bitmap等信息,其余信息还是存放在默认的表空间(ibdata1)中;
2.重做日志文件
    详见Mysql事务总结一文章;
