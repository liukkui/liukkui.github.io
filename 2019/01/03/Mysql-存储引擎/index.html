Mysql-存储引擎

发表于 2019-01-03 | 分类于 mysql | 评论数：
本文字数： 6.2k | 阅读时长 ≈ 6 分钟

1.Mysql存储引擎

1.存储引擎也称为表类型
    存储引擎是MySQL服务的核心组件,它规定了数据存储时的存储结构;在Mysql中,存储引擎是插件式的;由于同一个数据库中的不同的表可以使用不同的存储引擎,所以存储引擎是表级别的概念,存储引擎也被称为"表类型",每张表可以使用不同的存储引擎类型;
2.Mysql中最常用的存储引擎为InnoDB与MyISAM;从Mysql5.5.5开始,Mysql中的默认存储引擎类型为InnoDB;由于InnoDB是99.9%的Mysql用户都会使用的默认存储引擎,所以在Mysql系列文章中的没有明确说明时,默认都是针对InnoDB来说明的,比如:索引、事务等;这两种最常用的两种存储引擎的特性对比：
    MYISAM：支持表级锁、不支持行级锁、不支持事务、不支持外检约束、支持全文索引、表空间文件相对小
    INNODB：支持表级锁、行级锁、支持事务、支持外键、不支持全文索引、表空间文件相对较大
    这里只是简略说明InnoDB与MyISAM的区别,下文会详细再进行说明;
3.Mysql中的其他存储引擎类型：
    1.CSV：将CSV文件(以逗号分隔字段的文本文件)作为MySQL表文件;
    2.Memory
        内存存储引擎,速度快,但是一旦断电数据将会丢失,支持hash索引,支持表级锁,常用于临时表
    3.Performance_Schema
        从mysql5.5之后,多出了Performance_Schema数据库,Performance_Schema数据库中的表的表类型均为Performance_Schema,此数据库用于存储与数据库的性能相关的信息,用户无法创建使用这种存储引擎的表,但是dba可以通过Performance_Schema数据库中的信息进行性能分析,PERFORMANCE_SCHEMA数据库服务启动后此库中将不断的收集数据,mysql停机后此库中的表将不存在数据,类似于linux中的/proc目录;
    4.Archive
        创建此种表类型的表往往用于存储归档信息、安全审计信息、历史信息等,创建数据仓库时可能会用到此种表类型,使用archive表类型的表只支持select和insert操作,不能更新和删除操作,支持行级锁;
    5.BLACKHOLE
        黑洞存储引擎接受但不存储数据,检索结果总是返回一个空集;该功能可用于分布式数据库设计,数据自动复制,但不是本地存储;类似于linux中的/dev/null; 
    6.MRG_MyISAM
        此引擎也被称为MERGE存储引擎,如果一些myisam表的表结构完全相同,可以将这些MyISAM表合并成的一张MRG_MYISAM虚拟表;适用于VLDB(Very Large Data Base)环境,比如数据仓库;
    7.Federated联合
        利用federated引擎可将本地数据表映射至远程MySQL数据表,从而就可以解决应用程序中繁多的跨机器连接数据库问题,其实federated相当于一个访问其他远程mysql server的代理接口,它通过创建一个到远程mysql server的客户端连接,通过FEDERATED引擎创建的表只是在本地有表定义文件,数据文件则存在于远程数据库中;

2.InnoDB、MyISAM和Memory的区别

InnoDB与MyISAM的最大区别就是是否支持事务!Mysql的事务是非常非常重要的逻辑;下面就对这常用的三种存储引擎进行说明;

2.1.InnoDB存储引擎

1.InnoDB存储引擎特点
    1.支持事务,适合处理大量短期事务 
    2.支持行级锁(锁的颗粒度)
        锁是实现事务隔离性重要基础之一,行级锁意味着一旦两个线程争用同一个资源时,它就需要对数据进行加锁,而且加锁的力度是行row级的；
    3.读写阻塞与事务隔离级别相关 
    4.可缓存数据和索引
    5.支持聚簇索引(数据和索引放在一起称为聚簇索引)
    6.崩溃恢复性更好(由于支持事务,可使用事务的redo log进行恢复)
    7.支持MVCC(多版本并发控制机制)高并发 
        1.InnoDB采用了MVCC来支持较高并发;而且事务的4种隔离级别中,MVCC只在RC:读已提交和RR:可重复读两个隔离级别下工作;
        2.最重要的是并且基于间隙锁策略防止InnoDB事务隔离级别RR幻读的出现;(隔离级别原理文章中会详细说明)
    8.支持外键
    9.从MySQL5.5后支持全文索引 
    10.从MySQL5.5.5开始为默认的数据库引擎
    11.支持热备、冷备、温备、逻辑备份和物理备份  
2.InnoDB的数据库文件
    1.默认情况下所有InnoDB表的数据和索引放置于同一个表空间文件中;也就是Mysql数据目录下的ibdata文件;
    2.如果开启了InnoDB的参数innodb_file_per_table=ON,则每个基于INnoDB的表使用单独的表空间,独立表空间的命名规则为：表名.ibd,每张表有两个文件：
        tb_name.ibd：存放数据文件(数据和索引)
        tb_name.frm：表格式定义文件
            frm是format:也就是每张表都有字段每个字段都有自己的格式
        ~]# ll /mysql/mysql3/hellodb/
            -rw-rw---- 1 mysql mysql  8630 Nov  7 23:40 coc.frm
            -rw-rw---- 1 mysql mysql 98304 Nov  7 23:41 coc.ibd
        说明：
            1.正是由于此参数的作用让InnoDB的众多高级特性才得以实现,比如单表备份(导入导出)功能；
            2.这些单独的表空间文件仅存储该表的数据、索引和插入缓冲bitmap等信息,其余信息还是存放在默认的表空间(ibdata1)中;
            3.InnoDB表的大小只受限于操作系统文件的大小,一般最大为2GB
3.InnoDB的关键特性
    1.插入缓冲(insert buffer)
    2.两次写(Double write)
    3.自适应哈希索引(adaptive hash index)
    4.异步io(Async IO)
    5.刷新领接页(Flush Neighbor Page)
    这些关键特性会在InnoDB存储引擎架构系列文章中进一步说明;

2.2.MyISAM存储引擎

1.MyISAM存储引擎特点
    1.不支持事务(和InnoDB最大的区别,所以很少人使用MyISAM引擎)
    2.表级锁定
        表级锁,由于即使是只改表中的一行数据他也需要把整张表锁住,在高并发场景下,锁竞争是常态,锁的颗粒度过于粗糙,不支持高并发业务场景;
    3.读写相互阻塞,写入不能读,读时不能写 
    4.只缓存索引 
    5.不支持外键约束 
    6.不支持聚簇索引(数据和索引是分开存放的)
    7.读取数据较快,占用资源较少 
    8.不支持MVCC（多版本并发控制机制）高并发 
    9.崩溃恢复性较差
        由于不支持事务,就不能使用事务日志进行数据恢复;只能通过手工或者自动修复,很大可能会造成数据丢失;
    10.是MySQL5.5.5前默认的数据库引擎 
2.MyISAM存储引擎适用场景：
    只读(或者写较少)、表较小(可以接受长时间进行修复操作)
3.MyISAM的数据库文件
    MyISAM的数据库文件是将数据和索引分开放在不同的文件中,保证了安全性,每张表有三个文件：
        tbl_name.frm 表结构信息(Format)
        tbl_name.MYD 数据信息(MYData)
        tbl_name.MYI 索引信息(MYIndex)
    ~]# ll /mysql/mysql3/mysql/
        -rw-rw---- 1 mysql mysql   9582 Nov  7 23:40 db.frm
        -rw-rw---- 1 mysql mysql    880 Nov  7 23:40 db.MYD
        -rw-rw---- 1 mysql mysql   5120 Nov  7 23:40 db.MYI

2.3.Memory存储引擎

Memory存储引擎为了提高数据的访问速度,将数据存在内存,每一个表实际上和一个磁盘文件关联;文件是frm;特点是：
    1.支持的数据类型有限制,比如：不支持TEXT和BLOB类型,对于字符串类型的数据,只支持固定长度的行,VARCHAR会被自动存储为CHAR类型;
    2.支持的锁粒度为表级锁;所以在访问量比较大时,表级锁会成为Memory存储引擎的瓶颈;
    3.由于数据是存放在内存中,一旦服务器出现故障,数据都会丢失
    4.查询的时候,如果有用到临时表,而且临时表中有BLOB、TEXT类型的字段,那么这个临时表就会转化为MyISAM类型的表,性能会急剧降低;
    5.默认使用hash索引
    6.如果一个内部表很大,会转化为磁盘表;

3.管理存储引擎(查看表类型)

1.查看mysql支持的所有存储引擎类型
    MySQL [(none)]> show engines;
2.查看当前Mysql默认的存储引擎 
    MySQL [(none)]> show variables like '%storage_engine%';
    +------------------------+--------+
    | Variable_name          | Value  |
    +------------------------+--------+
    | default_storage_engine | InnoDB |
    | storage_engine         | InnoDB |
    +------------------------+--------+
3.设置Mysql默认的存储引擎 
    可以通过set命令设置,为了永久生效,需要写在my.cnf配置文件中
    ~]# vim /etc/my.cnf
        [mysqld]
        default_storage_engine= InnoDB 
4.查看某个库中所有表使用的存储引擎类型
    MySQL [(none)]> show table status from hellodb\G;  #比如hellodb库的所有表
    *************************** 1. row ***************************
               Name: classes
             Engine: InnoDB         #存储引擎类型
            Version: 10
         Row_format: Compact        #行格式,详见InnoDB的行记录格式相关文章
               Rows: 8
     Avg_row_length: 2048
        Data_length: 16384
    Max_data_length: 0
       Index_length: 0
          Data_free: 0
     Auto_increment: 9
        Create_time: 2019-05-07 23:40:51
        Update_time: NULL
         Check_time: NULL
          Collation: utf8_general_ci
           Checksum: NULL
     Create_options: 
            Comment: 
5.查看库中指定表的存储引擎
    可以使用show table status like 'tb_name'\G;或者show create table tb_name;命令;比如：
    MySQL[hellodb]> show create table hellodb.coc;
        | Table | Create Table
        | coc   | CREATE TABLE `coc` (
          `ID` int(10) unsigned NOT NULL AUTO_INCREMENT,
          `ClassID` tinyint(3) unsigned NOT NULL,
          `CourseID` smallint(5) unsigned DEFAULT NULL,
          PRIMARY KEY (`ID`)
        ) ENGINE=InnoDB AUTO_INCREMENT=15 DEFAULT CHARSET=utf8 |
6.设置一个表的存储引擎类型
    可以在创建表时设置表的存储引擎类型,也可以先使用默认存储引擎,然后再修改;
        CREATE TABLE tb_name(... ) ENGINE=InnoDB;
        ALTER TABLE tb_name ENGINE=InnoDB; 
    比如：
        MySQL [hellodb]> alter table coc engine=myisam;   
        #将coc表的存储引擎从InnoDB修改成MyISAM
7.每个表都可以任意选择存储引擎吗？
    外键约束（FOREIGN KEY)不能跨引擎使用;Mysql支持多种存储引擎,每一个表都可以指定一个不同的存储引擎;但是要注意:外键约束是用来保证数据的参照完整性,如果表之间需要关联外键,却指定了不同的存储引擎,这些表之间是不能创建外键约束的;所以说存储引擎的选择也不完全是随意的;
