1.Flannel网络插件详解

flannel项目

写在前面：
    这里先说明一下，网络插件都是为pod提供网络的，前文中我们说过k8s中有三种网络，另外两个网络都是由k8s自己管理的，只有pod网络是需要网络插件实现的!


1.Flannel简单说明
    1.Flannel是CoreOS提出用于解决容器集群跨主机通讯的网络解决方案；也就是让集群中的不同节点主机创建的Docker容器或k8s集群的Pod都具有全集群唯一的虚拟IP地址，从而s实现我们前文分析的让不同节点上的容器能够获得"同属一个内网"且"不重复的"IP地址；
        这里再明确说明的是不管是什么网络插件它解决的都是k8s网络要求的基本核心问题：即k8s网络模型要求每一个Pod和pod之间需要直接通过IP通信并且在不经过NAT能互相通信；
    2.Flannel实质上是一种覆盖网络overlay（叠加网络）解决方案；也就是将TCP数据包装在另一种网络包里面进行路由转发和通信，目前已经支持udp、vxlan、host-gw、aws-vpc、gce路由等数据转发方式，默认是VxLAN，关于其他几种"转发模式"后文会一一说明其性能和使用的场景；
    3.Flannel为每个主机分配一个subnet，容器从此subnet中分配IP，这些IP可在主机间路由，容器间无需NAT和端口映射就可以跨主机通讯。Flannel让集群中不同节点主机创建容器时都具有全集群唯一虚拟IP地址，并连通主机节点网络；Flannel虽然对网络要求较高，要引入封装技术，转发效率也受到影响，但是却可以平滑过渡到SDN网络，VXLAN技术可以和SDN很好地结合起来，值得整个网络实现自动化部署，智能化运维和管理，较适合于新建数据中心网络部署.
2.flannel的工作逻辑
    如上图示，左边是flannel官方给你的原理逻辑图，右边是简化版，下文以简化版解释flannel的工作逻辑：Flannel没有使用默认的docker0桥，而是使用一个独立的桥cbr0（cni0），在ifconfig命令中显示为cni0，flannel.1是flannel封装隧道协议的隧道接口桥，任何创建的pod都会自动关联到cni0桥，因此pod获得都是10.244.1.1/255.255.255.0网段的地址即10.244.1.*地址，如果通信目标其他主机上的cni0桥的子网地址：10.244.2.*/255.255.255.0时，如果不是本地cni0的子网地址就会通过内核级的路由被送到flannel.1上，flannel.1会把报文打上隧道报文送到另外一个节点，从而完成跨节点的隧道间协议转发，而且默认是VXLAN协议，比如：
        节点1上的container1需要与container3通信时，container1的报文会先通过cni0接口发现不是本地通信会通过内核级路由转发给flannel.1，Flannel.1.会通过隧道协议（VxLAN、UDP、host-gw）的一种封装协议封装后通过当前主机的物理网卡eth0到达另一主机的物理网卡eth0经过报文解封装后达到另一主机的flannel.1后再一次解封装协议报文后传递给cni0，然后到达container3完成跨节点的容器间通信；在通过flannel.1隧道传输时，可以认为报文的掩码会从24变成16，相当于两个不同主机上的container是同一网段间的容器通信；
2.1.flanneld存在的一个疑问
    在flannel工作逻辑的图中，不知道官方在把默认后端UDP改成VXLAN之后还是把以前的架构图贴出来了，关于图上Flanneld的作用在使用vxlan作为后端分析数据报文时感觉并没经过flanneld，参考网上的文章发现有人也有这个疑问就是：
        使用UDP作为后端网络时，flanned会将flanne.1设备的流量经过自己的处理发送给对端的flanneld;但是在分析vxlan作为后端网络时明显不是这么做的，在vxlan中flanned作用是获取必要的mac地址，ip地址信息，没有直接处理数据流。这里一直不理解它图上报文为何经过flanned...
    后来慢慢的理解：
        如上图示，直到我看到Openshift默认也是使用Flannel网络插件的架构图后，我的理解是官方原图上画的实线（画的实线并不严谨）并不是说数据报文经过flanneld转发，而是说flanneld是flannel的守护进程程序，它负责从远端的ETCD集群同步本地和其它host上的subnet信息并为POD分配IP地址，但它并不直接处理数据流；又找了很多文章发现就是应该这样理解；

3.Flannel的几个重要配置字段参数解释：
    在flannel的kube-flannel.yml中的configmap中有几个重要的网络相关的配置字段参数解释：
    ~]# vim kube-flannel.yml     #这里只列出官方清单中关于configmap的内容
        ...
          cni-conf.json: |
            {
              "name": "cbr0",
              "plugins": [
                {
                  "type": "flannel",
                  "delegate": {
                    "hairpinMode": true,
                    "isDefaultGateway": true
                  }
                },
                {
                  "type": "portmap",
                  "capabilities": {
                    "portMappings": true
                  }
                }
              ]
            }
          net-conf.json: |
            {
              "Network": "10.244.0.0/16",
              "Backend": {
                "Type": "vxlan"
              }
            }
    其中：
        1.cni-conf.json部分是定义插件类型为flannel，从cbr0网卡的网络中为pod分片IP地址；
        2.重要的是net-conf.json中的内容；
            "Network": "10.244.0.0/16"：
                由于它是overlay叠加网络，在前文中说到要实现这种网络逻辑就必须有一个大网段，各个节点的网段都是这个大网段的子网，这里就是这个逻辑；而10.244.0.0/16就是flannel提供的CIDR格式的"大网段"网络地址，而各个节点的cni0（如上文中的10.244.1.0/255.255.255.0）的网段都是这个B类网的一个子网；而calico使用192.168.0.0/16作为"大网段"地址的;
            SubnetLen：默认即可不需要修改
                SubnetLen是指将10.244.0.0/16这个大网段切成子网时，每个子网所用的掩码长度，默认是24为掩码；这也是为什么每个节点的cni0网络都是10.244.0~255.0/24为掩码了；
            SubnetMin：默认即可不需要修改
                SubnetMin是指可分配的子网的最小地址段，比如：10.244.10.0/24意思是cni0的子网默认是从10.244.10.0/24开始使用的，而10.244.0~9.0/24就不能给节点的Pod使用
            SubnetMax：
                SubnetMax和SubnetMin相反，指可分配的子网的最大地址段；
        3.net-conf.json中的Backend
            Backend是定义flannel使用哪一种封装协议的，默认有vxlan、udp、host-gw等，而vxlan是flannel默认的，而且是生产环境必然使用的，不过在开启vxlan时一般也会把vxlan的Directrouting（直接路由）属性打开实现网络性能的最大化，详细说明见下文，在生产环境下我们一般都会修改成如下配置：
            net-conf.json: |
            {
              "Network": "10.244.0.0/16",
              "Backend": {
                "Type": "vxlan"，
                "Directrouting": true    #生产环境会开启直接路由属性
              }
            }
    备注：
        这里说明一下为什么flannel使用10.244.0.0/16作为Pod网络，因为按照子网掩码是24划分，如下集群就可以支持255节点，如果这里假设使用"10.0.0.0/8"作为Pod网络还是按照24位掩码划分就可以支持6w多节点，但这仅仅只是理论上的，毕竟当每个节点只有100个pod，网络带宽也会很拥堵的.
            1.10.244.0.0/16--->
                master: 10.244.0.0./24
                node01: 10.244.1.0/24
                ....
                node255: 10.244.255.0/24
            可以支持255个节点
            2.10.0.0.0/8
                10.0.0.0/24
                ...
                10.255.255.0/24
                可以支持6万多个节点

4.k8s集群使用的flannel作为网络插件时网络带宽说明
    Flannel最好部署在千兆网络下，不要部署在万兆或者百兆网络中；
    flannel之前测试使用百兆网络时集群会出现各种莫名其妙的问题!
    flannel在万兆网络下的性能太差，原因是：
        1.CPU运算受制于网络中断处理单核运算能力等因素，flannel万兆能跑到2G就到顶了；除非买特定网卡搞定UDP Offload，那么flannel还是可以跑在万兆网络的，否则就有性能瓶颈；
        2.这就是为何机器计算要买显卡而不是靠CPU一样的道理，而且UDP Offload最好是物理机，否则还要搞定VMware或Openstack的offload的穿透；

2.Flannel的三种Backend详解

后端和各自协议参考文章

1.Flannel的backend
    1.前面说过Flannel实质上是一种覆盖网络overlay（叠加网络）解决方案，也就是将TCP数据包装在另一种网络包里面进行路由转发和通信，它支持UDP、VxLAN、host-gw、AWS VPC、GCE和Ali Vpc路由等数据转发方式；
    2.而由于flannel是一个应用程序，它的守护进程（flannled）可以管理、修改跨节点的flannel.1接口之间的报文是通过哪种协议进行封装转发的，这种封装方式称为flannel的Backend（后端）；而Flannel的Backend有下面几种类型，用的最多的是前三种；

2.1.Vxlan：
    Vxlan可以理解为是一个二层协议隧道，因为是flannel的默认模式，不过vxlan也有两种模式：
    1.vxlan：
        第一种是vxlan叠加网络模式，利用内核级别的VXLAN来封装host之间传送的包;
    2.Directrouting：
        Directrouting是vxlan的一种特殊属性；虽然vxlan是一种隧道报文方式，但是它可以降级称为host-gw方式，也就是说vxlan可以直接像host-gw那样在二层物理直接通信，条件是：
            如果所有宿主机节点都在同一个物理网络中（不跨路由器）就工作为host-gw模式，如果不在同一个物理网络中（不跨路由器）就工作为vxlan模式；
        我们可以在部署flannel模式为vxlan时将Directrouting（直接路由）属性打开，有助于性能提升；
    Vxlan的传输效率低的问题：
        如上图Vxlan封装二层报文的格式，是虽然是flannel的默认、常用模式，但由于需要额外的封包和拆包，它的MTU是小于默认的1500；这也是很多人认为Flannel网络传输效率相对低的原因；
2.2.UDP：
    UDP是指四层UDP协议隧道，是所有模式中性能最差的；设计出UDP是有原因的：
        因为在flannel设计最初时有些发行版的linux内核还比较老不支持VxALN，为了能够兼容这些较老系统，flannel设计出UDP这种代替Vxlan的方案；早期很多人认为flannel性能很差就是因为当时UDP的性能差决定的，而如今的VXLAN性能其实并不差，所以在生产环境中是不会使用UDP模式的.
2.3.host-gw：
    1.flannel不仅提供基于封装类型的技术vxlan，也提供基于路由技术的互联技术host-gw模式，host-gw是host-gateway的意思，是将主机作为网关，基于路由技术完成纯三层的ip转发；
    2.host-gw直接路由的逻辑是：把node节点上的每个Pod容器通过物理网桥直接接入到物理网络中，把node节点的网络接口当做pod的网关使用，这样相当于是所有Pod在物理网络（同一个网段）中直接通信，这种不需要解封包的方式在三种模式中性能是最好的，但是有局限性：
        1.因为通过物理网桥直接接入到物理网络中相当于是在物理网络中直接通信，就意味着都是使用MAC地址通信，那么所有宿主机节点IP都必须在同一个物理网络中不能跨路由器!这样的话网络设备间的距离就受限了、相应的数量、范围也就受限了!所以host-gw模式仅适用于二层直接可达的网络；
        2.host-gw模式虽然流量几乎无损耗，但是需要维护n(n-1)/2条路由
    3.如上图示，可以解释为什么host-gw要求主机网络二层直接互联？
        从上面的抓包结果可以看出host-gw在传输层走的是tcp，然后在网络层的源IP和目的IP均是容器的IP，也就是虚拟IP，这就决定了二层互联，因为只有交换机是不关注源IP和目的IP；假如两台主机在两个vlan中，二层不通，三层通，那么就需要路由器，而路由器是无法识别容器的这些ip，当然也可以配置路由规则，但是显然没有这么做的。
2.4.aws-vpc：
    使用 Amazon VPC route table 创建路由，适用于AWS上运行的容器
2.5.gce：
    使用Google Compute Engine Network创建路由，所有instance需要开启IP forwarding，适用于GCE上运行的容器
2.6.ali-vpc：
    使用阿里云VPC route table创建路由，适用于阿里云上运行的容器
使用场景说明：
    所有节点都在同一个网段，并且二层网络可能通信，可以考虑选择Host-gw模式。反之节点之间不能通过二层网络通信，可能在不同vlan中，可以考虑选择Vxlan模式，也可以考虑使用 Vxlan Directrouting 模式(默认是false)

之前我们说过由于docker官方并没有提供多主机的容器通信方案，在没k8s之间我们使用也是使用flannel或者calico来解决docker的跨主机间容器通信问题的，和在k8s上不同的是k8s有etcd集群，而docker集群需要额外部署etcd来保存集群中每个节点的网络信息；参考案例:

3.2.为二进制部署的k8s集群部署Flannel网络插件

flannel-kubernetes集成
flannel-cni插件

因为部署部署k8s集群的方式有很多，比如kubeadm、kubease（ansible部署）、手动二进制部署，k8s集群部署方式不同flannel的部署也不相同，而主流的ansible和二进制部署实际上是一样的，这里就简单介绍kubeadm和详细说明二进制部署的k8s如何启用flannel插件；

1.kubeadm部署flannel说明
    kubeadm部署的k8s集群部署flannel网络插件分为两步：
    1.使用kubeadm初始化k8s集群时通过--pod-network-cidr选项指定pod网络时填flanne默认的网段地址：10.244.0.0/16
        ~]# kubeadm init --kubernetes-version=v1.13.3 --pod-network-cidr=10.244.0.0/16 --dry-run --service-cidr=10.96.0.0/12 --ignore-preflight-errors=Swap
    2.等待初始化等系列工作完成后部署flannel的pod即可
        ~]# kubectl apply -f https://raw.githubusercontent.com/coreos/flannel/master/Documentation/kube-flannel.yml
            关于这个清单里都定义了什么，在二进制部署中会说明
    3.验证：
        1.通过kubectl get pods -n kube-system确保所有flannel的pod都成功运行
        2.通过kubectl get nodes命令能够显示正确的节点信息说明flannel部署成功
    4.flannel的配置文件
        kubeadm部署并启用flannel插件后，flannel的相关配置信息会保存在这个文件中
            ~]# vim /etc/cni/net.d/10-flannel.conflist 
                {
                  "name": "cbr0",
                  "plugins": [
                    {
                      "type": "flannel",
                      "delegate": {
                        "hairpinMode": true,
                        "isDefaultGateway": true
                      }
                    },
                    {
                      "type": "portmap",
                      "capabilities": {
                        "portMappings": true
                      }
                    }
                  ]
                }

2.ansible或二进制部署的k8s集群启用flannel插件
    1.先在k8s中启用CNI网络插件接口
        1.在前一章说过k8s启用CNI网络插件接口是需要两个目录的，一个是网络插件的配置文件路径，一个是网络插件的二进制程序文件路径，这一步就是将所有的网络插件二进制程序文件放到k8s指定的目录中去，它们对应的配置文件路径这里不需要创建，因为在下文的kube-flannel.yml清单中的init容器中指定好了；
        2.CNI插件下载地址：https://github.com/containernetworking/plugins/releases
            ~]# 上传CNI插件到k8s的所有node节点上；
            ~]# mkdir -p /opt/cni/bin   #创建CNI默认加载插件的程序文件的路径
            ~]# tar xf cni-plugins-amd64-v0.7.4.tgz -C /opt/cni/bin/
                #将flannel的二进制程序文件放到此默认路径
            ~]# ls /opt/cni/bin/
                bridge  dhcp  flannel  host-device  host-local  ipvlan  loopback  macvlan  portmap  ptp  sample  tuning  vlan
                #确保这些二进制程序文件都生成了，后面安装flannel或者caclio时，会自动在此目录下查找对应网络插件的可执行文件，用于容器IP地址的分配；
    2.安装flannel
