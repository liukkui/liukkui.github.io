MySQL-备份和恢复工具之Mysqldump

1.Mysql备份和恢复

1.mysql数据备份的重要性和备份数据
    备份数据的最终目的是为了在出现一些意外情况时，能够通过备份将数据还原，所以单单的备份数据往往是无法满足还原时的需求的，所以需要备份的数据有：
    1.数据
    2.二进制日志、InnoDB的事务日志(是否一定备份取决于对数据安全的需求级别)
    3.程序代码（存储过程、函数、触发器、事件调度器） 
    4.服务器的配置文件、定时任务，sudo权限等信息
2.mysql的备份类型
    不管是哪种数据库,在备份时都会有"备份类型",而备份类型可分为：
    1.全量备份：
        全量备份也叫完全备份，全量备份就是对某个时间点的所有数据进行一个完全的备份，对应时间点的所有数据都被包含在完全备份中
    2.部分备份
        部分备份指的是只备份数据子集，如部分库或部分表;而部分备份又可以分为：
        1.差异备份
            1.差异备份"是对上一次"全量备份"以后变化的数据进行备份,备份较慢,还原简单;还原方式是:完全备份+最后一个差异备份(+二进制日志);
            2.比如:如上图示：周日2点对数据库进行了"全量备份",当下周一对数据库做差异备份时，将备份从周日2点以后到周一差异备份时期间的所有变化的数据,如果下周二对数据库进行差异备份，则会备份从周日2点以后到周二差异备份时期间的所有变化的数据，同理，如果下周三对数据库进行差异备份,依次类推,当还原数据时，只需要拥有周日的完全备份与最近一次的差异备份即可,中间的差异备份是不需要的;这就是差异备份的备份和还原逻辑;
        2.增量备份
            "增量备份"是对上一次"备份"以后变化的数据的备份;如上图示：差量备份"与"增量备份"唯一的区别就是备份的"起始参考点"不同;"差异备份"的起始点是最近一次的"全量备份"，"增量备份"的起始点是最近的上一次的"备份";所以增量备份的还原方式是：完全备份+N个增量备份(+二进制日志)
    3.时间点恢复
        1.由于不管是全量备份、差异备份、增量备份都不能解决另外一个问题：比如周日2点我们进行了全量备份，我们想要在下周一2点进行第二次备份，但是如果在周一2点之前数据库崩溃了,而我们并未做任何差异备份或者增量备份,那么周日2点~周一2点之间的数据就必须通过二进制日志来还原;
        2.由于mysql开启二进制日志后,所有数据变化都会被记录到二进制日志中;那么我们的还原方式是：完全备份+二进制日志重放
            可以先通过备份将数据还原到周日2点，然后再通过二进制日志将时间点周日2点以后的操作"重放"一遍,这样数据就能恢复到了崩溃之前的全部数据;
3.mysql备份数据的方式
    mysql备份数据的方式安装逻辑和方式不同分为：
    1.热备、温备、冷备
        1.热备是指在数据库正常运行的情况下进行备份;即在热备期间，数据库的读写操作均可正常进行,但是热备实现起来很复杂,不能只依靠简单的cp拷贝等命令，而是需要专门的备份工具;而且mysql中的myisam存储引擎不支持热备，InnoDB存储引擎支持热备;
        2.温备期间，数据库只能进行读操作，不能进行写操作，即数据库在可读但不可写的状态下进行备份,此时需要加全局读锁(FLUSH命令);
        3.冷备期间，数据库的读写操作均不可进行;虽然备份出的数据的可靠性是最高的,但实际情况是不可能停止数据库的读写的;
        4.说明：为什么MyISAM不支持热备，而InnoDB支持热备？
            1.对数据集做热备时，因为数据集都在做读写操作，为了避免产生脏数据，就要求某些存储引擎支持快照功能，而InnoDB的每一个事务启动时都是基于MVCC创建快照实现的；
            2.InnoDB实现方式：当打算开始备份时，对整个数据集做一次阻断内部做一个快照，而后只需要对这个快照做一次备份就行了；InnoDB每一次数据修改都有一个LSN(日志序列号)，但是MyISAM确没有；
            3.所以说备份技术是和存储引擎有关系的；
                MyISAM：支持温备和冷备,不支持热备
                InnoDB：三种都支持
    2.物理备份和逻辑备份
        1.物理备份
            物理备份就是直接备份数据库所对应的数据文件;不需要专用的备份工具、不用基于协议而是直接cp、tar命令操作;需要注意的是：物理备份与存储引擎有关
        2.逻辑备份
            1.逻辑备份不是备份物理文件的,需要通过特定工具从数据库中导出数据并另存备份,这类备份工具会作为mysql的客户端通过mysql协议连到mysql服务端把它的所有数据导出进行存档备份,与存储引擎无关，占用空间少，速度慢，可能丢失精度；
            2.逻辑备份的缺点是：
                逻辑备份工具通常都是基于协议工作的客户端工具，需要先转换成文本,像浮点型数据丢失数据精度有可能更占用空间；
4.mysql中常见的备份工具
    1.mysqldump
        mysqldump是mysql自带的逻辑备份工具,即它会将数据从数据库中读出，转化为对应的sql语句;mysqldump能够实现完全备份或部分备份,适用于所有存储引擎（因为逻辑备份与存储引擎无关）,但是：
            使用innodb表类型的表能够使用mysqldump进行热备,而这种热备是通过启动一个大事务进行的;
            使用myisam表类型的表只能够使用mysqldump进行温备。
        如果数据量较小(不大于2G)，使用mysqldump是一种选择;
    2.cp,tar等复制归档工具
        cp和tar是物理备份工具,只适用于冷备的方式
    3.lvm2快照+二进制日志
        实现几乎热备(先加锁，做快照后解锁);借助文件系统工具cp、tar进行备份;
    4.通过select语句进行部分备份
        通过select语句将表中的数据导出到指定文件中，例如导出a表数据到/testdir/a.backup中，在使用select语句备份a表的同时，最好将a表的表结构也备份一份，因为还原的时候很有可能用到;比如：
            > select * from a into outfile '/testdir/a.backup';
            如果想要通过a.backup文件恢复a表的内容，可以创建一个新的表，然后使用load data将a.backup中的数据导入到新表中，示例如下。
            > create table b(id int);
            > load data infile "/testdir/a.backup" into table b;
    5.xtrabackup
        Xtrabackup是Percona组织提供的物理备份工具,支持完全备份、增量备份、差异备份;xtrabackup支持对Innodb存储引擎进行热备,支持并行备份
    下文只要介绍mysqldump和xtrabackup的备用工具使用方式;
5.备份方案和备份策略
    1.假如有一个10G/100G的数据库，如何做备份？
        显然如果使用mysqldump是即为麻烦的而且很慢，如果是使用xtrabackup更可行：
        策略：
            每周一次完全备份+每天一次增量备份+二进制日志做时间点还原
    2.结合我们公司线上业务的实际情况，我们的备份方案主要采取全备+binlog备份方式,其中:
        全备分为逻辑备份+物理备份，同时主从复制也作为一种备份的方式存在，从而最大程度降低数据故障带来的风险；
    3.低成本、高安全性、且能够快速恢复的备份策略的几个要点：
        1.全量备份+增量备份+定期演练；
        2.1小时延时从库；
        3.双份1小时延时从库；

2.mysqldump

mysqldump命令使用说明

1.生产环境使用mysqldump的备份方案：
    1.如果数据量不大时，可以使用mysqldump一周做一次完全备份，然后每一天（每半天）做一次通过复制二进制日志文件做增量；
    2.每半天去复制滚动一次二进制日志文件，将此前的复制出来，过半天再滚动一次，然后再把前一个复制出来；
    3.恢复时使用：完全备份+N分增量的二进制日志
2.mysqldump说明：
    1.mysqldump是mysql自带的逻辑备份工具;它的备份原理是:
        1.通过协议连接到mysql数据库，将需要备份的数据查询出来，将查询出的数据转换成对应的insert语句，当我们需要还原这些数据时，只要执行这些insert语句，即可将对应的数据还原;
        2.mysqldump是把Schema和数据存储在一起、巨大的SQL语句、单个巨大的备份文件;
    2.mysqldump的优缺点
        1.优点：
            可以直接使用文本处理工具处理对应的备份数据，因为备份数据已经被mysqldump转换为了对应的insert语句，所以，我们可以借助文件系统中的文本处理工具对备份数据进行直接处理；
        2.缺点：
            当数据为浮点类型时，会出现精度丢失
    3.mysqldump对myisam和innodb存储引擎的支持方式：
        1.mysqldump对innodb存储引擎支持热备，innodb支持事务，我们可以基于事务通过mysqldump对数据库进行热备；
        2.mysqldump对myisam存储引擎只支持温备，通过mysqldump对使用myisam存储引擎的表进行备份时，最多只能实现温备，因为在备份时会对备份的表请求锁，当备份完成后，锁会被释放
    说明：
        mysqldump的备份过程属于逻辑备份，备份速度、恢复速度与物理备份工具相比较慢,而且mysqldump备份的过程是串行化的,不会并行的进行备份,如果想要并行备份,可以使用mydumper，而且当数据量较大时一般不会使用mysqldump备份因为效率较低,而是使用xtrabackup备份;
3.mysqldump语法和常见选项
    ~]# mysqldump --help
    三种语法备份数据库的语法：
        1.mysqldump [OPTIONS] database [tables]
        2.mysqldump [OPTIONS] --databases [OPTIONS] DB1 [DB2 DB3...]
        2.mysqldump [OPTIONS] --all-databases [OPTIONS]
        注意：
            1.第一种方式不会自动创建create_databases语句，因为他只会把db_name中的的每个table名称创建出来，当恢复时,还是需要先创建db_name；
            2.下面种方式都会有create database语句
    常见选项：
        -A,--all-databases      #备份所有数据库，含create database 
        -B,--databases db_name… #指定备份的数据库，包括create database语句 
        --single-transaction    #启动一个巨大的事务
        -E,--events：           #备份相关的所有event scheduler 
        -R,--routines：         #备份所有存储过程和自定义函数 
        --triggers：备份表相关触发器，默认启用,用--skip-triggers，不备份触发器 
        --default-character-set=utf8 #指定字符集 
        --master-data[=0|1|2]：      #此选项须启用二进制日志
            下文详解此参数选项;默认值为1
        -F,--flush-logs ：
            #备份前滚动日志，锁定表完成后，执行flush logs命令,生成新的二进制日志文件，配合-A或-B选项时，会导致刷新多次数据库。建议在同一时刻执行转储和日志刷新，可通过和--single-transaction或-x，--master-data一起使用实现，此时只刷新一次日志
        --skip-TZ-UTC：
            #在使用的mysqldump备份含有时间戳类型列的表时，需要指定--skip-TZ-UTC选项，否则可能会导致备份数据异常；推荐使用的日期时间，与时区无关且存储范围更大，可以避免一些问题
        --ignore-table=数据库名.表名 
            #备份时忽略某个库中的某个表,如--ignore-table=hellodb.students
        --compact               #去掉注释，适合调试，生产不使用 
        -d,--no-data            #只备份表结构 
        -t,--no-create-info     #只备份数据,不备份create table 
        -n,--no-create-db       #不备份create database，可被-A或-B覆盖
        --flush-privileges      #备份mysql或相关时需要使用 
        -f,--force              #忽略SQL错误，继续执行
        --hex-blob              #使用十六进制符号转储二进制列，当有包括BINARY，VARBINARY，BLOB，BIT的数据类型的列时使用，避免乱码
        -q,--quick             #不缓存查询，直接输出，加快备份速度 
4.mysqldump重要的参数选项详细说明
    1.-B,--databases db_name…
        当需要只备份数据库中的指定库时可以使用-B或者--databases选项,而且该选项会在备份时自动生成创建数据库的语句,这样在还原时就不需要事先创建对应的空数据库再还原了;
    2.--ignore-table=数据库名.表名 
        #当我们备份某个数据库,但不想备份该库中的某个表时，可以使用该选项忽略这个表,如：--ignore-table=hellodb.students
    3.--master-data[=0|1|2]
        1.在实际进行数据恢复时，往往需要基于时间点还原，通常的做法是先通过最近一次的备份将数据恢复到备份时的样子，然后再通过二进制日志，将备份之后的更改操作重放一遍，这样就将数据恢复成了故障之前的完整模样;比如:
            1.在备份开始时,二进制日志对应的position为123，那么我们在进行时间点还原时，则需要先通过"备份文件"将数据恢复为备份时的样子，然后再通过二进制日志文件，将position 123之后的所有操作重放一遍；
            2.而备份开始时二进制日志的position位置就必须在备份时通过--master-data选项标记到备份文件中,这样我们才可以在恢复时才知道应该从哪里重放剩下的二进制日志;
        2.而且备份时二进制日志文件的position对于主从复制结构来说统一是非常重要的;因为在实际生产中我们可能需要增加主从复制结构中的从节点数据,而从节点的数据做法一般是：
            先将主库中的数据先备份一份出来然后导入到从库中,但是从主库中备份数据时往往是热备的,主库还是在持续写入数据,那么我们将主库备份的文件导入到从库后,还需要告诉从库接下来从二进制日志的哪个位置同步接下来的数据,所以就必须在备份主库的时候使用--master-data标记出二进制日志的position;
        3.--master-data选项有3个可用值：0、1、2
            1.值为0：表示在使用mysqldump进行备份时，不记录对应二进制日志文件位置，将此值显式的设置为0与不使用此选项的效果相同;
            2.值为1：默认为1
                表示在使用mysqldump进行备份时，记录对应二进制日志文件位置，此值为默认值，也就是说使用--master-data与使用--master-data=1的效果相同，如果将此选项的值设置为1，则会在备份文件中生成对应的"CHANGE MASTER TO"语句，此语句中标明了备份开始时二进制日志的前缀名以及其所处的position(位置)，生成此语句的目的是在主从复制结构中的"从服务器"中通过备份sql还原数据以后，告诉"从库"，从"主库"的二进制日志文件中的哪个位置开始"同步";如果我们没有使用主从复制结构，同时又想要在备份时记录二进制日志文件的position，则可以将此选项的值设置为2
            3.值为2：
                表示在使用mysqldump进行备份时，记录对应二进制日值文件的位置，如果将此选项的值设置为2,则会在备份文件中生成对应的"CHANGE MASTER TO"语句，此语句中标明了备份开始时二进制日志的前缀名以及其所处的position(位置)，但是"CHANGE MASTER TO"语句将会被注释，与此值为1时不同，此选项值为1时，"CHANGE MASTER TO"语句不会被注释，此选项值为2时，"CHANGE MASTER TO"语句会被注释，所以，如果只是单纯的为了记录备份时的二进制日志文件位置，那么将此选项值设置为2即可
    4.--flush-logs
        1.我们知道mysql默认的二进制日志大小是由max_binlog_size变量定义的,默认大小为1G;每当二进制日志的大小满1G时,对应的二进制日志文件就会发生滚动，生成一个新的二进制文件并将原来的1G大小的二进制日志保存;
        2.而当我们备份数据库并且需要记录二进制日志的position位置时,如果二进制日志不发生滚动,那么这个position位置就可能在一个900M二进制文件的最后端，当我们向借助这个二进制日志还原时需要在原文件找很久才能找到这个位置;因此：
            --flush-logs的选项的意义就在于只要开始备份时就会滚动一次二进制日志,这样position就在这个新的二进制日志文件开头了,当我们向重放日志时就非常方便了;
    5.其他三个重要选项
        因为在数据库中可能还存在一些存储过程和存储函数、一些触发器、事件表等，这些"东西"也需要备份，以免最终的备份"不全";下面三个选项就是与其相关的：
            --routines选项：表示备份时，存储过程和存储函数也会被备份;
            --triggers选项：表示备份时，触发器会被备份;
            --events选项：表示备份时，事件表会被备份;
5.mysqldump备份InnoDB和MyISAM存储引擎时使用的不同参数选项：
    1.InnoDB存储引擎:
        1.mysqldump支持对innodb存储引擎进行温备、热备（热备是指在备份的时候能够对数据库进行读写）,在mysql事务文章中说过mysql的默认事务隔离级别是:可重复读（Repeatable-Read）;
        2.如果表类型为innodb,那么在"可重复读"的事务隔离级别中就可以保证热备出的数据的一致性的;它的备份逻辑是：
            在隔离级别为"可重读"的事务下,所有备份的操作都是在一个"独立的事务"中完成的,其他读写操作是在这个备份事务之外进行的，所以利用"可重读"事务的隔离性就可保证读写操作并不会对备份操作造成影响;
        3.mysqldump备份InnoDB存储引擎类型表的参数选项：
            --single-transaction：
                1.此选项只适用于Innodb,不适用MyISAM;此选项会开始备份前，先执 行START TRANSACTION指令开启事务;
                2.此选项通过在单个事务中转储所有表来创建一致的快照。 仅适用于存储在支持多版本控制的存储引擎中的表（目前只有InnoDB可以）;转储不保证与其 他存储引擎保持一致。在进行单事务转储时，要确保有效的转储文件（正确的表内容和二进制日志位置），没有其他连接应该使用以下语句：
                    ALTER TABLE，DROP TABLE，RENAME TABLE，TRUNCATE TABLE 
                3.此选项和--lock-tables（此选项隐含提交挂起的事务）选项不能同时使用!
                4.备份大型表时，建议将--single-transaction选项和--quick结合一起使用;如果开启了二进制日志，也不要忘了加上--master-data选项；
    2.MyISAM存储引擎：        
        1.由于MyISAM存储引擎不支持事务,所以不支持热备,只支持温备,即使使用--single-transaction选项对myisam表进行备份也不会有任何作用;
        2.而对于MyISAM表的备份只能通过锁表的方式,即在备份期间对备份的表施加全局读锁来完成,所以MyISAM表的备份是通过施加锁来保证数据一致性的;
        3.mysqldump备份MyISAM存储引擎类型表的参数选项：
            -x,--lock-all-tables：
                1.对所有库的所有表加读锁，此选项与--single-transaction选项不能同时使用；
                2.需要注意的是如果数据量大时，长时间锁表可能会导致长时间无法并发访问数据库;
             -l,--lock-tables：
                如果需要对指定的数据库进行备份时，也可以使用--lock-tables选项，只对指定库的所有表加锁,默认为on；使用--skip-lock-tables选项可禁用,对备份MyISAM的多个库,可能会造成数据不一致
            备注：上面两个选项也可以对innodb存储引擎的表进行备份,只不过不是热备而是温备,但不建议使用

    4.删除数据库还原测试：

2.MyISAM表备份示例：

    ~]# mysqldump -uxtrauser -p1234 -h192.168.34.118 --lock-tables --databases hellodb -p > /data2/hellodb1.sql
3.--master-data选项示例
    ~]# mysqldump -uxtrauser -h192.168.34.118 -p1234 --master-data=1 --databases hellodb > /data2/hellodb1.sql
    ~]# mysqldump -uxtrauser -h192.168.34.118 -p1234 --master-data=2 --databases hellodb > /data2/hellodb2.sql
    结果：
        1.结果如上图所示,--master-data=1时,备份中生成CHANGE MASTER TO语句未被注释,而且也在二进制日志binlog中记录了position位置;所以当--master-data=1时适合为新的从库同步数据;

6.如何实现把每个库备份到各自的备份文件中，分库备份？
    分析：
        多个库备份到一个文件中肯定是不合理的，所以生产环境是分库备份的,但是有时也只是某个库中的某个表有问题需要恢复，所以有时需要分库、分表同时进行；
    分库备份脚本：
        ~]# vim backup.sh
            #!/bin/bash
            for db in `mysql -e 'show databases' |grep -Ev '^(Database|information_schema|performance_schema)$'`;do mysqldump -B $db|gzip  > $db`date +%F`.sql.gz;done
        或者：
        ~]# mysql -e 'show databases' |grep -Ev '^(Database|information_schema|performance_schema)$'|sed -r 's/(.*)/mysqldump -B \1 |gzip > \/data\/\1.sql.gz/'  |bash
7.生产环境实战备份策略：
    1.InnoDB建议备份策略  
        mysqldump –uroot –A –F –E –R  --single-transaction --master-data=1 -flush-privileges  --triggers --default-character-set=utf8 --hex-blob >$BACKUP/fullbak_$BACKUP_TIME.sql 
    2.MyISAM建议备份策略  
        mysqldump –uroot –A –F –E –R –x --master-data=1 --flush-privileges  -triggers  --default-character-set=utf8  --hex-blob >$BACKUP/fullbak_$BACKUP_TIME.sql 
8.生产环境误删除恢复示例：
    问题描述：
        对于mysql正在运行过程当中，周一0点做了全量备份，但是在第二天上午10点删除了一个库或者删除了一个students表发生了一次误操作，在12点的时候发现数据库故障了，应该怎么解决？
    解决思路：
        1.周一0点做的全量备份时一般肯定加了--master-data=2记录了备份时的二进制日志位置，所以先将完全备份恢复；
        2.将备份时的二进制日志位置之后的二进制日志导入到一个新的文件中，因为这个二进制日志中有删除的操作，所以找到误操作的语句进行删除，然后再replay到数据库了，保证数据的完整性；
    备注：
        1.如何找到删除的是哪行或者update语句忘了加where的语句？
            ~]# mysqlbinlog --no-defaults -v -v --base64-output=DECODE-ROWS mysql-bin.000005 | grep -B 15 'students'
        2.为了避免忘了加where的问题，强烈建议在[mysql]段落开启这个参数：
            safe-updates
        3.强烈推荐二进制日志binglog是基于row行记录的;
9.Mysql的全量备份+异地备份脚本
    一般Mysql数据库备份会采用在MYSQL从库上执行全量备份+增量备份方式。在从库备份避免Mysql主库备份的时候锁表造成业务影响;
    ~]# vim db_backup.sh
        #!/bin/bash
        # description:  MySQL buckup shell script  
        # author:       dycre.com
        # 192.168.10.10 为专门的备份服务器，需要做一下服务器之间免密码登录
        #备份的数据库名
        DATABASES=(
            "cre_manage"
            "cre_diaodu"                    
        )
        USER="root"
        PASSWORD="cre123"
        MAIL="cre@dayang.com" 
        BACKUP_DIR=/data/backup
        LOGFILE=/data/backup/data_backup.log 
        DATE=`date +%Y%m%d_%H%M`
        cd $BACKUP_DIR
        #开始备份之前，将备份信息头写入日记文件
        echo "--------------------" >> $LOGFILE   
        echo "BACKUP DATE:" $(date +"%y-%m-%d %H:%M:%S") >> $LOGFILE   
        echo "-------------------" >> $LOGFILE
        for DATABASE in ${DATABASES};do
            /usr/local/mysql/bin/mysqldump -u$USER -p$PASSWORD --events  -R --opt  $DATABASE |gzip >${BACKUP_DIR}\/${DATABASE}_${DATE}.sql.gz
            if [ $? == 0 ];then
                echo "$DATE--$DATABASE is backup succeed" >> $LOGFILE
            else
                echo "Database Backup Fail!" >> $LOGFILE   
        done
        #判断数据库备份是否全部成功，全部成功就同步到异地备份f服务器
        if [ $? == 0 ];then
            /usr/bin/rsync -zrtopg   --delete  /data/backup/* root@192.168.10.10:/data/backup/  >/dev/null 2>&1
        else
            echo "Database Backup Fail!" >> $LOGFILE   
            #备份失败后向管理者发送邮件提醒
            mail -s "database Daily Backup Fail!" $MAIL   
        fi
        #删除30天以上的备份文件 
        find $BACKUP_DIR  -type f -mtime +30 -name "*.gz" -exec rm -f {} \;
10.备份脚本
    ~]# vim /usr/local/mysql/run/mysql_backup_full.sh
        #!/bin/bash
        #这个脚本用来对数据库作全备份
        user=root
        password=123456
        backupdir=/opt/mysql/backup/        #备份目录
        #取一天前备份时生成的那个二进制日志文件
        val_binlog=$(less /opt/mysql/binlog/mysql-binlog.index | awk -F"/" '{print $5}' |tail -n 2|head -n 1)
        #创建备份目录
        [ ! -d $backupdir ] && mkdir -p $backupdir
        #备份除Database|information_schema|mysql|test|performance_schema这几个数据库之外的所有库
        for db in `mysql -u $user -p $password -e 'show databases' |grep -Ev '^(Database|information_schema|performance_schema)$'`;do mysqldump -u $user -p $password --lock-all-tables --routines --events --triggers --master-data=2 --flush-logs -A > $backupdir$db`date +%F-%H:%M:%S`.sql.gz;done
        #删除两天前的备份文件，即保留三个备份文件
        find $backupdir  -type f -mtime +2 -exec rm -f {} \;
        #删除两天前的二进制日志文件,即保留三个二进制日志文件
        mysql -u $user -p$password -e "PURGE MASTER LOGS TO '$val_binlog'"
    ~]# chmod 700 /usr/local/mysql/run/mysql_backup_full.sh
        #由于此脚本里面包含了mysql的root密码，为了安全，设置700权限
    ~]# crontab -e  （创建计划任务）
        #分 时 日 月 周 用户名 命令
        0 1 * * * cd /usr/local/mysql/run;./mysql_backup_full.sh
        指定计划任务在每天的凌晨1点进行数据备份和删除
