5.RBD(Rados Block Device)的原理和使用

更新于：2019.09

1.RBD全称(RADOS Block Devices)，是一种建构在RADOS存储集群之上为客户端，rbd向下是Rados-Cluster的客户端，rbd向上所提供的任何一个rbd接口都可以当成是一个独立而且完整的image磁盘设备使用，所以它提供块设备接口的存储服务中间层,这类的客户端包括虚拟化程序KVM（结合qemu）和云的计算操作系统OpenStack和CloudStack等，RBD的应用场景有：
    1.被linux主机使用
        由于linux主机是由户空间和内核空间组成，硬盘是被内核识别后才表现为dev下的一个个设备,然后借助用户空间的工具进行格式化使用,所以如果linux主机想使用rbd虚拟出的磁盘设备就需要内核中有一个模块能够识别ceph，好在在内核中就有rbd.ko模块能够把自己当成客户端连入rbd服务上检索出它关联使用的镜像文件.
    2.被KVM虚拟机使用
        当创建kvm虚拟机时需要指定使用何种磁盘设备，而kvm可以在指定磁盘时是使用libvirt通过rbd协议连入rbd检索出某个映像文件当做磁盘设备存储文件。
    rbd协议：
        不管是哪种使用场景，它们中间必须经过rbd协议衔接，rbd是CS架构，Server是Rados-Cluster上的库librbd,librbd不需要监听任何套接字确可以借助Rados-Cluster提供块数据服务，Client可以是rbd.ko模块或者KVM的libvirt。
2.RBD基于RADOS存储集群中的多个OSD进行条带化(条带也叫对象Object)，支持存储空间的简配（thin-provisioning）和动态扩容等特性，并能够借助于RADOS集群实现快照、副本和一致性;通过将存储池提供的存储服务抽象为一到多个image（表现为块设备）向客户端提供块级别的存储接口:
    1.在rbd存储池中创建image(块设备)
        RBD支持两种格式的image，由于ceph分层特性COW克隆特性只支持v2格式的镜像，所以现在默认都是使用v2格式了。
    2.客户端访问RBD设备的方式有两种
        1.通过内核模块rbd.ko将image映射为节点本地的块设备，相关的设备文件一般为/dev/rdb#（#为设备编号，例如rdb0等）
            ~]# find / -name rbd.ko*
            /usr/lib/modules/3.10.0-957.21.3.el7.x86_64/kernel/drivers/block/rbd.ko.xz
            #确认当前客户端有此模块才能映射到内核中
        2.另一种则是librbd提供的API接口，它支持C、C++和Python等编程语言，qemu即是此类接口的客户端。
3.rbd命令(image的增删改查等命令)
    RBD相关的管理有如image的创建、删除、修改和列出等基础CRUD操作，也有分组、镜像、快照和回收站等相的管理需求，这些均能够通过rbd命令完成；而学习RBD的使用主要就是学习RBD设备中的image映像文件的增删改查和image映像文件快照的增删改查、回滚、合并等操作!!!
    1.创建并初始化RBD存储池的三个步骤：
        使用块设备客户端之前必须事先创建专用的存储池、启用rbd并完成其初始化:
        1.创建存储池：ceph osd pool create {pool-name} {pg-num} {pgp-num}
        2.启用rbd：ceph osd pool application enable {pool-name} rbd
        3.rbd初始化：rbd pool init -p {pool-name}
        示例：
            ~]$ ceph osd pool create kube 64 64           #创建kube存储池
            ~]$ ceph osd pool application enable kube rbd #启用rbd功能
            ~]$ rbd pool init kube  #初始化存储池kube
    2.创建并查看image
        1.创建image的两种命令格式:rbd create ...
            rbd create --size <size> --pool <pool-name> <image-name>
            或者
            rbd create --size <size> <pool-name>/<image-name>
            示例:
                ~]$ rbd create --pool kube --size 2G vol01
                或者
                ~]$ rbd create --size 2G kube/vol02
                    在存储池kube中创建名为vol01的2G映像
        2.查看image:rbd ls
            rbd ls [-p <pool-name>] [--long] [--format json|xml] [--pretty-format] <pool-name>    
            示例：
            ~]# rbd ls -p kube -l  #查看
                NAME    SIZE   PARENT  FMT  PORT  LOCK
                vol01   2 GiB           2
                vol02   2 GiB           2
        3.rbd create创建image时的其他几个有用选项：
            --object-size <object-size>:文件被切成对象的块大小，默认是4MiB
            --image-feature <image-feature>:
                镜像文件的特性，镜像文件支持N个特性，详见下文image特性
       4.rbd info命令可以获取特定image的详细信息，每段代表意义如下：
            ~]$ rbd info /kube/vol01
            rbd image 'vol01':
                size 2 GiB in 512 objects
                order 22 (4 MiB objects)
                id: 14ns8smsda8
                block_name_prefix: rbd_data.14ns8smsda8
                format：layering,exclusive-lock,object-map,fast-diff,deep-flatten
                op_features:
                flags:
                create_timestamp: Wed Mar 6 10:38:20 2019
            每段代表意义：
                size M GiB in N objects：
                    #image空间大小为M，共分割至N个对象（分割的数量由条带大小决定）；
                order 22 (4 MiB objects)：
                    #块大小（条带）的标识序号，有效范围为12-25，分别对应着4K-32M之间的大小；
                id：#当前image的标识符；
                block_name_prefix：这个image中的每一个对象object都是以此为前缀命名的。
                format： image使用v2格式
                features：
                    当前image启用的功能特性，其值是一个以逗号分隔的字符串列表，例如layering、exclusive-lock等；
                op_features：可选的功能特性；    
4.image特性
    ~]$ rbd create image --image-feature arg,默认启用的5个特性feature有：layering(+),exclusive-lock(+*),object-map(+*),fast-diff(+*),deep-flatten(+-),其他feature和代表意义是：
        1.layering:支持分层镜像、是否支持克隆；
        2.striping:是否支持数据对象间的数据条带化v2；
        3.exclusive-lock:支持独占锁(也叫排他锁)
            即是否支持分布式排他锁机制以限制同时仅能有一个客户端访问当前image；(和k8s-volume的单路只读逻辑是一样的)
        4.object-map:支持对象映射（依赖exclusive-lock）
            即是否支持object位图，主要用于加速导入、导出及已用容量（快速计算出已使用空间容量不是通过扫描整个空间，而是通过扫描object位图中的块数*块大小快速计算得来的）统计等操作，依赖于exclusive-lock特性；
        5.fast-diff:快速计算差异（依赖object-map）
            即是否支持快照间的快速比较操作，依赖于object-map特性；fast-diff特性有利于在deep-flatten展平操作时进行快速合并.
        6.deep-flatten:支持快照扁平化操作
            1.layering特性使得克隆image的时候：父子image之间采用COW，他们之间的对象文件存在依赖关系，而flatten操作的目的是解除父子image的依赖关系，但是子image的快照并没有解除依赖，deep-flatten特性使得快照的依赖也解除；
            2.所以deep-flatten深层展平是一种真正做深层分离原卷与快照卷关系时，把原卷上的数据实现向对应快照卷上复制时的一种将多层合为一层或分离存在依赖关系的多层卷时的操作。
        7.journaling:支持记录IO操作（依赖独占锁）
            即是否支持日志IO，即用户在修改image上的数据时是否记录日志，日志能够在必要时确保数据一致性和实现快速恢复；它依赖于exclusive-lock特性；
        8.data-pool:
            是否支持将image的数据对象存储于纠删码存储池，主要用于将image的元数据与数据放置于不同的存储池；
    备注：
        从J版开始，创建的镜像image文件默认就支持前5个特性，而这5个特性被linux客户端作为磁盘被加载到内核中使用时这三个特性object-map,fast-diff,deep-flatten是不被支持的，所以需要禁用掉.
5.客户端image映射及断开
    在RBD客户端节点上以本地磁盘方式使用块设备之前，需要先将目标image映射至本地内核，需要安装ceph-common包才能使用rbd命令，而且如果Ceph存储集群端启用了CephX认证，客户端还需要指定用户名和keyring文件以及ceph.conf配置文件。
    1.映射的命令：
        ~]$ rbd map [--pool <pool>] [--image <image>] [--id <user-name>] [--keyring </path/to/keyring>]
    2.查看已经映射的image
        ~]$ rbd showmapped      #显示每一个rbd磁盘的map详情
    3.断开映射
        ~]$ umount /dev/***   #先umount再unmap
        ~]$ rbd unmap [--pool <pool>] [--image <image>] <image-or-device-spec>
    4.客户端开机映射rbd块设备
        使用rbd命令手动映射在节点重启映射会丢失；而ceph-common软件包中包括一个rbdmap的shell脚本，该脚本可对一个或多个RBD进行自动化rbd map和rbd unmap操作，该方式主要优点是由Init系统触发了在启动时自动映射和装载RBD映像（以及在关机时卸载和取消映射）
        ~]# yum install ceph-common -y
        ~]# vim /etc/ceph/rbdmap
            #poolname/imagename  id=client,keyring=/etc/ceph/ceph.client.keyring
            #该shell脚本的配置配置参数文件为默认为/etc/ceph/rbdmap
        如果想要开机挂载rbd块设备，可以配置如下：
        ~]# vim /etc/ceph/rbdmap
            kube/vol01 id=client.kube,keyring=/etc/ceph/ceph.client.kube.keyring
            注意:即使是rbd的默认池，此处的poolname不能省,参数不是必须的，此处也可以省略，可以直接写kube/vol01
        ~]# systemctl enable rbdmap.service
        ~]# systemctl start rbdmap.service    #设置rbdmap服务启动
    5.写入fstab中实现自动挂载
        ~]# vim /etc/fstab
            /dev/rbd0  /mnt/  xfs  noauto 0 0
        注意：
            1.指定"noauto"（或"nofail"）安装选项;这样可以防止Init系统尝试过在相关设备甚至还没有存在之前就映射设备，rbdmap.service通常是在启动顺序中很晚才触发的
            2.设备名是/dev/rbd0,也可以写成/dev/rbd/kube/vol01（官方推荐）
6.管理image
    1.Image特性管理
        1.J版本起，image默认支持的特性有layering、exclusive-lock、object-map、fast-diff和deep-flatten五个；
        2.rbd create命令的--feature选项支持创建时自定义支持的特性；
        3.现有image的特性可以使用rbd feature enable或rbd feature disable命令修改
            ~]$ rbd feature disable kube/vol01 object-map fast-diff deep-flatten   #禁用即可
    2.调整image的大小
        命令：rbd resize [--pool <pool>] [--image <image>] --size <size> [--allow-shrink] [--no-progress] <image-spec>
        增大：
            rbd resize [--pool <pool>] [--image <image>] --size <size>
        减少：
            rbd resize [--pool <pool>] [--image <image>] --size <size> [--allow-shrink]
    3.删除image的命令：remove和trash
        1.彻底删除
            rbd remove [--pool <pool>] [--image <image>] [--no-progress] <image-spec>
            注意：删除image会导致数据丢失，且不可恢复；建议使用trash命令先将其移入trash，确定不再需要时再从trash中删除；
        2.先移到回收站，必要时可恢复
            rbd trash {list|move|purge|remove|restore}
            示例：
                ~]$ rbd trash move kube/vol01      #移到回收站
                ~]$ rbd trash list -p kube   #查看回收站中的image
                    147sjnx9cnc vol01 
                ~]$ rbd trash restore kube/vol01 --image-id 147sjnx9cnc
                    #从回收站恢复时需要指定image-id
7.image的快照基本操作
    1.快照机制：
        1.做快照是很快的，因为快照仅仅是创建了访问原卷数据的路径，没有任何数据修改之前快照是空的，当需要修改数据时，会通过COW写时复制机制把需要修改的数据复制到快照上修改后保存在快照上，所以原卷上始终是没修改之前的旧数据，修改后的数据都在快照上，所以快照是实现备份的效果；
        2.快照有全量快照和增量快照两种，全量快照使用镜像分离技术实现，增量快照有写时复制COW（Copy-On-Write）和写时重定向ROW（Redirect-On-Write）两种实现方式。
    3.rbd命令及许多高级接口（包括QEMU、libvirt、OpenStack等）都支持设备快照
    创建快照：
        ~]$ rbd snap create [--pool <pool>] --image <image> --snap <snap>
        或者
        ~]$ rbd snap create [<pool-name>/]<image-name>@<snapshot-name>
        注意：
            在创建映像快照之前应停止image上的IO操作，且image上存在文件系统时，还要确保其处于一致状态；
    列出快照：
        ~]$ rbd snap ls [--pool <pool>] --image <image> [--format <format>] [--pretty-format] [--all]
    回滚快照:
        ~]$ rbd snap rollback [--pool <pool>] --image <image> --snap <snap> [--no-progress]
        注意：
            将映像回滚到快照意味着会使用快照中的数据重写当前版本的image，而且执行回滚所需的时间将随映像大小的增加而延长    
    设置image上的快照数量上限：
        创建快照一定会影响性能，而且数量过多必然会导致image上的原有数据第一次修改时的IO压力增加。
        ~]$ rbd snap limit set [--pool <pool>] [--image <image>] [--limit <limit>]
        解除快照数量限制：
            ~]$ rbd snap limit clear [--pool <pool>] [--image <image>]
    重命令快照：
        ~]$  rbd rename [--pool <pool>] [--image <image>]
    删除快照：
        ~]$ rbd snap rm [--pool <pool>] [--image <image>] [--snap <snap>] [--no-progress] [--force]
        提示：
            Ceph OSD会以异步方式删除数据，因此删除快照并不能立即释放磁盘空间;
    清理快照：
        删除一个image的所有快照，可以使用rbd snap purge命令
        ~]$ rbd snap purge [--pool <pool>] --image <image> [--no-progress]
    示例：
        ~]$ rbd snap create kube/vol01@snap01      #创建
        ~]$ rbd snap rollback kube/vol01@snap01    #回滚
        ~]$ rbd snap rm kube/vol01@snap01          #删除
        ~]$ rbd snap limit set kube/vol01 --limit 10 #设置快照数量
        ~]$ rbd snap limit clear kube/vol01        #清除数量限制
8.Ceph快照的分层克隆技术(基于镜像的快照再做镜像)
    1.Ceph的一个非常好的特性在于支持在一个块设备image的快照基础上创建一到多个COW或COR（Copy-On-Read）类型的克隆，这种中间快照层（snapshot-layering）机制提了一种极速创建image的方式，快照分层技术使用场景：
        1.应用于OpenStack等云平台时，使用快照形式保护ceph-RBD镜像，快照是只读的，但COW克隆是完全可以写 ，可以多次来孵化实例，对云平台来说是非常有用的。
        2.还可以为Qemu虚拟机创建一个image并安装好基础操作系统环境作为模板，对其创建创建快照层后，便可按需创建任意多个克隆作为image提供给多个不同的VM（虚拟机）使用，或者每创建一个克隆后进行按需修改，而后对其再次创建下游的克隆
    2.通过克隆生成的image在其功能上与直接创建的image几乎完全相同，它同样支持读、写、克隆、空间扩缩容等功能，惟一的不同之处是克隆引用了一个只读的上游快照，而且此快照必须要置于"保护"模式之下,支持COW和COR两种类型
        1.COW是为默认的类型，仅在数据首次写入时才需要将它复制到克隆的image中
        2.COR则是在数据首次被读取时复制到当前克隆中，随后的读写操作都将直接基于此克隆中的对象进行
    3.在RBD上使用分层克隆的方法非常简单：创建一个image，对image创建一个快照并将其置入保护模式，而克隆此快照即可
        创建克隆的image时，需要指定引用的存储池、镜像和镜像快照，以及克隆的目标image的存储池和镜像名称，因此，克隆镜像支持跨存储池进行；
    创建被保护protect的原始快照（parent父快照）：
        下游image需要引用上游快照中的数据，快照的意外删除必将导致数据服务的中止，因此在克隆操作之外，必须将上游的快照置于保护模式
            rbd snap protect [--pool <pool>] --image <image> --snap <snap>
    克隆快照(基于快照生成镜像)：
        rbd clone [--pool <pool>] --image <image> --snap <snap> --dest-pool <dest-pool> [--dest <dest>
        或者：
        rbd clone [<pool-name>/]<image-name>@<snapshot-name> [<pool-name>/]<image-name>
            #指明从存储池A中的镜像Aclone到存储池B的镜像B(可跨存储池克隆)
    显示一个快照上的所有子镜像：
        rbd children [--pool <pool>] --image <image> --snap <snap>
        如：rbd children kube/vol01@par-clone
            基于vol01的快照par-clone克隆的所有镜像
    展平克隆的image：
        1.所谓的镜像展平是指：基于原始的父快照克隆的镜像都会自动保留对父快照的引用，当删除父快照上层的原始镜像时为了避免下层克隆出来的所有镜像文件丢失，必须对image做展平操作，以实现删除原始镜像时数据能够复制到所有下层克隆出来的镜像文件。
        2.展平克隆所需的时间随着映像大小的增加而延长 
        3.要删除某拥有克隆子项的快照，必须先平展其子image
        命令：
            rbd flatten [--pool <pool>] --image <image> --no-progress
            #展平时只需要指定展平的image和所在的pool即可
    取消快照保护：
        必须先取消保护快照，然后才能删除它
        用户无法删除克隆所引用的快照，需要先平展其每个克隆，然后才能删除快照
        命令：
        rbd snap unprotect [--pool <pool>] --image <image> --snap <snap>
    示例：
        rbd snap create kube/vol01@par-clone      #创建被保护的快照par-clone 
        rbd snap protect kube/vol01@par-clone #将par-clone至于保护模式
        rbd clone kube/vol01@par-clone kube/vol03
            #基于vol01的快照par-clone生成新的镜像vol03
        rbd children kube/vol01@par-clone   #显示该镜像的所有克隆
        rbd flatten kube/vol03              #先展平
        rbd snap unprotect kube/vol01@par-clone #再取消保护父快照
        rbd snap rm kube/vol01@par-clone #即使删除父快照，vol03还是正常使用

–rbd两种使用方式

rbd使用方式是Ceph三种方式中最成熟的组件，多数用在配合OpenStack中Cinder使用，作为其后端存储；RBD使用方式有内核rbd.ko模块和librbd两种方式，并且这两种方式都需要客户端支持，所以导致Windows系统无法使用RBD块设备，使用架构图如上；

1.内核rbd.ko挂载方式(mount、map方式)
    1.先在Ceph节点上创建存储池、用户、权限、复制keyring等操作
        ~]$ ceph osd pool create kube 64 64           #创建kube存储池
        ~]$ ceph osd pool application enable kube rbd #启用rbd功能
        ~]$ rbd pool init kube  #初始化存储池kube
        ~]$ rbd create --pool kube --size 2G vol01    #创建镜像image文件
            注意：
                如果centos7的内核比较旧，还需要把image默认启用的5个feature的后4个关闭客户端挂载才不会显示出错(只禁用后3个也可以)
            ~]$ rbd feature disable kube/vol01 exclusive-lock object-map fast-diff deep-flatten
        ~]$ ceph auth add get-or-create client.kube mon 'allow r' osd 'allow * pool=kube'  #创建用户及权限
        ~]$ ceph auth get client.kube -o ceph.client.kube.keyring
            #导出client.kube用户的keyring文件然后复制给客户端
        ~]$ scp ceph.client.kube.keyring root@172.30.0.100:/etc/ceph/
            #复制ceph.client.kube.keyring文件到客户端172.30.0.100的/etc/ceph/目录下
        ~]$ scp ceph.conf root@172.30.0.100:/etc/ceph/
            #客户端主机也需要ceph.conf配置文件；
    2.客户端挂载rbd
        ~]# find / -name rbd.ko*
        /usr/lib/modules/3.10.0-957.21.3.el7.x86_64/kernel/drivers/block/rbd.ko.xz
            #CentOS6系列版本的内核没有集成RBD驱动，CentOS7中的3.10内核中默认已经集成了RBD驱动
        ~]# lsmod | grep rbd     
            rbd                    83640  0 
            libceph               306625  1 rbd
            #确认客户端内核加载了rbd模块，如果每加载可使用modprobe rbd命令加载
        ~]# yum install ceph-common     #安装客户端程序包，才有rbd命令
        ~]# ceph --user kube -s 
            #由于客户端主机上没有创建cephadm账户，所以需要使用--user指定kube用户
        ~]# rbd --user kube map kube/vol01
            /dev/rbd0
            #将kube/vol01这个image映射为本地的/dev/rbd0磁盘
        ~]# mkfs.xfs /dev/rbd0     #格式化磁盘
        ~]# mount /dev/rbd0 /mnt   #挂载在/mnt目录下使用
        ~]# rbd showmapped         #显示每一个rbd磁盘的map详情
            id pool namespace image      snap device    
            0  rbd            kube/vol01    /dev/rbd0
        ~]# cd /mnt && touch f1.txt #写入测试
    3.开机映射rbd块设备
        ~]# vim /etc/ceph/rbdmap
            kube/vol01 id=client.kube,keyring=/etc/ceph/ceph.client.kube.keyring
            注意:即使是rbd的默认池，此处的poolname不能省,参数不是必须的，此处也可以省略，可以直接写kube/vol01
        ~]# systemctl enable rbdmap.service
        ~]# systemctl start rbdmap.service
    4.客户端取消挂载
        ~]# umount /mnt 
        ~]# rbd unmap /dev/rbd0
            #不但要取消挂载还要取消映射
        ~]$ rbd ls -p kube -l 
            NAME    SIZE   PARENT  FMT  PORT  LOCK
            vol01   2 GiB           2         excl
            vol02   2 GiB           2
            #客户端主机取消映射后，rbd存储池中该imageLOCK字段就会显示excl

2.libvirt、RBD和librbd
    第一种是内核rbd.ko模块挂载rbd块设备，其实使用最多还是通过librbd使用rbd块设备;比如：openstack的存储中使用RBD是最多的，openstack就是通过QEMU\KVM虚拟机连接到rbd块设备的，支持rbd驱动的QEMU/KVM会将对磁盘的请求转换为通过librbd访问Ceph中的rbd设备，访问的配置信息（连入ceph的用户名和密钥）是配置在libvirt配置文件中;
    1.客户端主机安装
        客⼾端主机是为KVM Hypervisor主机，它运⾏的虚拟机的磁盘通过rbd协议使⽤Ceph上的rbd存储池中的image；在一台配置比较大的主机上安装KVM虚拟化套件并启动相关服务，此处列出简略步骤；
        ~]# yum install libvirt qemu-kvm -y
        ~]# systemctl start libvirtd
            #安装libvirt与qemu-kvm，并启动libvirtd守护进程；
        ~]# yum install ceph-common 
            #安装ceph-common，并配置好与ceph集群通信的配置⽂件等。
        创建一个libvirt启动虚拟机的xml模板，在模板上指定data interface console和dievice
    2.创建认证到存储池上的⽤⼾账号client.libvirt，并给予相应的访问权限，例如以rbdpool为例：
        ~]$ ceph auth get-or-create client.libvirt mon 'allow r' osd 'allow class-read object_prefix rbd_children, allow rwx pool=rbdpool'
            #osd上的授权需要额外有class-read、object_prefix（只读取对象前缀的权限）、rbd_children（读取镜像文件的克隆）
        ~]# ceph auth get client.libvirt -o ceph.client.libvirt.keyring
        ~]# scp ceph.client.libvirt.keyring root@172.30.0.80:/etc/ceph
            #将keyring文件复制到安装KVM套件的主机上
    3.将client.libvirt⽤⼾的认证信息导⼊为libvirtd上⼀个可⽤的secret
        启动libvirtd后就可以使用virtsh命令创建虚拟机了，但不管是使用virsh还是virt-manger或virt-install创建虚拟机，底层还是使用libvirt连入ceph集群，而libvirtd守护进程连入认证到ceph集群需要使用账号和密钥，其中账号是创建xml文件中定义的，密钥是从ceph上获取并导入给secret的，相关命令是virsh secret-define和virsh secret-dumpxml等；
        3.1.先创建包含登录到ceph集群用户的xml文件
            ~]# vim ceph-client-libvirt-secret.xml  #复制下面内容到文件中修改
                <secret ephemeral='no' private='no'>
                    <usage type='ceph'>
                    <name>client.libvirt secret</name>
                    </usage>
                </secret>
            其中：
                type是指这是登录到ceph集群用到的用户账号xml文件；
                <name>是认证到ceph上的用户client.libvirt，已创建好的ceph用户
                secret是通过此xml文件生成后再导入进来的
        3.2.生成secret
            ~]# virsh secret-define --file ceph-client-libvirt-secret.xml
                Secret d1477e08-1d29-33e8-9999-k8s6axd9ca8c created
            将上面的xml使⽤virsh命令创建出secret，命令会返回创建的secret的UUID
        3.3.将client.libvirt的密钥导⼊到刚生成的secret中
            ~]# ceph auth print-key client.libvirt #获取用户密钥
                AQCwkz9cSDLhNSAjx+JS+sK8SN2DS5W==   
            ~]# virsh secret-set-vaule --secret d1477e08-1d29-33e8-9999-k8s6axd9ca8c --base64 AQCwkz9cSDLhNSAjx+JS+sK8SN2DS5W== 
                Secret avlue set
                #将获取的用户密钥当做secret值导入到secret中
            ~]# virsh secret-get-vaule --secret d1477e08-1d29-33e8-9999-k8s6axd9ca8c
                AQCwkz9cSDLhNSAjx+JS+sK8SN2DS5W==
                #反之通过secret可以查看到它的值，也就是用户client.libvurt的密钥
        这样libvirtd守护进程就可以通过此secret认证到ceph集群了；
    4.创建空的image，或者导⼊已经的磁盘镜像内容，以运⾏基于image的KVM虚拟机；
        4.1.导⼊命令格式：
            qemu-img convert -f qcow2 -O raw /PATH/FROM/IMAGE_FILE rbd:<rpd-pool-name>/<image-name>
            例如：
               ~]# qemu-img convert -f qcow2 -O raw cirros-0.3.4-x86_64-disk.img rbd:rbdpool/cirros-0.3.4 
            查看导⼊⽣成的镜像：
                ~]# rbd info -p rbdpool cirros-0.3.4
                rbd image 'cirros-0.3.4': size 39 MiB in 10 objects order 22 (4 MiB objects) id: 13896b8b4567
                block_name_prefix: rbd_data.13896b8b4567 format: 2 features: layering, exclusive-lock, object-map, fast-
                diff, deep-flatten op_features: flags: create_timestamp: Tue Mar 5 10:11:38 2019
        4.2.或者创建新的空⽩image
            ~]# rbd create --image centos-7.6-0001 --pool rbdpool
    5.创建VM（两个vm模板）
        1.创建VM配置⽂件，下面是一个接近最小化的vm模板；其中的disk配置段指明了要使⽤的rbd image，⽽auth配置段指明了认证到Ceph集群时使⽤的secret的UUID。
            <domain type='kvm'>
              <name>testvm</name>
              <memory>131072</memory>
              <vcpu>1</vcpu>
              <os>
                <type arch='x86_64'>hvm</type>
              </os>
              <clock sync="localtime"/>
              <devices>
                <emulator>/usr/libexec/qemu-kvm</emulator>
                <disk type='network' device='disk'>
                   <source protocol='rbd' name='rbdpool/cirros-0.3.4'>
                      <host name='mon01' port='6789'/>
                   </source>
                   <auth username='libvirt'>
                      <secret type='ceph' uuid='7384444b-3d69-44e8-9999-1f821cd9ca8c'/>
                   </auth>
                   <target dev='vda' bus='virtio'/>
                </disk>
                <interface type='network'>
                  <mac address='52:54:00:25:a1:80'/>
                  <source network='default'/>
                  <model type='virtio'/>
                </interface> 
                <graphics type='vnc' port='-1' keymap='de'/>
              </devices>
            </domain>
            其中：
                disk段：表示使用rbd的rbdpool/cirros-0.3.4镜像作为此虚拟机的设备以及连入ceph主机的主机名和端口
                auth段:即上面第二步创建好的secret的UUID
        2.创建虚拟机
            将上面的vm模板保存为xml文件放在/etc/libvirt/qemu目录下命名为textvm.xml，并基于此xml启动为虚拟机
            ~]# virsh define testvm.xml 
            ~]# virsh start testvm
            ~]# virsh list
                1  testvm  running     #确认启动即可
            ~]# virsh domblklist testvm
                vda rbdpool/cirros-0.3.4
                #查看虚拟机的磁盘设备状态，可以看到是基于rbd的磁盘文件cirros-0.3.4加载启动的
