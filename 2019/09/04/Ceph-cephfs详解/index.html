6.CephFS(Ceph File System)文件系统接口的原理和使用详解

更新于：2019.09

6.1.CephFS的原理和使用

–CephFS的存储逻辑

文件系统接口是计算机技术领域用到的存储访问中最通用的、最普遍的接口，即便有些厂商或者软件存储直接提供了块设备（如ceph-RBD、SAN），当客户端使用时仍然还需要将这个块设备映射到内核中，分区、格式化创建文件系统、挂载才能使用它，所以能够直接使用裸设备作为存储的应用程序并不多；因此ceph直接还提供了CephFS这种文件系统接口给用户直接存储数据，避免上面的一系列（映射、分区、格式化、挂载等）操作。

1.经典(传统)文件系统概述(用于在原理上区别下面的CephFS)：
    1.1.索引式文件系统将底层存储数据的磁盘空间切分为"块"），并为每个文件对象所占用所有块建立一个索引表进行统一存放在一个称为"元数据区"的空间中：
        1.索引表就是块地址数组，每个数组元素就是块的地址，于是，一个文件对象的块可以分散存储到离散块空间中.
        2.索引表的索引结构称为inode，用于索引、跟踪一个文件对象的权限、隶属关系、时间戳和占据的所有的块等属性信息，不过却不包括文件名和文件内容本身
    1.2.文件名及其隶属的目录层级关系通过Dentry(目录项)进行描述：
        1.每个Dentry就像一个映射表，它保存了本级目录或者文件名以及紧邻的下一级目录或者文件的名称与各自inode的映射关系.
        2.而Dentry自身也需要由专用的inode对象承载，它也拥有自己的inode，于是这种映射便可组织出多级别层次来，这个多级别的层次起始于一个惟一的称之为“根”的起始点，从而形成一个树状组织结构.
2.CephFS的存储逻辑：如上图示
    1.CephFS为什么需要部署MDS以及MDS的作用：
        1.1.启用CephFS接口需要在基础的Rados-Cluster存储集群之上需要额外运行一个MDS（Metadata-Server元数据服务器）的守护进程，由于Rados-Cluster本身就是一个对象存储服务，Rados-Cluster是没办法分开管理上面所说的传统文件系统中的数据和元数据的功能；
        1.2.因此MDS是专用于模拟传统文件系统所应该具有的将数据和元数据分离存储的方式而专门提供的一个守护进程服务。而且MDS只用来管理元数据；CephFS依赖于专用的MDS（MetaData-Server）组件管理元数据信息并向客户端输出一个倒置的树状层级结构：
            1.将元数据先缓存于MDS的内存中然后再同步到Metadatapool,可以加快元数据的访问
            2.将cephfs的元数据的更新日志journal流式化后存储在RADOS集群上，journal日志是用来恢复mds里的元数据缓存。
            3.重启mds的时候会通过replay的方式从osd上加载之前缓存的元数据
            4.保存了文件系统的元数据(对象里保存了子目录和子文件的名称和inode编号)
        1.3.客户端(ceph.ko)-->MDS-->Rados-Cluster
            客户端要想使用ceph文件系统接口，内核级必须有对应的文件系统模块与之支持也就是ceph.ko模块，ceph.ko作为客户端每一次访问文件时必须基于套接字先连入到MDS来获取、返回文件的inode元数据信息，然后ceph.ko再到Rados-Cluster上以对象的方式把对象模拟成传统文件系统的"块"来加载文件的数据完成文件存取操作。
    2.CephFS需要元数据存储池(metadatapool)和数据存储池(datapool)
        1.不管是RBD、CephFS、RadosGW，在Ceph（Rados-Cluster）上所有数据都应该先放在存储池pool中再映射到OSD上，而CephFS文件系统管理其数据和元数据时是分别放置在不同的存储池中进行的；
        2.Metadata Pool
            1.所有的元数据都是由MDS守护进程管理的，可以理解为：MDS作为客户端连入Rados-Cluster的CephFS的metadata-pool存储池存储元数据；
            2.而元数据是一类很密集的IO访问，需要占用的空间不大，但是对于性能的要求比较高，所以Metadata-Server会在本地使用内存当做高速缓存，所有操作都先在内存中完成，一段时间后再同步到MetadataPool当中实现元数据存储的.
        3.datapool
            数据直接写入datapool存储池
        4.需要注意的是：CephFS的Matadatapool一定只能被MDS访问（客户端是不能直接访问的），客户端对元数据的访问必须经由MDS来实现：
            当客户端打开一个数据时，它首先会向MDS请求此数据的inode,inode信息会告诉客户端此数据所有对象存放datapool中的位置和编号，然后客户端基于编号就可以加载到所有对象并访问到数据了。
    3.客户端挂载CephFS访问Ceph集群的方法有2种：
        1.基于内核文件系统：
            即基于内核中的模块完成挂载，主要依赖ceph.ko模块和libcephfs库文件.
        2.用户空间文件系统：(FUSE客户端，需要内核):
            FUSE是指Filesystem-in-Userspace，主要用于非特权用户能够无需操作内核而创建文件系统，借助于libcephfs与Ceph集群交互.
3.部署、启用CephFS、客户端挂载的步骤：
    1.至少有一个节点运行ceph-mds守护进程
        可以通过部署Ceph集群时在管理节点使用ceph-deploy命令部署，部署完后默认处于standby状态，只有真正创建完CephFS的元数据存储池和数据存储池后才会转为Active模式；
    2.创建存储池：Metadatapool和datapool
        ceph osd pool create metadata-pool-name #创建元数据池
        ceph osd pool create data-pool-name     #创建数据池
    3.激活文件系统(也就是将MDS关联到两个存储池并使用)
        创建文件系统的命令格式：ceph fs new
            ceph fs new <fs-name> metadata-pool-name data-pool-name
        查看文件系统状态：
            ceph fs status <fsname>
            ceph mds stat
    4.创建CephFS客户端账号和secret文件    
        由于Ceph启用了CephX认证，所以CephFS的客户端完成授权认证后方可挂载访问文件系统，关于创建账号和授权见前文章，这里只列步骤：
        1.创建专用客户端账号，命令如下：
            ceph auth get-or-create <username> mon 'allow r' mds 'allow rw' osd 'allow rwx pool=<data-pool-name>' 
            注意：
                由于元数据池仅可由MDS访问，无须授权给客户端账号，所以在OSD部分只需要授权客户端访问数据池data-pool-name即可，而且数据池上的权限指定应根据实际需要进行。
        2.保存用户账号的密钥key信息于secret文件，用于客户端挂载操作认证之用
            由于CephFS客户端挂载时需要分开指定username和key信息，所以还需要将此用户的key信息单独导出放到secret文件中；
            1.先导出keyring密钥,通过keyring文件获取key
                ceph auth get <username> -o cluster-name.user-name.keyring
                #到出用户账号到指定名称的keyring中
            2.再将key保存到secret文件
                ceph authtool -p -n <username> cluster-name.user-name.keyring > <username>.key
                #保存到指定名称的secret中
            或者一步到位也行：
                ceph auth print-key <username> > <username>.key
        3.拷贝密钥文件到需要挂载CephFS的客户端主机上
            scp <username>.key root@server01:/etc/ceph/
            #拷贝到客户端主机
    5.1.基于内核的客户端挂载(需要key文件)
        1.客户端主机环境准备
            • 确保内核装载了ceph.ko模块  ~]$ modinfo ceph #装载ceph模块
            • 安装ceph-common程序包
            • 提供ceph.conf配置文件
            • 获取到用于认证的密钥文件
        2.挂载CephFS命令：
            mount -t ceph <MON1:Port>[,<MON2:Port>,<MON3:Port>] mount_point -o name=USERNAME,secretfile=/PATH/TO/USER-SECRET-KEY
            或者：
                mount -t <MON1:Port>[,<MON2:Port>,<MON3:Port>] mount_point -o name=USERNAME,secret=xxxx
            其中：
                1.挂载时使用逗号指定一个或多个monitorIP+Port，mount_point是指cephfs模拟出的根文件系统和客户端的挂载点，指明连接的用户和secret密钥文件路径。
                2.其实也可以secret也可以直接指定秘钥，而不是使用secretfile方式
        3.查看挂载状态
            stat -f mount_point
        4.写入/etc/fstab文件，实现自动挂载
            ~]$ vim /etc/fstab
                <MON1:Port>[,<MON2:Port>,<MON3:Port>]:/ /data cephname=fsclient,secretfile=/etc/ceph/fsclient.key,_netdev,noatime 0 0
            其中：
                _netdev选项是指挂载时间超时跳过否则主机无法正常启动，noatime是指为了提升文件系统性能建议不要实时更新访问的时间戳。
    5.2.基于FUSE客户端挂载(需要keyring文件)
        1.客户端主机环境准备
            • 安装ceph-fuse程序包
            • 获取到客户端账号的keyring文件和ceph.conf配置文件
        2.挂载CephFS
            ceph-fuse [-n client.username] [-m mon-ip-addr:mon-port] <mount-point> [OPTIONS] [--client_mountpoint/ -r <sub_directory>]
            其中：
                1.指定用户名、monIP+Port,mount-point和内核挂载时不同，这里只需要指定客户端的挂载点即可，ceph-fuse会默认使用keyring文件；
                2.-r root_directory把文件系统内的root_directory作为根挂载，而不是整个Ceph文件系统树。
        3.写入/etc/fstab，实现自动挂载
            #DEVICE PATH TYPE OPTIONS
            none <mount-point> fuse.ceph ceph.id=<user-id> [,ceph.conf=<path>],_netdev,defaults 0 0
            注意：
                使用fuse挂载时是没有设备的，所以是none,挂载类型时fuse.ceph,选项有用户名、ceph.conf配置文件，不要指明keyring文件因为它会自动在/etc/ceph目录下查找，_netdev同上原理。
        4.卸载：
            fusemount -u <mount_point>
    6.总结：相对于NFS来说
        1.用于底层Rados功能，使用副本池拥有冗余能力，不存在单点
        2.底层Rados由N个节点主组成，可以分散IO和增加节点增加IO扩展性更高
        CephFS还可以在前端虚拟成NFS使用，但一般不这么做，可以参考：
            https://mp.weixin.qq.com/s/1TMfqxxE5I2eIhtXPPoRpQ

6.2.CephFS动态子树分区概念和多活MDS高可用原理&实现

Ceph论文译文参考

–CephFS-MDS动态子树分区逻辑

–多活MDS架构

MDS是CephFS的瓶颈？
    CephFS文件系统是将数据分为元数据Metadata和数据Data分开存放的，Datapool可以随着底层Rados-Cluster的OSD增加完成I/O的自动扩展，Metadatapool也可以扩展，但是从客户端-->MDS-->Metadatapool流程可以知道MDS是Metadatapool的唯一访问入口，当有大量的CephFS客户端同时访问MDS时，Metadatapool可能不会成为瓶颈而MDS服务确有很大可能成为CephFS的瓶颈!
单纯增加MDS节点无法分担写操作从而达到扩展瓶颈的效果!
    CephFS的扩展实际上还是对MDS守护进程的扩展，MDS保存的是文件系统的元数据，而文件系统的元数据其实就是一个树状结构（如上图）；比如当有2个MDS时，当客户端通过MDS01写操作和在MDS02上读操作是同一个文件时，就必须保证这两个MDS是互相同步保持一致（相当于是主从复制的副本池），这样一来不管写到哪个MDS都需要同步给其他MDS，写操作没有任何分散负载效果，所以单纯的增加MDS节点没有意义。
1.元数据分区的解决方案：
    如上说的单纯增加元数据节点的数量是无意义的，所以为了实现文件系统数据和元数据的负载均衡，分布式文件系统业界提供了将名称空间分割治理的解决方案，方案基本思想是：通过将文件系统根树及其热点子树分别部署于不同的元数据服务器进行负载均衡，从而赋予了元数据存储线性扩展的可能，一般有几种分区方法：
        1.静态子树分区
            1.1.静态子树分区的思想是指：通过手工分区方式，把A主机上的根文件系统的一个或多个子目录树如：d1挂载到另外B主机的文件系统上（如NFS），那么所有写在A-d1子树目录下的都会写到了B主机的文件系统目录下，即把一个子目录关联到另外一个分区上去从而减轻了当前文件系统的负载；"静态"子树是指只能减轻被挂载的那个子目录（A-d1）的负载，无法减轻A主机上的其他子目录的负载。
            1.2.静态子树分区方案在经过长时间数据存取后肯定会出现元数据负载不均衡的情况，可以由管理员手动重新进行分配来解决，所以静态子树分区方式适应于数据位置固定的场景，不适合动态扩展、或者有可能出现异常的场景。
        2.静态hash分区
            2.1.静态hash分区是指对A文件系统的目录或者文件名进行路径hash计算来分配数据存储的位置。
            2.2.这种方式适合数据分布均衡、且需要应用各种异常的情况，但不太适合需要数据分布固定、环境变化频率很高的场景。
        3.惰性混编分区
            惰性混编分区是指将静态hash分区和传统的文件系统结合使用，不做详细说明
        4.动态子树分区
            通过实时监控集群节点的负载，动态调整子树分布于不同的节点。
            这种方式适合各种异常场景，能根据负载的情况，动态的调整数据分布，不过如果大量数据的迁移肯定会导致业务抖动，影响性能。
2.Ceph动态子树分区
    1.动态是相对于静态而言，静态是将文件系统的某个子目录上的元数据指定由固定的某一个MDS来管理，而"静态"的缺点是肯定会带来元数据分布不均的情况，所以为了避免这种情况，使用"动态子树方式"按照当前每个子目录的实际负载来动态迁移分配当前子树的元数据到其他MDS。
    2.Ceph就是使用动态子树分区方式，支持多个MDS同时工作，如上图1示：
        1.其中一个（MDS1）负责根，根上的某个子树动态的分配给第二个MDS服务器，以此类推...
        2.Rank级别：
            MDS是有等级划分的，负载根文件系统管理的MDS的Rank为0级别，Rank1、rank2...它们都属于Rank0的子树。
        3.MDS的冗余实现，如上图1示
            多个MDS此时只是分担元数据负载而不具有数据冗余能力，虽然元数据是存到Metadata-pool中的，即使管理它的MDS宕了其他MDS可以加载但速度非常慢（元数据是小而密集的IO请求），所以冗余方案有：
            1.公共冗余MDS(备用MDS、standby)：
                公共备用MDS会代替任何一个可能出现故障的主MDS
            2.每个MDS配置一个单独的冷备MDS：
                冷备MDS的逻辑和存在的问题：主MDS宕机了，冷备MDS需要从Metadatapool中加载数据重新提供服务，中间肯定会出现卡和切换耗时比较高的问题。
            3.每个MDS配置一个在线备用MDS(热备MDS)：
                在线备用MDS(热备MDS)的逻辑是：主MDS的元数据操作不但要同步到内存中，同时也要同步给在线备用的MDS，在线备用MDS虽然不提供服务但实时更新数据，这样主从切换减少了数据加载的步骤就会大大减少切换耗时.
    3.Ceph的MDS集群是基于动态子树分区策略,自适应地分配缓存元数据，分层分布在一组节点上, 如上图2所示：
        1.通过使用计数器统计每个MDS中元数据的访问量。任何操作使得受影响inode及其上层节点直到根目录的计数都增加,从而提供每个MDS一个权值，来描述最近的载荷分布。定期比较MDS权值, 通过迁移以保证元数据工作负载均匀分布。共享的永久存储和及其名空间锁的结合保证这样的迁移是可行的，转移一些内存缓存的内容到新的节点, 对相干锁或客户端功能影响最小。新导入元数据将写入新MDS的日志中,同时，还有日志分录新旧MDS以确保迁移是安全的（类似于两阶段提交）。子树分区保持以最小化前缀复制开销同时也要保护本地性。
        2.跨多个MDS节点复制元数据时,inode内容分为三组,每个都有不同的语义一致性:安全（owner,mode）、文件（size,mtime）、不可变性质（inode-number,ctime,layout), 当不可变性质不变，安全和文件锁被独立的状态机管理，都拥有不同状态集，并且根据不同的访问和更新在不同状态之间转换。例如：
            onwer和mode用于路径访问的安全检查，很少改变，只需要很少几个状态，但是客户端多种访问模式的集合，因为它反应在MDS中，可以控制客户端的访问能力。
3.单活MDS及其高可用和多活MDS(multi-MDS)
    综上，MDS作为CephFS的唯一入口，所以：1.MDS进程必须要做冗余；2.如果只有一个MDS进程那么它会成为一个较重文件系统的瓶颈，为了避免其成为瓶颈我们使用元数据分区的动态子树分区方式实现多个MDS同时工作（多活MDS，multi-mds），避免每个MDS成为单点还需要对每个MDS实现高可用；所以MDS的部署方案就有3种方式：
    1.单活MDS
        只有一个MDS，不安全且易成为CephFS文件系统的瓶颈，不推荐
    2.单活MDS高可用
        对外提供服务只有一个active-mds,多个standby-mds，当active-mds挂掉，standby-mds会立马接替，保证集群高可用性；而standby-mds有2种备用方案：
        1.冷备MDS
            冷备MDS其实只是准备了一个备用的mds进程，与主进程保持心跳关系，并不备份lru元数据，一旦主的mds挂了，备份mds会从Metadatapool中replay()元数据到缓存，replay肯定会消耗一定的时间，主从切换一定会有延迟。
        2.热备MDS
            热备除了准备了备用的mds进程，元数据缓存先写入主MDS，然后由主mds时时刻刻同步给热备MDS，当主的mds挂了，热备的mds直接变成主mds，并且没有replay()的操作，所以主从切换消耗的时间非常少。
    3.多活MDS(multi-MDS)
        使用元数据分区的动态子树分区方式实现，多个MDS同时工作分担前端的读写操作，避免MDS称为CephFS的性能瓶颈，这也是CephFS在Luminous版本中多元数据服务器（Multi-MDS）的功能和目录分片（dirfragment）的功能特性。

4.多活MDS(multi-MDS、active-activr MDSs)-->重要
    1.多主MDS模式是指CephFS将整个文件系统的名称空间切分为多个子树并配置到多个MDS之上，不过，读写操作的负载均衡策略分别是子树切分和目录副本，意思是：
        1.将写操作负载较重的目录切分成多个子目录到其他active-MDS以分散负载.
        2.为读操作负载较重的目录创建多个副本到多个MDS以均衡负载.
    2.子树分区和迁移的决策是一个同步过程，各MDS每10秒钟做一次独立的迁移决策，每个MDS并不存在一个一致的名称空间视图，且MDS集群也不存在一个全局调度器负责统一的调度决策;
    3.各MDS彼此间通过交换心跳信息（HeartBeat，简称HB）及负载状态来确定是否要进行迁移、如何分区名称空间，以及是否需要目录切分为子树等
        管理员也可以配置CephFS负载的计算方式从而影响MDS的负载决策，目前，CephFS支持基于CPU负载、文件系统负载及混合此两种的决策机制
    4.动态子树分区依赖于共享存储完成热点负载在MDS间的迁移，于是Ceph把MDS的元数据存储于后面的RADOS集群上的专用存储池中，此存储池可由多个MDS共享
        MDS对元数据的访问并不直接基于RADOS进行，而是为其提供了一个基于内存的缓存区以缓存热点元数据，并且在元数据相关日志条目过期之前将一直存储于内存中
    5.为了确保多个MDS之间能够共享数据，CephFS使用元数据日志来解决容错问题：
        1.即每个活跃的MDS都是通过日志保存在底层的Metadatapool中实现的，这个Metadatapool类似于LFS（Log-Structured File System）和WAFL（ Write Anywhere File Layout）的工作机制
        2.CephFS元数据日志文件的体积可以无限增长以确保日志信息能顺序写入RADOS，并额外赋予守护进程修剪冗余或不相关日志条目的能力
    6.每个Ceph上可以创建多个CephFS文件系统（但建议只创建一个），而且每个CephFS都会有一个易读的文件系统名称和一个称为FSCID标识符ID，并且每个CephFS都应该配置至少一个Active MDS守护进程对外提供服务。
    7.一个MDS集群中可处于Active状态的MDS数量的上限由ceph.conf中max_mds参数配置，它控制着可用的rank数量，默认值为1；
        1.rank是指CephFS上可同时处于Active状态的MDS守护进程的可用编号，其范围从0到max_mds-1；
        2.一个rank编号意味着一个可承载CephFS层级文件系统目录子树元数据管理功能的Active状态的ceph-mds守护进程编制，max_mds的值为1时意味着仅有一个0号rank可用；
        3.刚启动的ceph-mds守护进程没有接管任何rank，它随后由MON按需进行分配
        4.一个ceph-mds一次仅可占据一个rank，并且在守护进程终止时将其释放
        5. 一个rank可以处于下列三种状态中的某一种：
            1.Up：rank已经由某个ceph-mds守护进程接管
            2.Failed：rank未被任何ceph-mds守护进程接管
            3.Damaged：rank被占用且占用的ceph-mds守护进程处于故障状态，其元数据处于崩溃或丢失状态
                在管理员修复问题并对其运行"ceph mds repaired"命令之前，处于Damaged状态的rank不能分配给其它任何MDS守护进程.
    8.增加rank值以配置active-MDS数量
        要想实现在Ceph集群中配置多个active-MDSs，需要更改系统默认的rank=1的值；
        1.增加Active MDS的数量的命令：
            ceph fs set <fsname> max_mds <number>
            注意：
                1.仅当存在某个备用守护进程可供新的级别使用时，文件系统中的实际级别数才会增加；
                2.多活MDS的场景依然要求存在备用的冗余主机以实现服务HA，因此max_mds的值总是应该比实际可用的MDS数量至少小1；
        2.降低Acitve MDS的数量的命令：
            ceph mds deactivate {System:rank|FSID:rank|rank}
            注意：减小max_mds的值仅会限制新的rank的创建，对于已经存在的Active MDS及持有的rank不造成真正的影响，因此降低max_mds的值后，管理员需要手动关闭不再被需要的rank；
        3.变动max_mds的值后查看其状态
            ceph status <fsname>
    9.手动设置MDS的管理故障转移机制
        出于冗余高可用的目的配置一定数量Standby状态的ceph-mds守护进程如何接管、怎么接管失效的MDS，称为MDS管理故障转移机制，而且CephFS提供了四个选项参数可以手动控制Standby状态的MDS守护进程如何工作：
        1.mds_standby_replay：也就是上面说的热备MDS高可用模式
            1.其值为布尔型值，true表示当前MDS守护进程将持续读取某个特定的Up状态的rank的元数据日志，从而持有相关rank的元数据缓存，并在此rank失效时加速故障切换；
            2.一个up状态的rank仅能拥有一个replay守护进程，多出的会被自动降级为正常的非replay型MDS
        2.mds_standby_for_name：设置备用的MDS进程仅备用于指定名称的MDS(rank)；
        3.mds_standby_for_rank：两种用法
            1.此选项不但可以设置此备用MDS进程仅备用于指定的rank，它不会接替任何其它失效的rank；
                即有rank0和1时，只设置为rank0的备用MDS，rank1即使故障它也不会接替
            2.此选项还可以在当一个Ceph集群上创建了多个CephFS文件系统时，他还可以为跨多个文件系统为多个rank进行冗余，但需要联合使用下面的参数来指定；
                即只设置为指定fscid的文件系统中的rank0的备用MDS。
        4.mds_standby_for_fscid：联合mds_standby_for_rank参数的值协同生效;
            同时设置了mds_standby_for_rank：备用于指定fscid的指定rank；
            未设置mds_standby_for_rank时：备用于指定fscid的任意rank；

5.部署单活MDS高可用和多活MDS高可用
    此前部署Ceph时只有一个active-mds，在此基础上实现单活高可用和多活MDS高可用
    ~]$ ceph mds stat
        cephfs-1/1/1 up {0=mon01=up:active} #当前集群只有一个active-mds=mon01
    1.单活MDS高可用
        ~]$ ceph-deploy mds create mon02 mon03 #将mon02和mon03也部署为mds节点
        ~]$ ceph fs status cephfs 
            +-------+--------+--------+-----------+-----+------+
            |  Rank | State  |   MDS  | Activity  | dns | inos |
            +-------+--------+--------+-----------+------+-----+
            |   0   | active | stor01 | Reqs: 0/s | 10  |  13  |
            +-------+--------+--------+-----------+-----+------+
            +--------------+
            | Standby MDS  |
            |     mon02    |
            |     mon03    | 此时就是一个单活的MDS高可用，mon2和mon3会在
            +--------------+  mon01故障时接替mon01
    2.多活MDS高可用
        ~]$ ceph fs set cephfs max_mds=2    #设置活动的MDS数量为2
        ~]$ ceph fs status cephfs 
            +-------+--------+--------+-----------+-----+------+
            |  Rank | State  |   MDS  | Activity  | dns | inos |
            +-------+--------+--------+-----------+------+-----+
            |   0   | active | mon01  | Reqs: 0/s | 10  |  13  |
            |   1   | active | mon02  | Reqs: 0/s | 10  |  13  |
            +-------+--------+--------+-----------+-----+------+
            +--------------+
            | Standby MDS  |
            |  mon03       | 
            +--------------+ 
            此时只有一个备用MDS：mon03,可以通过管理故障转移机制设置备用模式
        ~]$ vim /etc/ceph/ceph.conf
            ...
            [mds.mon03]
            mds_standby_for_fscid=cephfs
            意思是指明mon03为名为cephfs文件系统的备用mds，由于cephfs有两个活动mds,所以不管是mon01还是mon02故障，mon03都会以冷备mds的方式接管，但切换时间会较长；
        ~]$ vim /etc/ceph/ceph.conf
            ...
            [mds.mon03]
            mds_standby_for_name=mon01
            mds_standby_replay=true
            [mds.mon04]
            mds_standby_for_name=mon02
            mds_standby_replay=true
            #如果有2个activemds,2个standbymds，可以通过上面两个选项来指定各自的备用mds和其模式为热备，以实现故障切换时加速切换速度.
