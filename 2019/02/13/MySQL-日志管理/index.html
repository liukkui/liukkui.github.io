Mysql的日志管理

1.对于一个数据库管理系统来讲，它的日志文件种类非常多，这些日志文件表现在mysql物理视图上的各类文件；而且Mysql还有很多种不同类型的存储引擎,每种存储引擎的也会有自己的日志,比如：
    对于MyISAM来说每个表都有三个文件（*.MYD,*.MYI,*.frm），对于InnoDB来说每个表有两个文件（*.ibd,*.frm）这些都是真正的存储的数据文件，为了保证mysql服务器正常运行和记录mysql运行中产生的事件、错误等，mariadb有诸多类型的日志功能，这些日志文件最终还要表现为存储的一种文件的，所以mariab除了数据、索引文件之外还有第三类文件
2.Mysql中日志可以按照功能分为如下几类：
    1.通用日志(也叫查询日志)：query log(general log),默认未开启
    2.错误日志:error log
    3.慢查询日志:slow query log
    4.二进制日志:binary log
    5.中继日志:reley log
    6.innodb存储引擎级的事务日志：transaction log
        事务日志又分:redo log和undo log

1.通用日志(也叫查询日志)：query log(general log),默认未开启

通用日志的重要说明：
    1.查询日志在mysql中被称之为general log（通用日志），不要被"查询日志"的名字误导，错误的以为查询日志只会记录select语句;其实不然,查询日志记录了数据库执行的命令，不管这些语句是否正确都会被记录，这也是general log之所以"通用"的原因；
    2.默认情况下Mysql的查询日志是未开启的,因为当开启查询日志后,由于数据库操作命令有可能非常多而且执行比较频繁,据库可能需要不停的写入查询日志，这样会增大服务器的IO压力，增加很多系统开销；但是开启查询日志可以帮助我们分析问题,比如：分析哪些语句执行密集，执行密集的select语句对应的数据是否能够被缓存等;但可以根据实际情况决定是否开启查询日志，如果需要可以手动开启;
1.查询日志的3个相关参数：
    1.general_log=ON|OFF
        表示查询日志是否开启，ON表示开启，OFF表示未开启，默认为OFF
    2.log_output=FILE|TABLE|FILE,TABLE|NONE
        表示当查询日志开启以后，以哪种方式存放，在mysql中有3种方式存储查询日志：
        1.FILE：
            file是指将查询日志存储在指定的日志文件中,可以结合general_log_file参数设定日志文件名;
        2.TABLE:
            table是指将查询日志存储在mysql库的general_log表中;
        3.FILE,TABLE
            FILE,TABLE"是指将查询日志同时存放于指定的文件与general_log表中，不过一般不会进行这样的设置，因为这样会徒增很多IO压力;
        需要注意的是:log_output不止控制查询日志的输出，慢查询日志的输出也是由此参数进行控制，也就是说log_output设置为file，就表示查询日志和慢查询日志都存放到对应的文件中，设置为table，查询日志和慢查询日志就都存放在对应的数据库表中;如果需要开启查询日志,建议设置为"table";
    3.general_log_file=/path/file
        当log_output设置成file或者file,table时,可以通过此参数指定查询日志的文件;默认情况下，此文件名与mysql所在的主机名相同,而且此处是相对位置,是相对于my.cnf中datadir变量所对应的目录位置,相对位置默认是/var/lib/mysql目录,如果安装mysql时更改了mysql数据目录,以实际目录为准;
2.开启查询日志并设置存储为file
    1.因为在Mysql中日志的参数都是全局变量，所以设置这些变量时需要加上global关键字,而且生产环境中需要要将下面这些配置写入my.cnf配置文件中;
        > set global general_log=ON;         #开启查询日志
        > set global log_output="file";      #将查询日志存储为file
        写在my.cnf配置文件中：
            ~]# vim /etc/my.cnf
            [mysqld]
            general_log=ON;
            global log_output="file"
            general_log_file=/data/mysql3/mysql-master.log
    2.查看设置是否生效
        MariaDB [(none)]> show global variables like '%gen%';
        +------------------+------------------+
        | Variable_name    | Value            |
        +------------------+------------------+
        | general_log      | OFF              |
        | general_log_file | mysql-master.log |#日志文件名默认为所在主机名.log
        +------------------+------------------+
        MariaDB [(none)]> show global variables like 'log_output';
        +---------------+-------+
        | Variable_name | Value |
        +---------------+-------+
        | log_output    | FILE  |
        +---------------+-------+

2.错误日志：error log

需要说明的是：错误日志不仅仅记录了mysql执行过程中的错误信息,还会记录其他信息,错误日志会记录如下信息：
    1.mysql执行过程中的错误信息,称为错误日志的主要原因
    2.mysql服务器启动、关闭所产生的事件信息,但未必是错误信息
    3.event scheduler运行一个event时产生的日志信息;
    4.mysql主从复制结构中,从服务器IO复制线程产生的启动信息、错误信息
    所以说,错误日志的作用不仅仅与记录错误，它是非常重要的信息日志文件;
1.错误日志的2个相关参数：
    1.log_error：
        log_error变量用于设置错误日志的存放位置,如果想要修改错误信息位置并且永久生效需要将该参数写在my.cnf配置文件中;
    2.log_warnings=0|1|2
        log_warnings变量的值也与错误日志有关,log_warnings作用是否将警告信息也记录到错误日志文件中,它的值有3种：
        1.值为0：0表示不记录警告信息
        2.值为1:表示警告信息一并记录到错误日志中,默认是1
        3.值为2：表示"失败的连接"的信息和创建新连接时"拒绝访问"类的错误信息也会被记录到错误日志中;
        同样,如果想要永久使配置生效，需要将该参数写在my.cnf配置文件中比如：log_warnings=2
2.定义错误日志
    使用命令配置都是临时有效的，要想永久有效就需要写在my.cnf配置文件中:
    ~]# vim /etc/my.cnf     #将配置写在my.cnf配置文件中
        [mysqld_safe]
        log-error=/data/mysql3/error_log.log 
        log_warnings=2
    MariaDB [(none)]> show GLOBAL VARIABLES like '%log_error%';
        +---------------+----------------------------+
        | Variable_name | Value                      |
        +---------------+----------------------------+
        | log_error     | /data/mysql3/error_log.log |
        +---------------+----------------------------+
    MariaDB [(none)]> show GLOBAL VARIABLES like '%log_warning%';
        +---------------+-------+
        | Variable_name | Value |
        +---------------+-------+
        | log_warnings  | 2     |
        +---------------+-------+

3.慢查询日志:slow query log

需要说明的是：
    1.慢查询日志不只是记录执行比较慢SELECT的sql语句,还包括INSERT、DELETE、UPDATE等DML操作;
    2.慢查询并不一定是查询语句自己执行速度慢导致的，有可能是这个查询语句所需要查询依赖的表被施加了写锁阻塞了查询;
    3.mysql会把执行超过10秒的语句定义为慢查询语句;在生产环境中,如果线上请求超时,应该去看一下慢查询日志,先找到慢查询日志中执行时间较长的语句,然后利用mysqldumpslow命令去分析原因;
1.慢查询日志的几个重要参数
    1.slow_query_log和log_slow_queries=[ON|OFF]
        这两个参数都是定义是否开启慢查询日志的,不过在mysql5.6之前的版本是使用log_slow_queries指定是否开启慢查询日志,在mysql5.6之后的版本都是使用slow_query_log代替log_slow_queries;
    2.log_output=FILE|TABLE|FILE,TABLE|NONE
        同上文的通用日志中的参数,log_output既用于控制查询日志的输出类型也控制慢查询日志的输出类型;慢查询日志建议输出为file;
    3.slow_query_log_file:
        当log_output定义为file时,可以使用此参数定义慢查询日志文件及其路径;同样,路径是my.cnf中datadir相对路径,慢查询的日志文件名默认是以mysql所在主机名来命名的："主机名-slow.log";
    4.long_query_time
        表示"多长时间的查询"被认定为"慢查询"，此值得默认值为10秒，表示超过10秒的查询被认定为慢查询
    5.log_queries_not_using_indexes
        表示如果运行的sql语句没有使用到索引,是否也被当做慢查询语句记录到慢查询日志中，OFF表示不记录，ON表示记录
    6.log_throttle_queries_not_using_indexes=[ON|OFF]
        此参数是mysql5.6.5版本新引入的参数，当log_queries_not_using_inde设置为ON时，没有使用索引的查询语句也会被当做慢查询语句记录到慢查询日志中，使用log_throttle_queries_not_using_indexes可以限制这种语句每分钟记录到慢查询日志中的次数，因为在生产环境中，有可能有很多没有使用索引的语句，此类语句频繁的被记录到慢查询日志中，可能会导致慢查询日志快速不断的增长，管理员可以通过此参数进行控制;
2.开启慢查询日志并设置慢查询时间为5s
    使用命令配置都是临时有效的，要想永久有效就需要写在my.cnf配置文件中:
    ~]# vim etc/my.cnf
        slow_query_log=ON
        slow_query_log_file=/data/mysql3/master-slow.log
        long_query_time=5
    MariaDB [(none)]> show GLOBAL VARIABLES like '%slow%';
    +-------------------------------+------------------------------+
    | Variable_name                 | Value                        |
    +-------------------------------+------------------------------+
    | slow_query_log                | ON                           |
    | slow_query_log_file           | /data/mysql3/master-slow.log |
    | log_slow_rate_limit           | 1                            |
    | log_slow_filter               |                              |
    | log_slow_queries              | ON                           |
    | log_slow_verbosity            | Query_plan,explain           |
    | log_queries_not_using_indexes | OFF                          | 
    +---------------------------+----------------------------------+
    MariaDB [(none)]> show GLOBAL VARIABLES like 'long_query_time';
    +-----------------+-----------+
    | Variable_name   | Value     |
    +-----------------+-----------+
    | long_query_time | 5.000000  |
    +-----------------+-----------+
    MariaDB [(none)]> show global status like '%slow_queries%';
    +---------------+-------+
    | Variable_name | Value |
    +---------------+-------+
    | Slow_queries  | 0     |
    +---------------+-------+
    开启慢查询日志后,如果有记录可以通过该命令查看有多少条慢查询语句;
3.mysqldumpslow命令
    mysqldumpslow命令是mysql自带的慢查询日志统计分析工具,通过mysqldumpslow命令我们可以更加方便的从不同的维度对慢日志进行排序、查找、统计;但是要想使用mysqldumpslow命令,log_output的值必须设置成file,而不能是table;
    ~]# mysqldumpslow -h
        -v:输出详细信息
        -t NUM:输出慢查询日志中前多少行的记录
        -g PATTERN：通过正则表达式匹配我们想要查看的内容
        -s Order:-s代表排序,输出的记录可以按照下面这些参数进行排序：
            c: 执行计数
            l: 锁定时间
            r: 返回记录
            t: 执行时间
            al:平均锁定时间
            ar:平均返回记录数
            at:平均执行时间
        其中：
            默认是按照平均查询时间at进行排序的,在使用中会经常用到按照c：查询次数来排序,因为查询次数很频繁但是时间不高也是有必要优化的
    示例：
        ~]# mysqldumpslow -v -s t -t 10 master-slow.log

4.二进制日志:binary log

1.二进制日志说明
    1.Mysql中的二进制日志记录了所有修改了数据库的语句或者有可能会改变数据库的语句,也就是说select、show这种不修改数据库的语句二进制日志是不会进行记录的;由于它是以二进制格式保存的，所以叫二进制日志;
    2.二进制日志文件不能使用cat等命令查看,需要使用二进制日志的客户端命令工具mysqlbinlog来查看;
    3.二进制日志的主要作用是：
        1.用于mysql主从架构中,从节点从主节点复制同步数据;
        2.用于mysql基于时间点的数据备份和恢复；
    4.为了安全,建议将二进制日志文件和mysql的数据分开存放到不同的磁盘上,而且这些磁盘最好都做raid10来保证数据安全;
2.二进制日志binlog是怎样用于时间点恢复的？
    1.二进制日志最重要的作用之一就是解决在全量或增量备份到下次备份期间数据库崩溃时的这一段未备份数据的恢复!比如：
        1.在周一0点进行全量备份后,周一12点数据库突然崩溃了,而0~12点间没有任何备份计划,此时就只能通过二进制日志来恢复0~12点之间的数据,因为上次完全备份到数据库崩溃之时之间的所有操作都会记录在二进制日志中,我们只需要将该时间点后的日志"重放一遍"即可恢复到数据库崩溃之时的样子;
        2.借助二进制日志恢复数据需要使用mysqlbinlog命令和二进制日志文件,详见下文恢复示例;
    2.而且mysql中的二进制日志binlog与oracle中的归档日志作用是一样的;
3.二进制日志的3种记录格式：
    记录在二进制日志中的格式是基于"语句"还是"行",在有恢复数据的必要情况下还是有很大区别的,所以mysql二进制日志记录提供了三种格式：
    1.基于"语句"记录：statement
        假设A表有1000条数据,当执行update A set test='test'语句时,如果使用statement模式,那么二进制日志中只会记录这条update语句,而不会记录被修改的1000行数据;可以看出statement模式的优点是binlog日志量少、IO压力小、性能较高;缺点是为了能够尽量的完全一致的还原操作,除了记录语句本身以外,可能还需要记录一些相关的信息，而且在使用一些特定的函数时，并不能保证恢复操作与记录时完全一致;
    2.基于"行"记录：row
        还是以A表有1000条数据,当执行update A set test='test'语句为例,如果使用row模式,当执行这条update语句以后,由于1000条数据都会被修改,所以会有1000行数据被记录到二进制日志中;可以看出row模式的优点是能够完全的还原或者复制日志被记录时的操作,缺点是记录日志量较大、IO压力大、性能消耗较大;
    3.混合模式：mixed
        混合使用上述两种模式,一般的语句使用statment方式进行保存，如果遇到一些特殊的函数，则使用row模式进行记录，这种记录方式被称之为mixed，看上去这种方式似乎比较完美,但是在生产环境中,为了安全我们都会使用row模式;
    可以通过binlog_format变量设置二进制日志的记录方式,为了使配置永久生效需要写在my.cnf配置文件中：binlog_format=row
4.二进制日志的几个重要参数：
    1.sql_log_bin=[ON|OFF]
        此变量表示是否记录二进制日志,默认ON；需要特别注意的是:
            1.sql_log_bin是一个会话级别的变量，只能在当前会话中使用set sql_log_bin命令进行设置,而不能使用set global sql_log_bin命令进行设置;
            2.sql_log_bin也不能写在my.cnf文件中,否则可能会无法启动mysql;
    2.log_bin=/PATH/BIN_LOG_FILE
        关于此参数项有几点需要强调说明：
            1.log_bin表示二进制日志存放的路径和二进制日志文件的文件名前缀,不能是ON或OFF（很多人以为是ON或OFF）,所以完整的二进制文件名是BIN_LOG_FILE.00001这种格式的;
            2.只有sql_log_bin=ON和log_bin=/PATH/BIN_LOG_FILE同时都设置了,mysql才能记录二进制日志：
            3.log_bin是一个只读变量!这就意味着不能通过set命令设置,只能通过写入my.cnf文件中;
    3.binlog_format=STATEMENT|ROW|MIXED
        二进制日志记录的格式，默认是STATEMENT,强烈建议改成row
    4.max_binlog_size=1073741824：
        单个二进制日志文件的最大大小,默认为1G,以字节为单位;到达最大值会自动滚动,也就是后缀名编号自动加1（*.000001、*.000002这种格式）;
    5.sync_binlog=1|0
        sync_binlog和事务日志中redo_log的innodb_flush_log_at_trx_commit参选选项是一样的都是设置将内存数据同步到磁盘上的刷写策略,此参数项的值有0和1两个:
            1.值为1：表示每1次事务提交之后，都会立即将内存中的二进制日志立即同步到磁盘中的二进制日志文件中；
            2.值为0：表示当事务提交之后，mysql不会立即将内存中的binlog刷写到磁盘中的binlog日志文件中，而是由文件系统决定什么时候刷写，这可能取决于文件系统的缓存机制;此值设置为0时,一旦操作系统宕机,那么将丢失未从内存中同步到磁盘中的binlog;
        所以当此值设置为0时,安全性最差,但是性能最高;当此值设置为1时,安全性最高,性能最差;所以强烈建议将此参数值设置为1;
    6.expire_logs_days=0:
        二进制日志可以自动删除的天数,默认是0,0表示不启用根据时间自动删除的功能;
5.生产环境建议设置二进制日志的参数如下：
    ~]# vim /etc/my.cnf
        [mysqld]
        log_bin=mybinlog          #二进制日志的文件名前缀(相对路径)
        binlog_format=row
        max_binlog_size=524288000 #设置满500M时滚动
        sync_binlog=1
    MariaDB [(none)]> show GLOBAL VARIABLES like '%log_bin';
        +---------------+-------+
        | Variable_name | Value |
        +---------------+-------+
        | log_bin       | ON    |
        | sql_log_bin   | ON    |
        +---------------+-------+  
        注意：只有这两个参数同时为ON时,mysql才会记录二进制日志
    MariaDB [(none)]> show GLOBAL VARIABLES like '%binlog%';
        +-----------------------------------------+----------------------+
        | Variable_name                           | Value                |
        +-----------------------------------------+----------------------+
        | binlog_format                           | ROW                  |
        | max_binlog_size                         | 524288000            |
        | sync_binlog                             | 1                    |
        +-----------------------------------------+----------------------+
6.查看、清除二进制日志的相关命令
    1.查看mysql产生的所有二进制日志文件列表命令：
        show binary logs;或者show master logs;
    2.查看mysql当前正在使用的二进制日志文件的命令
        show master status;
    3.查看指定二进制日志文件中的事件的命令：
        show binlog events in '二进制日志文件' from position limit [offset,] [row_count]
        注：详细使用见下文示例
    4.清除指定二进制日志的命令：
        purge {binary|master} logs {to '二进制日志' | before datetime_expr }
        注：详细使用见下文示例
    5.删除所有二进制日志，index文件重新记数
        RESET MASTER [TO #];
            #删除所有二进制日志文件，并重新生成日志文件，文件名从#开始记数，默认从1开始，一般是master主机第一次启动时执行，MariaDB10.1.6开始支持TO #
    6.滚动生成新的二进制日志文件
        FLUSH LOGS; 
            #也就是说不管二进制日志是否达到设置的max_binlog_size大小,使用flush命令也可以生成新的二进制日志文件;它和备份工具mysqldump的--flush-log选项的作用是类似的;
    示例：
        1.MariaDB [(none)]> show master logs;   #查看所有的二进制日志文件列表
        +------------------+-----------+
        | Log_name         | File_size |
        +------------------+-----------+
        | mysql-bin.000001 |       264 |
        | mysql-bin.000002 |     21930 |
        | mysql-bin.000003 |       486 |
        +------------------+-----------+
        2.MariaDB [(none)]> show master status; #mysql正在使用的二进制日志文件

7.mysqlbinlog命令和二进制日志中记录的信息说明： 
    1.mysqlbinlog命令
        1.除了在mysql提示符中使用show binlog events命令查看日志内容以外,还可以使用二进制日志的客户端命令工具mysqlbinlog查看对应的二进制日志；
        2.mysqlbinlog的命令格式：
            ~]# mysqlbinlog --help
            Usage: mysqlbinlog [options] log-files
            常用选项：
                --start-position=#    指定开始位置
                -stop-position=#      指定结束位置
                --start-datetime=     指定从哪个时间开始查看
                --stop-datetime=      指定查看的结束时间
                    时间格式：YYYY-MM-DD hh:mm:ss 
                --base64-output[=name]
                -v | -vvv
        示例：
            1.查看二进制日志mysql-bin.000003中的信息
                ~]# mysqlbinlog mysql-bin.000003
            2.查看二进制日志mysql-bin.000003中从245~416段的信息
                ~]# mysqlbinlog --start-position=245 --stop-position=416 mysql-bin.000003
            3.查看2018年9月2日20点00以后的日志信息
                ~]# mysqlbinlog --start-datetime "2018-9-2 20:00:00" mysql-bin.000003
    2.二进制日志中记录的信息说明： 
        如下图示是二进制日志mysql-bin.000003的文件内容,其中：
        1.第一部分是每个二进制日志文件固定的开头内容包括一些版本信息和属性信息,此部分不需要关注;
        2.第二部分是重点关注部分：
            每个at #：代表一个事件
            事件发生的日期和时间: 201107 23:24:21
            事件发生的服务器标识：server id 1
            事件的结束位置：end_log_pos 316,也代表下一个事件的开始标识
            事件的类型：Query,广义上来讲所有语句都必须先执行Select,所以这里都是查询
            事件发生时所在服务器执行此事件的线程的ID：thread_id=19
            语句的时间戳与将其写入二进制文件中的时间差：exec_time=0 
            错误代码：error_code=0 
            事件内容：后面三项都属于事件的内容
            GTID（mariadb10以上版本专属属性）：
                Global Transaction ID称为全局事务的ID号,表示在同一个复制集群中，每一个服务器上执行任何事务它都有唯一的标识符,GTID的作用是将来在复制时避免事务之间产生冲突;    
8.使用二进制日志基于时间点还原数据库
    只需指定二进制日志的起始位置（可指定终止位置）并将其保存到sql文件中,由mysql命令来载入恢复即可;当然直接通过管道送给mysql命令也可：至于是基于位置position来恢复还是基于时间点来恢复,这两种行为都可以;选择时间点来恢复比较直观些,并且跨日志文件恢复时更方便;
        ~]# mysqlbinlog --stop-datetime="2019-03-01 15:27:48" /tmp/mysql-bin.000008 | mysql -u user -p password
    恢复多个二进制日志文件时：
        ~]# mysqlbinlog mysql-bin.[*] | mysql -uroot -p password
    或者将它们导入到一个文件中后恢复:
        ~]# mysqlbinlog mysql-bin.000001 > /tmp/a.sql
        ~]# mysqlbinlog mysql-bin.000002 >> /tmp/a.sql
        ~]# mysql -u root -p password -e "source /tmp/a.sql"
