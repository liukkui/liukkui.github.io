1.rook项目和组件说明

–rook-ceph部署逻辑架构图

1.Rook是什么?
    官网：https://rook.io
    1.为了充分利用k8s的优势，我们可以把ceph集群部署在k8s之上（生产环境建议单独部署）；而Ceph集群是有状态应用，如果使用StatefulSet控制器部署则需要复杂的逻辑实现，而rook则是使用Operator控制器实现的；
    2.rook is Storage Operators for Kubernetes
        官方说rook是Storage Operators for Kubernetes，它将存储软件转变为自我管理、自我扩展和自我修复的存储服务，它通过自动化部署、引导、配置、置备、扩展、升级、迁移、灾难恢复、监控和资源管理来实现此目的；
    3.Rook使用底层云本机容器管理、调度和编排平台提供的工具来实现它自身的功能；
    4.Rook目前不仅支持Ceph，还支持NFS、Minio Object Store和CockroachDB等存储；

2.rook和k8s的集成机构和rook oprerator
    1.如上图1示，它说明了Ceph Rook如何与Kubernetes集成；
        随着Rook在Kubernetes集群中运行，Kubernetes应用程序可以挂载由Rook管理的块设备和文件系统，或者可以使用S3/Swift API提供对象存储。Rook oprerator自动配置存储组件并监控群集，以确保存储处于可用和健康状态；
    2.rook oprerator:生成很多创建、监控ceph的容器
        1.Rook oprerator是一个简单的容器，具有引导和监视存储集群所需的全部功能。oprerator将启动并监控ceph monitor pods和OSDs的守护进程，它提供基本的RADOS存储。oprerator通过初始化运行服务所需的pod和其他组件来管理池，对象存储（S3/Swift）和文件系统的CRD;
        2.oprerator将监视存储后台驻留程序以确保群集正常运行.
            Ceph mons将在必要时启动或故障转移，并在群集增长或缩小时进行其他调整；oprerator还将监视api服务请求的所需状态更改并应用更改；
        3.Rook oprerator还创建了Rook agent.
            这些agent是在每个Kubernetes节点上部署的pod。每个agent都配置一个Flexvolume插件，该插件与Kubernetes的volume controller集成在一起，负责处理节点上所需的所有存储操作，例如附加网络存储设备，安装卷和格式化文件系统；
        4.该rook容器包括所有必需的Ceph守护进程和工具来管理和存储所有数据-数据路径没有变化.
            因为rook并没有试图与Ceph保持完全的忠诚度。许多Ceph概念（如placement groups和crush maps）都是隐藏的，因此您无需担心它们。 相反，Rook为管理员创建了一个简化的用户体验，包括物理资源，池，卷，文件系统和buckets.同时，可以在需要时使用Ceph工具应用高级配置.

Ceph集群性能规划说明：
    对于Ceph集群来说，CPU一般不是问题，主要就是内存和网络。OSD之间的网络最好是10GbE+；内存会消耗多少呢？以ROOK为例，默认配置部署完成后（假设14块磁盘，每块4TB），会启动：
        14个ceph-osd、3个ceph-mon、2个ceph-mds、1个ceph-rgw、1个ceph-mgr；
    对于OSD进程来说，每个daemon每TB需要1GB，那么在这个上下文里：
        * 每个OSD需要4GB内存，14个OSD就需要56GB内存；
        * 3个ceph-mon就需要6GB内存；
        * 2个mds需要4GB内存
    但是，这是官方最小的内存推荐，在生产环境下建议：
        * 每个OSD每TB配置2GB内存，因为要考虑recovery，这个是很耗费内存的，于是上述配置需要112GB内存；
        * 每个ceph-mon配置4GB内存；
        * 每个ceph-mds需要配置10G内存
1.k8s集群准备和ceph节点规划
    1.k8s集群
        k8s集群使用二进制或者kubeadm部署都可以，但为了满足ceph高可用要求k8s必须至少有三个节点可用，此处部署使用的是ansible二进制部署的8节点k8s集群，其中:3master，5node，并且将node03、node04、node05作为ceph cluster集群节点；
    2.k8s上ceph的专用节点规划
        使用rook将ceph部署在k8s上时，ceph的很多进程如mon、mgr、ceph-osd等都是以pod形式提供的，但是在部署时它们可能会随机的运行在k8s的任何一个node节点上，因此为了使得使用rook部署的ceph时的所有以pod形式提供服务的进程只运行在特定的某些k8s-node之上，应该如下规划：
            第一种方案是：
                准备特定的k8s-node（准备足够的磁盘osd），将这些node打上污点taints，然后在使用rook部署时容忍此污点即可，就可以达到这些node只运行ceph相关的pod;
            第二种方案是：
                将这些node打上标签，部署ceph时让osd选择拥有此标签的node，但其他业务的pod也可以调度到这些node上；
        备注：生产环境下建议使用标签，将所有ceph相关的都统一调度到专用于Ceph的k8s节点上方便管理.
2.rook对后端存储磁盘的要求和rook使用存储方式
    1.前文说过Ceph的OSD设备在使用时有FileStore和BliueStore两种存储引擎，FileStore是使用文件系统的目录模拟成磁盘，BlueStore要求使用裸设备；
    2.rook使用存储方式
        rook默认使用所有节点的所有资源，operator自动在所有节点上启动OSD设备，Rook会用如下标准监控并发现可用设备：设备没有分区和设备没有格式化的文件系统，Rook不会使用不满足以上标准的设备。另外也可以通过修改配置文件，指定哪些节点或者设备会被使用(本文就是自定义节点和节点上的磁盘方式创建的Ceph集群)
3.本文之前安装的是v0.9，在v1.3新版本中安装中还是有些区别的：
    1.v1.3开始支持使用磁盘里没有格式化文件系统的分区用作OSD，不过没啥意义，因为基本上都是使用一整块完整的裸设备硬盘当做OSD的。
    2.部署时v1.3需要先执行common.yaml且全局只有创建了一个名称空间rook-ceph，还增加了其他项.

3.部署(以rook:v0.9版本为例，rook新版本安装步骤和清单内容可能会有所不同)

1.在k8s中Operator控制器通常是用来部署、管理有状态应用的，而有状态应用会表现为很多pod组成的一个集群；而Operator控制器通常也是一个pod，这个pod中有很多自定义资源（CRD）；所以rook部署ceph分为两步：
    1.先部署rook的Operator
    2.再借助于rookOperator中引入的自定义资源（CRD）去部署自定义的集群格式了，也就是ceph cluster；
2.前期准备工作：
    1.确保所有节点都安装了lvm2
        因为ceph也是通过关键lvm来格式化硬盘当做OSD使用的，所以需要所有k8s节点都安装lvm2，好在Centos、Ubuntu基本上都自动安装了.
        ~]# rpm -q lvm2
    2.确保内核编译有RBD模块
        ~]# modprobe rbd
        ~]# lsmod | grep rbd     
            rbd                    83640  0 
            libceph               306625  1 rbd
    3.在虚拟机上为node03、node04、node05添加两块新磁盘：sdb和sdc
        为node03、node04、node05各添加两块磁盘共计6块磁盘作为OSD盘，添加完成后扫描磁盘，确保主机能够正常识别到：
        1.扫描SCSI总线并添加SCSI设备
            for host in $(ls /sys/class/scsi_host) ; do echo "- - -" > /sys/class/scsi_host/$host/scan; done
        2.重新扫描SCSI总线
            for scsi_device in $(ls /sys/class/scsi_device/); do echo 1 > /sys/class/scsi_device/$scsi_device/device/rescan; done
        注：
            1.通过命令lsblk查看已添加的磁盘，能够看到sdb和sdc说明添加成功
            2.如果是将有数据的磁盘当做OSD使用，先使用下面命令将磁盘清理干净
            dd if=/dev/zero of=/dev/sdb bs=1M count=100 oflag=direct,dsync
    4.给用于安装Ceph Cluster的3个node节点打标签(非必要设置，取决于架构的规划)
        下文中在搭建中没有此步骤，因为只需要ceph-osd的Pod运行在node03、node04、node05上即可，其他进程的Pod运行哪个节点相对不是很重要，如果实在想要把所有ceph的pod都运行在特定的nodes上，可以参考下面通过打label，使用nodeAffinity的方式写法：
        1.先给node03、node04、node05打上额外的标签:app.storage=rook-ceph
            ~]# kubectl label node k8s-node03.cre.io app.storage=rook-ceph
            ~]# kubectl label node k8s-node04.cre.io app.storage=rook-ceph
            ~]# kubectl label node k8s-node05.cre.io app.storage=rook-ceph
            ~]# kubectl get nodes --show-labels
                NAME               STATUS  ROLES    AGE    VERSION   LABELS
                ...
                k8s-node03.cre.io   Ready  <none>  5d17h  V1.12.2 beta.kubernets.io/arch=amd64,beta.kubernetes.io/os=linux,kubernetes.io/hostname=k8s-node01.cre.io app.storage=rook-ceph
                注意这里的hostname名称下文一定要与此处相同，app.storage=rook-ceph在第2步骤使用
        2.将Mon、mgr等也部署在node03、node04、node05(非必须)
            ~]# vim cluster.yaml
                ...
                placement:
                  all:
                    nodeAffinity:
                      requiredDuringSchedulingIgnoredDuringExecution:
                        nodeSelectorTerms:
                        - matchExpressions:
                          - key: app.storage
                            operator: In
                            values:
                            - rook-ceph
            注：通过nodeAffinity进行标签匹配即可，taints的方式是通过先打污点再通过tolerations是否容忍污点定义的，逻辑是类似的。
3.其他注意事项：
    删除Ceph集群或者重建集群需要把之前Ceph组件节点上的集群的配置信息目录/var/lib/rook删除，不然启动monitor会失败
        kubectl delete -f cluster.yaml
        kubectl delete -f operator.yaml
        kubectl delete -f  common.yaml(v1.3版本需要把这个也删除)
        在所有节点删除目录/var/lib/rook
        ~]# vim clean-rook-dir.sh
            hosts=(
              k8s-node03
              k8s-node04
              k8s-node05
            )
            for host in ${hosts[@]} ; do
              ssh $host "rm -rf /var/lib/rook/*"
            done

        default             Active    5d18h
        rook-ceph-system    Active     11s 
        注：
            所有rook相关的pod都在此rook-ceph-system名称空间中，而和ceph cluster相关的pod都在rook-ceph名称空间中，注意rook和ceph cluster的pod是不在同一个名称空间的!
    ~]# kubectl get pods -n rook-ceph-system
        NAME                               READY  STATUS    RESTARTS   AGE
        rook-ceph-operator-b9968cfb99-2c78k 1/1   Running   0         9m3s 
        rook-ceph-agent-429xs               1/1   Running   0         7m48s
        ...    
        rook-discover-597sk                 1/1   Running   0         7m48s
        ...
    注：
        1.部署rook-ceph-operator过程中，会触发以DaemonSet的方式在集群部署Agent和Discoverpods。operator会在集群内的每个主机创建两个pod:rook-discover,rook-ceph-agent；其中agent是管理ceph的代理程序、discover是发现节点上可用的OSD程序；
        2.确保rook-discover-*和rook-ceph-agent-*都是Running的，如果不是Running可使用describe命令查看该pod，不过一般情况下都是镜像下载较慢报错导致的稍等即可，其他错误视describe时详细的报错信息解决.
        3.rook-discover-*和rook-ceph-agent-*的pod数量和k8s节点的数量保持一致，因为它是以DaemonSet方式部署的.

3.2.部署Ceph Cluster(根据生产环境自定义Ceph集群)

部署前清单修改说明：
    项目内的/cluster/examples/kubernetes/ceph/cluster.yaml的清单是需要根据实际生产环境下的规划进行修改的，最重要的就是自定义只在哪些nodes上部署ceph、这些nodes上的哪些磁盘当做OSD使用等等，下面解释几个重要段如何进行自定义：
~]# vim /cluster/examples/kubernetes/ceph/cluster.yaml
    ...
    ...清单内容省略，详细、重要且需要根据实际规划的字段如下...
    ...
1.通用字段说明：
    1.image: ceph/ceph:v13.2.4     #镜像版本，v13代表mimic版, v14是N版
    2.dataDirHostPath: /var/lib/rook 
        把当前目录当做ceph-osd使用的FileStore的后端存储，如果要使用FileStore存储引擎则保留该项；
    3.mon: 
        count: 3     #mon的数量必须是奇数个
    4.dashboard:     #rook部署的ceph集群会自动启用ssl协议的面板
    5.network:       #定义pod是否共享该所在节点的网络，默认是flase？
2.OSD所在节点的可自定义调度字段说明：
    前文说过rook部署的ceph集群，它的mon进程、rgw进程、mgr进程、ceph-osd进程都是以pod形式提供的，而OSD在前文规划中只由node03、node04、node05上的6块硬盘提供，所以ceph-osd进程的pod只能运行在此三个节点上，下面这段配置就是配置哪些节点当做OSD设备节点以及这些节点上的哪些磁盘设备（6块磁盘）可以当做OSD使用，下面是主要修改的字段：
    storage: # cluster level storage configuration and selection
      useAllNodes: true #是否使用k8s所有节点，规划的只是特定节点所以改成flase
      useAllDevices: flase  #是否使用每个节点上的所有磁盘设备，系统预留改成flase
      #deviceFilter: 
        当useAllDevices为flase时可以写正则表达式过滤出能当OSD用的磁盘
      config:
        #metadataDevice: "md0"  #对性能有着直接的影响
        #databaseSizeMB: "1024"   #使用bluestore时，rocksdb数据库占用大小
        #journalSizeMB: "1024"    #使用bluestore时，rocksdb.wal占用大小
        #osdsPerDevice: "1" 
        #encryptedDevice: "true" 
    #    nodes:        
    #    - name: "172.17.4.201"
    #      devices: # specific devices to use for storage can be specified for each node
    #      - name: "sdb"
    #      - name: "nvme01" # multiple osds can be created on high performance devices
    #        config:
    #          osdsPerDevice: "5"
    #      - name: "/dev/disk/by-id/ata-ST4000DM004-XXXX" # devices can be specified using full udev paths
    #      config: # configuration can be specified at the node level which overrides the cluster level config
    #        storeType: filestore
    #    - name: "172.17.4.301"
    #      deviceFilter: "^sd."
    注：
        nodes字段是指定安装ceph的node名称以及该node上可作为OSD使用的device名称，上面是官方的多示例写法。
3.Mon、Mgr、rgw、mds进程的pod的自定义调度机制说明：
    上面第2条的字段只是定义OSD必须所在的节点，而placement字段自定义的是除了ceph-osd进程pod之外的其他ceph中的进程Mon、Mgr、rgw、mds它们的Pod的调度机制，即是否将除了OSD之外的其他进程pod也部署在node03、node04、node05上，如果需要自定义的话，需要在前期准备时对node03、node04、node05打上label或者taints，然后再在下面字段中使用nodeAffinity或者tolerations选择都可以实现自由调度。
        #  placement:
        #    all:
        #      nodeAffinity:
        #        requiredDuringSchedulingIgnoredDuringExecution:
        #          nodeSelectorTerms:
        #          - matchExpressions:
        #            - key: role
        #              operator: In
        #              values:
        #              - storage-node
        #      podAffinity:
        #      podAntiAffinity:
        #      topologySpreadConstraints:
        #      tolerations:
        #      - key: storage-node
        #        operator: Exists
    注：下文中不会修改此部分字段，如有需要参考上文前期准备的第4部分操作.

1.按照前期架构规划自定义的最终cluster.yaml清单如下
    ~]# vim cluster.yaml
    ...
    ...前面都不更改...
    ...
    storage: 
      useAllNodes: flase
      useAllDevices: flase
      #deviceFilter: 
      config:
        #metadataDevice: "md0" 
        #databaseSizeMB: "1024"  
        #journalSizeMB: "1024"   
        #osdsPerDevice: "1" 
        #encryptedDevice: "true" 
      nodes:
      - name: "k8s-node03.cre.io"
        deviceFilter: "^sd[^a]"
      - name: "k8s-node04.cre.io"
        deviceFilter: "^sd[^a]"
      - name: "k8s-node05.cre.io"
        deviceFilter: "^sd[^a]"
    说明：
        1.将useAllNodes和useAllDevices都改成flase
        2.使用nodes选择用于安装ceph的节点，name:"k8s-node03.cre.io"的名字一定要和上文前期准备的第4步骤中命令kubectl get nodes --show-labels显示的hostname名称保持一致。
        3.规划中3个node节点上都有两块磁盘（sdb和sdc）当做OSD使用，所以在deviceFilter中使用正则表达式匹配过滤掉系统盘sda即可，也可以下面这种方式代替：
            - name: "k8s-node03.cre.io"
              devices:
              - name: "sdb"
              - name: "sdc"

5.rook提供RBD服务给k8s

rook官方启用rbd说明

k8s使用Rook部署的ceph集群提供的RBD服务和ceph-deploy、ceph-ansible部署提供的RBD服务在使用上是有不同逻辑的：
    1.ceph-deploy和ceph-ansible部署的ceph集群需要管理员使用rbd命令手动创建存储池、启用rbd、初始化、以PV&PVC或者SC方式提供给Pod使用；
    2.而Rook部署的ceph集群中，它的所有块存储的创建、存储池的创建等都能够被标准化为k8s自带的格式，它是借助于由rook定义的CRD格式通过写一个yaml清单来完成存储池创建、初始化、启动、创建SC等一系列操作代替手动使用rbd命令；
    所以Rook在提供块存储之前，它要求必须以yaml清单形式先创建StorageClass和存储池.
1.官方storageclass.yaml配置示例
    v0.9版本清单在rook/cluster/examples/kubernetes/ceph/storageclass.yaml
    v1.3版清单示例在rook/cluster/examples/kubernetes/ceph/csi/rbd/storageclass.yaml
        两个版本清单内容不太一样，下面是在其基础上完善的；
    ~]# vim storageclass.yaml
        apiVersion: ceph.rook.io/v1
        kind: CephBlockPool
        metadata:
          name: replicapool                
          namespace: rook-ceph
        spec:
          failureDomain: host
          replicated:
            size: 3 
        ---
        apiVersion: storage.k8s.io/v1
        kind: StorageClass
        metadata:
           name: rook-ceph-block
        provisioner: ceph.rook.io/rbd       #rook提供RBD服务的格式
        parameters:
          blockPool: replicapool
          clusterNamespace: rook-ceph
          fstype: xfs
        reclaimPolicy: Retain 
        注：
            yaml清单中创建了名为replicapool的存储池和名为rook-ceph-block的SC，副本池数量为3、为了数据安全将SC的回收改成Retain.
    ~]# kubectl apply -f storageclass.yaml         #创建
    验证创建的SC和存储池：
        ~]$ kubectl get SC
            NAME              PROVISIONER          AGE
            rook-ceph-block   ceph.rook.io/block   201m
            注：rook-ceph-block这个SC被成功创建
        登录Dashboard可以看到名为replicapool的存储池被成功创建.
2.示例
    1.参考官方文档中以静态方式提供给wordpress的示例，rook项目示例中wordpress.yaml和ysql.yaml定义了一个名为wp-pv-claim的pvc，指定storageClassName为rook-ceph-block，申请的存储空间大小为20Gi.最后一部分创建了一个名为wordpress-persistent-storage的volume，并且指定claimName为pvc的名称，最后将volume挂载到pod的/var/lib/mysql目录下；
    2.创建并成功运行wordpress和mysql后，这2个应用都会创建一个块存储卷，并且挂载到各自的pod中，查看声明的pvc和pv：
    ~]# kubectl get pvc
        NAME            STATUS             VOLUME                 CAPACITY  ACCESS  MODES  STORAGECLASS     AGE
        mysql-pv-claim  Bound pvc-5bfbe28e-0fc9-11e9-b90d-000c291c25f3   20Gi       RWO           rook-ceph-block   32m
        wp-pv-claim     Bound pvc-5f56c6d6-0fc9-11e9-b90d-000c291c25f3   20Gi       RWO           rook-ceph-block   32m
    ~]# kubectl get pv
        NAME                                       CAPACITY   ACCESS MODES   RECLAIM POLICY   STATUS   CLAIM                    STORAGECLASS      REASON   AGE
        pvc-5bfbe28e-0fc9-11e9-b90d-000c291c25f3   20Gi       RWO            Delete           Bound    default/mysql-pv-claim   rook-ceph-block            32m
        pvc-5f56c6d6-0fc9-11e9-b90d-000c291c25f3   20Gi       RWO            Delete           Bound    default/wp-pv-claim      rook-ceph-block            32m
        注：这里的pv会自动创建，当提交了包含StorageClass字段的PVC之后，Kubernetes就会根据这个StorageClass创建出对应的PV，这是用到的是Dynamic Provisioning机制来动态创建pv，PV支持Static静态请求，和动态创建两种方式.
    3.在Ceph集群端验证：
        ~]# kubectl -n rook-ceph exec -it rook-ceph-tools-45c7d099b2-q2d82 bash
        [root@rook-ceph-tools-45c7d099b2-q2d82 /]# rbd info -p replicapool pvc-5bfbe28e-0fc9-11e9-b90d-000c291c25f3
