iptables-NAT表管理

2017-02-01

NAT表：网络防火墙原理及示例

本地通信：两个主机在没有网关或者路由下进行通信，所有的本地通信是通过MAC地址进行通信的
        name-->IP-->MAC地址基于广播机制
        如果有交换机，交换机会查找MAC地址表(是通过源地址学习得到的),
        将数据报文直接发送到连接到交换机的目标主机上
网络通信：其实就是多个本地通信实现的，由多个路由器进行中继转发最后到达目标主机的
    而把linux服务器扮演成路由器的角色，则这台linux服务器充当是网关，就需要打开核心转发功能

FORWARD链

右边是生产环境
左边是研发测试环境：
隧道VPN

实验模拟：用linux服务器当做防火墙，需要先开启核心转发功能
    sysctl -w net.ipv4.ip_forward=1
    主机A上加路由:
    route add -net 192.168.10.10/24 gw 192.168.34.103
    主机B上加路由:
    route add -net 192.168.34.0/24 gw 192.168.10.10
    情景一：客户端确定服务端不确定；源地址确定，目标地址不确定
    情景二：服务端确定，客户端不确定；源地址不确定，目标地址确定

对于情景一的情况：
    客户端确定,服务端不确定；即源地址确定，目标地址不确定
示例1:
    1.先在FORWARD链上默认为拒绝
    如控制客户端只能访问httpd和ping操作，就可以在FORWARD链上添加如下规则
    tcp:80端口限制:
    iptables -A FORWARD 1 -s 192.168.34.0/24 -p tcp --dport 80 -j ACCEPT
    iptables -I FORWARD 2 -d 192.168.34.0/24 -p tcp --sport 80 -j ACCEPT
    icmp的ping限制：
    iptables -I FORWARD 3 -s 192.168.34.0/24 -p icmp --icmp-type 8 -j ACCEPT
    iptables -I FORWARD 4 -d 192.168.34.0/24 -p icmp --icmp-type 0 -j ACCEPT
    默认拒绝策略：
        iptables -I FORWARD 5 -j REJECT
    2.因为之前介绍过state的状态追踪功能：
    就可以简化成这样的规则：
    iptables -I FORWARD 1 -m state --state ESTABLISHED -j ACCEPT
    iptables -A FORWARD 2 -s 192.168.34.0/24 -p tcp --dport 80 -j ACCEPT
    iptables -I FORWARD 3 -s 192.168.34.0/24 -p icmp --icmp-type 8 -j ACCEPT
    iptables -I FORWARD 4 -j REJECT
    3.情景一因为都是内网的服务器，是不能接收外网的请求的
    所以对于ping和tcp只允许从内网出去的，外网是不能进来的，规则如下
    iptables -I FORWARD 1 -m state --state ESTABLISHED -j ACCEPT
    iptables -R FORWARD 2 -s 192.168.34.0/24 -p tcp --dport 80 -m state --state NEW -j ACCEPT
    iptables -I FORWARD 3 -s 192.168.34.0/24 -p icmp --icmp-type 8 -m state --state NEW -j ACCEPT
    iptables -I FORWARD 4 -j REJECT

示例2：控制访问ping,httpd：80,vsftp：21,dns：53
    规则如下：
    iptables -I FORWARD 1 -m state --state ESTABLISHED -j ACCEPT
    iptables -R FORWARD 2 -s 192.168.34.0/24 -p tcp -m multiport --dport 80,21,53 -m state --state NEW -j ACCEPT
    iptables -I FORWARD 3 -s 192.168.34.0/24 -p udp --dport 53 -m state --state NEW -j ACCEPT
    iptables -I FORWARD 4 -s 192.168.34.0/24 -p icmp --icmp-type 8 -m state --state NEW -j ACCEPT
    iptables -I FORWARD 5 -s 192.168.34.0/24 -p tcp -m state --state RELATED -j ACCEPT
    iptables -I FORWARD 6 -j REJECT
在这里需要指定-s 源，不然外网向内网主动发送报文请求也是可以到达内网的！

结论：
1.所以说有时服务器ping不通不代表服务不能访问，而且通过内网ping外网时，在外网服务器
进行抓包时，可以看出源地址是内网IP,非网关IP，目标地址是外网IP地址
[root@centos7 data]# tcpdump -i ens33 -nn icmp
tcpdump: verbose output suppressed, use -v or -vv for full protocol decode
listening on ens33, link-type EN10MB (Ethernet), capture size 262144 bytes
21:10:24.541810 IP 192.168.34.107 > 192.168.10.10: ICMP echo request, id 1764, seq 3, length 64
21:10:24.541877 IP 192.168.10.10 > 192.168.34.107: ICMP echo reply, id 1764, seq 3, length 64
2.用linux防火墙做网关网络防火墙，有一个问题，内网的机器是保护了，但是这台linux服务
是没有防护的，所以还需要在这台网络防火墙上在INPUT和OUTPUT链上加上规则，来防护
这台linux网关网络防火墙！所以当用一台linux服务器做网络防火墙时，INPUT,OUTPUT
FORWARD链是都需要设置规则的！
 
对于情景二的情况：
右边的图则控制：源地址不确定，目标地址确定
示例3：控制允许互联网的主机访问ping,httpd：80,vsftp：21,dns：53
    所以规则应该如下：
    iptables -I FORWARD 1 -p tcp -m state --state ESTABLISHED -j ACCEPT
    iptables -I FORWARD 2 -d 192.168.10.0/24 -p tcp -m multiport --dports 80,53,21 -m state --state NEW -j ACCEPT
    iptables -I FORWARD 3 -d 192.168.10.0/24 -p udp --dport 53 -j ACCEPT
    iptables -I FORWARD 4 -d 192.168.10.0/24 -p icmp --icmp-type 8 -m state --state ESTABLISHED  -j ACCEPT
    iptables -I FORWARD 5 -d 192.168.10.0/24 -m state --state RELATED -j ACCEPT
    iptables -I 6 FORWARD -j REJECT

NAT：


1.NAT规则是在FORWARD的基础上进行安全加固的，隐藏源地址和目标地址，达到保护内网的安全和提供服务的主机安全
2.当然NAT规则还是建立在FORWARD的基础上做地址转换，不能转发做NAT也没意义，所以放行和控制限制还是需要在FORWARD链上去做的
 
NAT: network address translation，地址转换
NAT技术的产生核心原因：隐藏本不需要公开的主机
    1.请求报文的源地址地址转换是人为在NAT上加规则实现的
    2.响应报文中的目标地址转换，是基于连接追踪功能实现的，不需要人为干预
比如内网的机器要访问互联网，就必然留下IP地址或者端口信息，则有些攻击木马通过真正的
    地址来扫描这些内网机器，如果有漏洞，则可能就会被攻击，所以NAT的技术的出现，就
    会隐藏我们内网机器访问互联网时的真正IP地址和端口。
实现隐藏的原理：
    1.将数据报文中的源地址IP，修改为具有很强防护能力的IP地址，比如网关等,这样的话，互联网上的主机看到的源地址就会是网关的地址，即使被攻击也是攻击网关，而网关一般是具有很高的防护能力的，所以就达到了隐藏源地址的目的.
    2.一般来说请求报文经过防火墙时，会拆掉以太网帧首部，看到目标地址不是自己时，会转发出去，但是对于NAT的情况，防火墙会修改数据报文的IP首部的源地址或者目标地址，然后重新封装数据报文，再到达目标主机，这样就会达到保护源主机和目标主机被攻击的风险，在这个过程中，请求报文由手动修改的，响应报文是由NAT连接追踪功能实现的。
NAT的方式：SNAT,DNAT,FullNAT,PortNAT
NAT规则可加的链：
    支持PREROUTING INPUT OUTPUT POSTROUTING,但是一般只在PREROUTING和
    POSTROUTING上做NAT规则，INPUT和OUTPUT只有在特殊情况下才做NAT。

那么在netfilter上是如何实现的？
SNAT的实现
数据报文经过防火墙时，如上图示，只有当数据报文进入防火墙时，才知道目标地址不是
自己，还是需要转发，所以不适合在PREROUTING链上做地址转发，如果目标地址不是当前
主机，就要经由FORWARD链进行转发，而FORWARD链本身确不支持地址转换功能，所以只
有在离开本机再一次进行路由选择，路由完之后才扔给网卡发送队列中，所以只能在POSTROUTING链上加地址转换规则
DNAT的实现

SNAT：POSTROUTING源地址转换

原理：
    请求报文发生了源地址修改：人为介入修改
    响应报文经过防火墙的NAT规则时，同连接追踪机制自动修改目标地址
    所以源地址转换适合隐藏服务端
作用：
    1.隐藏内网主机的IP地址，防止被攻击
    2.SNAT可以解决IPV4端口不够用的问题：
        IPV4的地址短缺的问题使得SNAT技术更加流行，因为私网地址使用一个公网地址，
        所以通过SNAT地址转换，报文通过公网IP出去的时候，都会把源地址(私网地址)转
        换成公网地址，才能与公网通信，响应报文回来的时候是发给公网IP的，则通过连接
        追踪功能返回给原来的私网地址。

–实验原理图

SNAT实验：
    192.168.34.0/24网段SNAT到192.168.10.11上，且只限制80和ping操作能进行
    规则如下：
    NAT的POSTROUTING链规则：
        iptables -t nat -I POSTROUTING 1 -s 192.168.34.0/24 ! -d 192.168.34.0/24 -j SNAT --to-source 192.168.10.11
    注意：这里的source是一个固定的地址，如果这个地址是变动的呢？
        那么可能下次这条规则就不生效了！-->
    控制和放行还是需要在FORWARD链上做规则的
        iptables -I FORWARD -s 192.168.34.0/24 -m state --state ESTABLISHED -j ACCEPT
        iptables -I FORWARD 2 -s 192.168.34.0/24 -p tcp --dport 80 -m state --state NEW -j ACCEPT
        iptables -I FORWARD 3 -s 192.168.34.0/24 -p icmp --icmp-type 8 -m state --state NEW -j ACCPET
        iptables -I  FORWARD 4 -j REJECT

验证：通过在192.168.10.10上抓icmp包和http日志也可以看出，确实是做了SNAT地址转换的
    [root@node7-3 data]#tcpdump -i ens33 -nn icmp
    14:56:14.315375 IP 192.168.10.11 > 192.168.10.10: ICMP echo request, id 1373, seq 1, length 64
    14:56:14.315473 IP 192.168.10.10 > 192.168.10.11: ICMP echo reply, id 1373, seq 1, length 64

    [root@node7-3 data]#tail /var/log/httpd/access_log 
    192.168.10.11 - - [29/Dec/2018:15:20:45 +0800] "GET / HTTP/1.1" 200 19 "-" "curl/7.29.0"

MASQUERADE：对SNAT的弥补，地址伪装

只能作用在POSTROUTING链上，对SNAT的缺点弥补
外网地址是动态的就用MASQUERADE,如果是静态的就用SNAT；
因为MASQUERADE会消耗更多的系统资源，能用SNAT就用SNAT，特殊场景才使用MASQUERADE

只能作用在POSTROUTING链上，对SNAT的缺点弥补
比如：
iptables -t nat -I POSTROUTING 1 -s 192.168.34.0/24 ! -d 192.168.34.0/24 -j SNAT --to-source 192.168.10.11
如果SNAT规则上的要访问的外网地址(192.168.10.11)是动态的，规则就失效了
这就要用到MASQUERADE了,如果是静态的就用SNAT；
当然能用SNAT还是要用SNAT的，特殊场景才使用MASQUERADE，因为MASQUERADE会消耗更多的系统资源
实验： 将SNAT的规则修改一下，不指定--to-source 192.168.10.10
规则如下：
    NAT的POSTROUTING链规则：
    iptables -t nat -A POSTROUTING -s 192.168.34.0/24 ! -d 192.168.34.10 -j MASQUERADE
验证：通过在192.168.10.10查看http日志也可以看出，确实是做了SNAT地址转换的，即使没指定source也是通过NAT服务器的随机端口进行访问的，达到地址伪装的目的
    [root@node7-3 data]#tail /var/log/httpd/access_log 
    192.168.10.11 - - [29/Dec/2018:18:07:36 +0800] "GET / HTTP/1.1" 200 19 "-" "curl/7.29.0"

DNAT：PREROUTING和OUTPUT目标地址转换

原理:
    请求报文的目标地址需要修改成真正的IP地址
    响应报文经过防火墙的NAT规则，通过连接追踪机制，自动修改目标地址
作用：
    1.隐藏提供服务的服务器的真实IP地址，防止被服务被劫持，如DNS劫持，http服务等
    2.DNAT也通过PortNAT方式可以解决IPV4地址不够用的问题
缺点：
    防火墙和目标主机必须在一个物理网络中 ，而且目标主机的网关必须指向防火墙的
    IP地址，多以受限于传输范围，可以使用FULLNAT解决必须在同一物理网络的问题
DNAT实验：
    访问192.168.10.10的http服务时，只需要访问防火墙NAT服务器的地址即可
    即把192.168.10.10DNAT到192.168.34.103上，这里只做简单的DNAT不做portnat
    规则如下：
    NAT的POSTROUTING链规则：
        iptables -t nat -R PREROUTING 1 -d 192.168.34.103 -p tcp --dport 80 -j DNAT --to-destination 192.168.10.10
    控制和放行还是需要在FORWARD链上做规则的
        iptables -I FORWARD -s 192.168.34.0/24 -m state --state ESTABLISHED -j ACCEPT
        iptables -I FORWARD 2 -s 192.168.34.0/24 -p tcp --dport 80 -m state --state NEW -j ACCEPT
        iptables -I FORWARD 3 -s 192.168.34.0/24 -p icmp --icmp-type 8 -m state --state NEW -j ACCPET
        iptables -I  FORWARD 4 -j REJECT
验证：在192.168.34.107上获取192.168.10.10的页面信息：curl 192.168.10.10
    因为DNAT是隐藏192.168.10.10的真实地址的，在互联网上只会暴露防火墙的192.168.34.103地址是提供http服务的，真正的地址得以保护，所以这里curl 192.168.34.103即可
    [root@node7-2 data]#curl 192.168.34.103
    haha
    192.168.10.10
    而在192.168.10.10查看http日志
    [root@node7-3 data]#tail /var/log/httpd/access_log 
    192.168.34.107 - - [29/Dec/2018:17:24:20 +0800] "GET / HTTP/1.1" 200 19 "-" "curl/7.29.0"
    192.168.34.107 - - [29/Dec/2018:17:26:42 +0800] "GET / HTTP/1.1" 200 19 "-" "curl/7.29.0"
从实验也可以看出DNAT的规则生效了，达到了隐藏真正提供服务的192.168.10.10地址
    从而保证了服务器被攻击的风险

PortNAT:端口映射


端口映射在docker容器中的实现方式更加方便，直接加-p选项即可,而且源端口和目标端口可以指定

进程与进程的通信，实际上就是端口号之间的通信
1.对于SNAT来说，因为访问是互联网的随机端口，所以都转换即可
2.而对于DNAT的情况来说：
    a.很多情况下，并不是说只要访问防火墙F的地址，都通过DNAT转换成主机A的IPA，
    而是当访问主机A的某个服务特定端口时，才进行地址转换
    b.而访问F的端口和要转换的IPA的端口不一定需要一样
        即IPF：port1-->IPA:port2，port1和port2可以不一样
3.DNAT也可以解决IPV4地址不够用的情况：如下图
那么就可以说地址转换可以发生在网络层，而且借助端口映射也可以发生在传输层首部(端口)

DNAT端口映射实验：如上图
访问192.168.10.10：8080的http服务时，只需要访问防火墙NAT服务器的地址即可
    即把192.168.10.10:8080DNAT到192.168.34.103:80上,在上面的DNAT基础上做修改即可
    规则如下：
    NAT的POSTROUTING链规则：
        iptables -t nat -R PREROUTING 1 -d 192.168.34.103 -p tcp --dport 80 -j DNAT --to-destination 192.168.10.10:8080
    控制和放行还是需要在FORWARD链上做规则的
        iptables -I FORWARD -s 192.168.34.0/24 -m state --state ESTABLISHED -j ACCEPT
        iptables -I FORWARD 2 -s 192.168.34.0/24 -p tcp --dport 80 -m state --state NEW -j ACCEPT
        iptables -I FORWARD 3 -s 192.168.34.0/24 -p icmp --icmp-type 8 -m state --state NEW -j ACCPET
        iptables -I  FORWARD 4 -j REJECT
验证：curl 192.168.34.103:80查看是否能获取到192.168.10.10：8080的页面
    [root@node7-2 data]#curl 192.168.34.103
    haha
    192.168.10.10

从实验也可以看出DNAT的端口映射规则生效了，映射的端口不一样也可以访问到页面

1.这里只模拟同一台服务器上的一个服务的DNAT端口映射情景
2.对于同一台服务上的多个服务DNAT端口映射也是同样道理，加上对应的PORTNAT规则即可
3.对于多台服务器上的多个服务DNAT端口映射同样是这样的，只需要在NAT服务器上添加多个地址，然后将对应地址进行多台服务器的服务映射即可

FullNAT：源地址和目标地址都修改

既是网关又是NAT
fullnat能实现网关和主机不在同一个网段
FullNAT的存在是为了弥补SNAT和DNAT模式下，主机和NAT服务器必须在同一个网段的缺点
    SNAT和DNAT都受限于NAT服务器和客户端都在一个网段
    1.在SNAT中，主机的网关都是需要指向NAT服务器的，不然如果在主机和防火墙之间存在路由器的话，那么出去的请求报文就不一定经由NAT服务器了，也就达不到地址转换的目的
    2.在DNAT中，如果提供服务器的主机和防火墙NAT服务器之间有路由器，那么响应的报文
    也不一定经由NAT服务器，也达不到转换目标地址的目的。
    3.所以FullNAT的存在就弥补了SNAT和DNAT的缺点
    4.主要是为了解决DNAT的必须近距离传输的问题

在说明k8s的存储卷之前，需要说明的是虽然k8s上的volumes和docker中的volumes都是为了保存数据，但还是有些区别的：
    1.docker上因为是在单机上运行容器的，删除容器只要加载本地之前挂载的卷就可以了。
    2.在k8s集群上，如果删除一个Pod,则这个pod的卷数据会保存在之前的这个node节点上了，而scheduler的调度是随机的，有可能不会运行在之前的node上，那么之前保存的数据是无意义，不能利用的.
    3.k8s是分布式集群，如果所有节点都挂载网络文件系统，那么数据就可以跨节点使用了;
    4.通常作为数据卷的存储提供方数据冗余实现方式有：
        1.存储设备做镜像，主备模式
        2.分布式存储
            分布式存储数据冗余方式不会是对存储节点设备做备份实现，而是在软件数据管理级别上对存入的每个数据对象做多个副本实现数据冗余,然后根据调度规则和冗余级别在存储集群的多个节点上将一份原始数据存储多份,而且分布式存储是有状态应用、支持向外扩展;
1.CSI和存储卷挂载到Pod中的逻辑
    1.CSI：Container Storage interface,容器存储接口;
    2.存储卷挂载到Pod中的逻辑：
        1.首先节点挂载多种外部存储系统是靠内核中的文件驱动实现的;而节点上的Pod是共享节点内核的;
        2.其次每个Pod中都有基础容器pause,同一个pod中共享pause容器的UTS(主机名),Network(网络名称空间),IPC(进程间通信，网络协议栈);
        3.所以Pod如果想使用外部存储卷是需要先通过volumes定义pause中定义卷,然后Pod再通过volumeMounts复制并挂载pause定义的卷;
    3.
2.定义存储卷和挂载存储卷
    1.定义存储卷可以使用命令kubectl explain pod.spec.volumes查看
        也就是要用存储卷需要现在pod.spec字段定义挂载卷类型和目录等信息
    2.挂载存储卷可以使用命令kubectl explain pod.spec.containers.volumeMounts
        也就是需要在pod.spec.containers中挂载上去才能用
3.K8S Volumes的种类:如上图示
    K8S内置的存储卷类型可以使用kubectl explain pod.spec.volumes命令查看：
    1.临时存储：emptyDir
    2.本地存储：hostPath和local
    3.网络存储：nfs(NAS类型)、iscsi fc(SAN类型)
    4.持久卷申请存储：PV和PVC
    5.特殊存储：ConfigMap和Secret
    6.分布式存储(也属于网络存储的一种)：Ceph(RBD、cephfs)和glusterfs
    7.云存储

1.临时存储：emptyDir

1.临时存储：
    emptyDir是最基础的Volume类型,它是节点主机上创建的临时目录,特点是：
        1.为容器提供缓存存储空间(支持在节点的内存中切分一部分空间作为缓存来用)
        2.为没有持久存储必要的同一Pod中的两个容器共享数据
        3.临时存储随着Pod的删除自动删除
    Pod删除时emptyDir也就没有了;根据这个特性，emptyDir特别适合Pod中的容器需要临时共享存储空间的场景;
2.详细用法说明：
    kubectl explain pod.spec.volumes.emptyDir
        medium      #默认是disk磁盘，还可以是Memory内存
        sizeLimit   #当medium是memory时，则必须进行限制
3.简单示例：
    volumes:
    - name: test
    #  emptyDir: {}  #如果medium和sizelimit两个字段都不定义，则为磁盘的任意空间
      emptyDir:
        medium: Memeory  #使用内存当空间
        sizeLimit: 200Mi #限制使用的内存空间为200M

2.本地存储：hostPath和local(更适合关系型数据库)

hostPath、local、emptyDir都属于本地存储,它们的特点是:数据保存在集群的特定节点上，并且不能随着应用飘逸，节点宕机时数据即不再可用;但是如果将数据库,特别是关系型数据库mysql部署到K8S上时,出于性能考虑我们会使用物理节点的local本地卷作为mysql的存储卷使用,其他的如网络分布式存储中的Ceph的RBD性能用在是数据库上也只是勉强可以;
1.hostPath类型  #节点级的目录
    ~]# vim myapp-hostpath-volumes.yaml 
        apiVersion: v1
        kind: Pod
        metadata:
          name: myapp
          namespace: volumes-test
        spec:
          containers:
          - name: myapp
            image: ikubernetes/myapp:v1
            volumeMounts:
            - name: website
              mountPath: /usr/share/nginx/html  #2.然后挂载到Pod中
              readOnly: true
          volumes:
          - name: website
            hostPath:
              path: /volumes/myapp         #1.先定义本地存储卷
              type: DirectoryOrCreate      
2：local类型
    loca是k8s1.10版本之后的类型，且是节点级的设备或者节点级目录;

3.网络存储：NFS

    ~]# kubectl explain pods.spec.volumes.nfs

4.持久卷申请存储：PV、PVC以及StorageClass(重点存储卷)

1.K8S为什么要设计出PV和PVC?
    1.比如如果我们想把外部的NFS或者Ceph的RBD当做存储卷使用，我们直接先创建好nfs的共享目录或者RBD镜像然后再挂载也很简单,那为什么还要使用PV和PVC呢？其实K8S设计PV和PVC的原因在于:
        1.第一个原因是：Pod通常是由应用的开发人员维护，而Volume则通常是由存储系统的管理员维护,开发或运维要想获得这些外部存储提供的共享目录，镜像文件名等信息就必须去问管理员,这样会将应用开发人员和系统管理员的职责耦合在一起,在小公司还可以接受,大公司中出于效率和安全性的考虑,就必须解决这个问题;
        2.第二个原因在于：从生产者消费者消费者模型中站在消费者的角度来看,应用层用户在消费存储服务的时候往往不希望、也不需要对底层存储细节知道的太多,它们只需要在使用存储时由K8S提供就行,自己没必要去考虑底层是以何种方式提供存储服务的;
2.而K8S的解决方案就是PV(PersistentVolume)和PVC(PersistentVolumeClaim)：
    1.PV:
        K8S把外部存储系统的所提供的存储能力抽象成K8S上的标准资源PV,PersistentVolume持久化存储卷;在PV中定义了存储服务所有相关的信息,也就是具体使用的存储类型和卷参数;
    2.PVC:
        PVC译为存储声明,是在K8s中一种抽象的存储卷类型,代表了某个具体类型存储的数据卷表达;其设计意图是：存储与应用编排分离，将存储细节抽象出来并实现存储的编排（存储卷）;这样K8s中存储卷对象独立于应用编排而单独存在，在编排层面使应用和存储解耦;
    3.那么PV和PVC是如何解决上面所说的生产者消费者模型的？
        简单来说PV是生产者关注的,PVC是消费者关注的;因为：
        1.首先由于PV中定义的是具体存储类型、存储挂载使用的详细参数;这就是上面所说的底层存储细节,PV是由管理员提供的,相当于生产者,消费者不需要关心PV是如何产生的;
        2.PVC关注的对象是消费者用户,作为消费者只需要在PVC中写明自己需要几个存储空间、每个存储空间的大小、使用哪种类型的存储等信息后,PVC就会自动关联到K8S集群中符合消费者需求定义的PV资源;
    所以对于消费者来说,它不需要关系PV与底层存储间的细节,只需要知道如何使用PVC定义自己所需要的存储资源就行了,这就是PV和PVC存在的意义;
3.PV和PVC的一些属性：
    1.PV是集群级别资源,不属于任何名称空间,由专门的存储运维人员负责管理;而PVC会关联到Pod上所以属于某个名称空间;
    2.PV和PVC总是成对出现的、是一一绑定关系,不存在一个PV被多个PVC绑定或者一个PVC绑定多个PV的情况;而且PVC必须与PV绑定后才能被Pod消费;所以在生产者消费者模型中,Pod消费PVC,PVC消费PV，而PV定义了与底层存储的细节属于生产者;
    3.PV和存储系统之间不是一一对应关系,而是与存储系统中可被单独访问的逻辑单元建立一一对应关系;存储系统中的逻辑单元是指如nfs共享的一个目录、RBD提供的一个image、云存储的一个云盘等;
4.PV的其他说明
    1.PV的回收策略
        PV的回收策略是指:由于Pod产生的数据是存在PV中的,当Pod删除时,PVC也可以删除,为了数据安全与PVC绑定的PV是否也跟着删除？而PV删除有二种策略：
            1.Delete:
                是指PVC删除时PV也跟着一起删除,为了数据安全这种策略肯定是不合适的;
            2.Retain：
                是指PVC删除时,但PV不删除,那么这个PV上的数据会被保存,但是不能再被其他PVC绑定了;如果想彻底删除数据,安全的操作方式是手动删除PV,然后在手动删除外部存储上逻辑单元中的数据;
    2.PV的供给方式：
        PV定义与底层存储系统间的信息,可想而知在大量需要PV的集群中手动创建一个个存储空间不同、访问接口、不同标签的PV对于管理员来说是相当麻烦的,所以PV有静态供给和动态供给两种方式：
        1.静态供给：
            先在存储系统上创建好一个个逻辑单元,然后手动创建PV与之建立对应关系;
        2.动态供给：
            PV动态供给需要K8S上的另一种资源SC:StorageClass实现,当：
            1.如果存储系统上需要事先有逻辑存储单元,就可以根据PVC的需求在底层存储上选择符合条件的逻辑存储单元动态的生成PV;
            2.如果底层存储的逻辑单元不存在,只有存储空间,PVC向SC请求调用存储系统的API存储接口,临时创建需求空间大小的逻辑存储单元,然后在SC内部把此存储单元创建为PV,并与PVC进行binding;
        两种供给方式比较：
            1.动态存储卷和静态存储卷最终的效果都是：Pod->PVC->PV的使用链路,且对象的具体模板定义都是一致的;动态存储卷和静态存储卷区别是：动态卷是插件自动创建PV，而静态卷是集群管理员手动创建PV
            2.提供动态存储卷的优势：
                1.动态卷让K8S实现了PV的自动化生命周期管理,PV的创建、删除都通过Provisioner完成;
                2.自动化创建PV对象，减少了配置复杂度和系统管理员的工作量;
                3.动态卷可以实现PVC对存储的需求容量和Provision出来的PV容量一致,实现存储容量规划最优;
    3.PV的lifecycle状态：
        provisioning: #PV的动态供给
        avaiable      #PV未被任何PVC关联
        bingding      #PV被绑定
        using         #PV被使用
        reclaiming    #PV被回收
5.PVC的其他说明：
    1.存储系统的多路访问模型：
        不管是任何存储它的被访问方式有如下几种,在PVC申请时可以根据需求的访问方式选择合适的后端存储：
        1.单路读写: RW0:ReadWriteOnce      iscsi
        2.多路读写: RWX:ReadWriteMany
        3.多路只读: ROX:ReadOnlyMany
        需要说明的是：PV是与后端存储的逻辑单元做映射的，存储系统是否支持多路读写是需要现在PV中定义,然后PVC中再定义需要的访问方式匹配合适的PV进行绑定,避免出现存储故障;
    2.PV和PVC的删除保护机制:
        当PVC和PV被Pod使用时,在K8S的1.10版本之后即使PVC和PV被删除也不会真的被删除;而是只有Pod被删除时PVC和PV才会被允许删除,这就是PVC和PV的删除保护机制;
6.PV和PVC的详细字段定义说明：
    ~]# kubectl explain persistentVolume.spec
        spec:
          accessModes:[RW0|RWZ|ROX]#定义存储系统的访问模型的字符串列表
          capacity:                 #定义存储容量
            storage                 #单位是Ki Mi Gi等等
          volumeMode: [Fileystem|block]  #使用文件系统或块设备接口的存储
          persistentVolumeReclaimPolicy:[Delete|Retain]   #定义PV的回收策略
          storageClassName:         #定义使用哪种存储类
          mountOptions:             #自定义挂载选项,下面这两项是默认的
    ~]# kubectl explain pvc.spec
        accessModes:[RW0|RWZ|ROX]  #访问权限必须要要绑定的PV权限的子集
        volumeMode:[Fileystem|block] #使用文件系统或块设备接口的存储
        resources                   #资源请求
            requests:               #资源需求
                storage:            #具体的需求存储空间大小
            limit                   #资源上限
        storageClassName     #存储类(PVC和PV必须在同一存储类中)
        selector   #通过特定的标签去选PV,如果不定义则从所有PV中选择合适的PV     
示例：
    ~]# vim pv-test.yaml   #定义PV
        apiVersion: v1
        kind: PersistentVolume
        metadata:
          name: pv-nfs-v1
          labels:
            storagefs: nfs   #该PV的标签
        spec:
        # accessModes: ["ReadWriteOnce","ReadWriteMany","ReadOnlyMany"]
          accessModes:
          - ReadWriteMany
          - ReadWriteOnce
          - ReadOnlyMany
          capacity:
            storage: 1Gi                #定义PV能提供多大存储空间
          volumeMode: Filesystem
          persistentVolumeReclaimPolicy: Retain #一定要定义PV回收策略为Retain
          nfs:
            path: /vols/v2
            server: 10.10.0.29
    ~]# vim PVC-test.yaml  #定义PVC
        apiVersion: v1
        kind: PersistentVolumeClaim
        metadata:
          name: redis-pvc            #定义PVC的名称，以便Pod中引用
          namespace: test-ns
        spec:
            selector: #定义关联到哪个PV，不指定则根据需求空间找相近空间大小的PV
              matchLabels:
                storagefs: nfs2
          accessModes:
          - ReadWriteMany
          volumeMode: Filesystem
          resources:
            requests:
              storage: 500Mi          #根据PV的存储空间定义PVC有多少空间
    ~]# vim pod-test-pvc.yaml
        apiVersion: v1
        kind: Pod
        metadata:
          name: redis
          namespace: test-ns
        spec:
          containers:
          - name: redis
            image: redis:alpine
            ports:
            - name: redis
              containerPort: 6379
            volumeMounts:          #将PVC关联的PV提供的空间给/data目录使用
            - name: redisdata
              mountPath: /data
              readOnly: false
          volumes:
          - name: redisdata
            persistentVolumeClaim:
              claimName: redis-pvc        #挂载PVC

4.1.存储类SC：StorageClass

SC:Storageclass是K8S上的标准资源,它是实现PVC动态创建存储单元PV的基础,SC中定义了如何连接外部存储API管理接口的方式;在使用SC时,PVC只能向同一个SC中的PV请求绑定,不能跨SC;
1.定义SC的详细字段定义说明
    ~]# kubectl explain sc
        apiVersion
        kind
        metadata
        parameters              #对接外部存储时的参数
        reclaimPolicy           #在此SC中的PV回收策略
        provisioner             #指明后端存储设备-----必须项
        volumeBindingMode       #后端存储支持的访问模型(RWX,ROX,RWO)
        allowVolumeExpansion    #后端存储系统是否支持空间拉伸
glusterfs类型的SC示例：
    apiVersion: storage.k8s.io/v1
    kind: StorageClass
    metadata:
      name: slow
    provisioner: kubernetes.io/glusterfs
    parameters:
      resturl: "http://127.0.0.1:8081"
      clusterid: "630372ccdc720a92c681fb928f27b53f"
      restauthenabled: "true"
      restuser: "admin"
      secretNamespace: "default"
      secretName: "heketi-secret"
      gidMin: "40000"
      gidMax: "50000"
      volumetype: "replicate:3"
Ceph RBD类型的SC示例：
    ~]# vim SC-rbd.yaml
        apiVersion: storage.k8s.io/v1
        kind: StorageClass
        metadata:
          name: rbd-dynamic
          annotations:
            storageclass.beta.kubernetes.io/is-default-class: "true"
        provisioner: kubernetes.io/rbd
        parameters:
          monitors: 172.30.0.16:6789,172.30.0.17:6789,172.30.0.18:6789
          adminId: admin        #管理员接入ceph集群使用的账户
          adminSecretName: ceph-admin-secret
          adminSecretNamespace: kube-system
          pool: rbddata         #表示从此存储池中动态创建image
          userId: k8s           #Pod接入ceph集群使用的账户
          userSecretName: ceph-k8s-secret
          userSecretNamespace: default
          imageFormat: "2"      #指定rbd镜像的版本，默认也为2
        reclaimPolicy: Retain   #SC的删除策略一定要为retain
    其中：
        SC中的注解annotations:is-default-class: "true"是指将此SC设为默认存储类，所有PV都从此SC中创建
        reclaimPolicy：Retain 
            默认情况下pvc的回收策略为"Delete"，也就是删除pvc后pv也会删除，pv删除那么也会触发ceph集群中对应的镜像删除，所以在定义存储类的的时候一定要指定创建的pv回收策略为"Retain";

5.特殊存储：ConfigMap和Secret

1.ConfigMap和Secret在K8S中的作用？
    1.我们知道在Docker中应用程序测试环境、生产环境需要使用不同的配置文件时,可以采用两种方式来配置：
        1.传递环境变量的方式
            可以在启动docker容器时通过-e或者enterpoint脚本传递不同的环境变量将容器部署在不同的环境下;
        2.配置文件挂载方式
            提前准备好不同环境下使用的配置文件,然后通过挂载卷方式将配置文件传递到容器中;也就是在启动docker容器时通过-v|HOSTDIR:VOLUMEDIR将配置文件映射到容器中;但是这种方式的缺点是修改本地路径下的原配置文件后,容器不会自动加载新的配置文件;
    2.K8S同样遇到这种问题,为了解决修改配置文件或参数值能够使Pod自动重载配置信息,就将配置文件做成了K8S上的标准资源ConfigMap和Secret;它们都是为Pod中的容器提供配置变更,可以理解为是K8S系统上一组控制器控制的Pod配置中心;
2.ConfigMap和Secret
    1.ConfigMap和Secret都是为Pod容器提供配置信息;这两种存储卷类型其数据是集群的一些对象信息,并不属于某个节点,使用时将对象数据以卷的形式挂载到节点上供应用使用;它两的区别在于：
        ConfigMap用来保存非敏感数据，Secret用来保存敏感数据,如密码、证书私钥等
    2.ConfigMap和Secret是如何将配置传递至Pod中的并且实现自动更新配置信息的？
        1.首先configMap和secret资源是使用KV键值对存储信息的,而我们可以向Pod传递简单的环境变量或者传递配置文件：
            1.如果只是向Pod传递简单的环境变量,在定义configMap时指定Key：环境变量名称和Value：该环境变量的值;
            2.如果是向Pod中传递配置文件,则Key是配置文件名、Value是具体的配置文件中的内容；
        2.创建好上面两种格式的Configmap后,在创建Pod时,传递给Pod的逻辑是：
            1.如果configmap中只定义了简单的环境变量,在Pod是通过将configmap中的key映射给Pod内部对应的环境变量,那么key的值Value也就传递给Pod内部对应的环境变量的值完成传递;
            2.如果configmap定义的是配置文件,在Pod是通过将configmap中的配置文件映射给Pod内的配置文件,那么configmap中的配置文件中的内容也就传递给了Pod内的配置文件;
        如果想变更配置,只需要修改Configmap中的value值对应的环境变量值或文件内容后,Pod会在一段时间内自动传递新的value值,完成配置更新;

5.1.ConfigMap

K8S的configMap最适用的场景是非敏感数配置信息;ConfigMap有基于命令和使用yaml清单两种创建方式,虽然命令行更简单,但是在生产环境中一般用来为Pod引入配置文件,所以会通过yaml清单格式保存在我们的资源配置中心,然后基于配置中心组件实现Pod的滚动更新或者金丝雀更新;
1.两种创建ConfigMap的两种方式
    1.基于命令行
        ~]# kubectl create configmap -h  #基于命令也有两种方式
        --from-literal=key1=value1
            literal是指K/V键值对,可以使用多个键值对定义多个KV,key不能省略不写
        --from-file=[key=]source    
            从文件中读取创建时key可以省略,即文件名当key;
    2.配置清单(比命令行麻烦，但是更易维护)
        ~]# kubectl explain configmap
            data:       #只有data字段，而没有spec字段
2.Pod加载ConfigMap资源配置信息的方式有env和存储卷两种：
    1.env:只是简单的从cm中映射变量给Pod
        ~]# kubectl explain pod.spec.containers.env
            env             #直接手动给变量
            - name          #镜像中所能接收的变量名
              value         #手动给值(一般用valuefrom)
              valuefrom     #从其他位置引用值
                configMapKeyRef     #引用一个configMap的键key方式
                    name            #引用的configmap资源名称
                    key             #引用的configMap资源中的key名称
                    optional  #被引用的configMap资源存在就可以,不存在则报错
                        true|false  #默认是必须存在
                secretKeyRef  #引用一个Secret资源的键key,用于敏感数据传输
            envfrom   #从其他地方加载环境变量
        缺点：
            如果修改configmap中的环境变量的值，pod是不会更新环境变量的，除非删除重建pod，新的环境变量才会生效.
    2.存储卷：相当于是把configmap当存储卷挂载到Pod
        ~]# kubectl explain pod.spec.volumes.confgiMap
            volumes:
            - name           #定义存储卷名称以便挂载时使用
              configMap:
                name         #configMap资源的名称
                items        #要引用configmap中的那些键映射为本地的文件
                - key        #configmap中的键名
                  mode #映射的文件权限时多少，没定义则使用外部的defaultMode
                  path  #映射到pod中叫什么文件名(可以和key一样，也可以随意更改)
                        必须是相对路径，而且是相对于挂载点而言
                defaultMode  #映射到本地的文件权限(默认是0644读写权限)
                optional     #引用的configMap资源或者引用的键不存在是否报错
3.示例：基于命令行创建configmap并通过env方式加载cm的变量给Pod
    ~]# kubectl create configmap filebeat-cfg -n config-ns --from-literal=redis_host="filebeat.defalut.svc.cluster.local" --from-literal=log_level="Info"
        #先创建名为filebeat-cfg的cm,并设定两个变量key和他的值,我们要把redis_host和log_level这两个key的值赋予下面的filebeat中的两个变量;
    ~]# vim filebeat-test.yaml
        apiVersion: v1
        kind: Pod
        metadata:
          name: filebeat
          namespace: config-ns   #Pod需要和调用的configmap在同一名称空间中
        spec:
          containers:
          - name: filebeat
            image: ikubernetes/filebeat:5.6.5-alpine
            env:
            - name: REDIS_HOST
              valueFrom:
                configMapKeyRef:
                  name: filebeat-cfg 
                  key: redis_host   #将cm中第一个key映射给REDIS_HOST
            - name: LOG_LEVEL
              valueFrom:
                configMapKeyRef:
                  name: filebeat-cfg  
                  key: log_level       #将cm中第二个key映射给LOG_LEVEL
        其中：
            cm中的第一个key:redis_host映射给filebeat中的变量REDIS_HOST;
            cm中的第二个key:log_level映射给filebeat中的变量LOG_LEVEL;
4.示例：基于命令行创建configmap并通过挂载存储卷的方式引入cm资源给Pod
    ~]# kubectl create cm nginx-cfg --from-file=nginx-server1.conf=./server1.conf --from-file=server-zhihui.conf=./server2.conf -n config-ns
        #从事先准备好的两个nginx配置文件中读取转为configmap资源,然后传递给Pod;
    ~]# vim nginx.yaml
        apiVersion: v1
        kind: Pod
        metadata:
          name: nginx
          namespace: config-ns  #Pod需要和调用的configmap在同一名称空间中
        spec:
          containers:
          - name: nginx
            image: ikubernetes/myapp:v1
            volumeMounts:
            - name: config
              mountPath: /etc/nginx/conf.d     #挂载的目录
          volumes:
          - name: config
            configMap:
              name: nginx-cfg            
              items:
              - key: nginx-server1.conf          #本地文件名
                path: server1.conf      #映射到pod中的文件名
              - key: server-zhihui.conf          #本地文件名
                path: server2.conf      #映射到pod中的文件名
    既然是把configmap当存储卷使用,那么containers中也会有volumeMounts来实现挂载,volumes中则是将两个配置文件server1.conf和server2.conf传递给Pod,并放在/etc/nginx/conf.d路径下;而且通过手动修改configmap的值,Pod内的文件内容也会自动更新;

5.2.Secret

1.ConfigMap最适用的场景是非敏感数配置信息;而Secret资源最适用于存储和管理一些敏感数据比如密码、token、密钥等敏感信息;比如：
    当我们把mysql运行在Pod中时,我们会把root的密码转为Secret资源然后传递给Pod,而不是使用configmap保存root密码的原因是：只要用户有cm所在名称空间的权限就可以查看cm中的详细内容,这是非常不安全的;
2.Secret的使用逻辑和Configmap没有区别,唯一不同的地方在于Secret的数据是被base64编码后存放的,不过base64并不是真正意义上的加密,可以通过base64进行解码得到原始内容;而在K8S上Secret有存在三种类型：
    ~]# kubectl create secret -h
        docker-registry
        generic
        tls
    1.tls
        tls类型时专用于把ssl的tls中的x509格式的证书和私钥打包到一个secret中,由于这个证书和私钥本身就是base64编码的,在打包到secret中会使用特殊的逻辑进行"打包";而且它不管原来的证书和私钥文件名是什么,都会通通被在打包成secret时都叫tls.key和tls.crt;
    2.generic：(Opaque、通用类型)
        generic类型是非上面说到的tls类型但也属于敏感信息,如redis密码、mysql密码,这种普通的密钥信息;任何非tls类型的敏感信息都归类为generic类型;
    3.docker-registry
        在众多的generic类型的信息当中有特殊作用的信息是docker-registry;因为拉取私有镜像仓库的镜像时是需要账号密码认证的,而在K8S中拉取镜像是由kubelet完成的,所以需要kubelet认证到私有镜像仓库上实现自动拉取,所以docker-registry类型的secret就是专用于让kubelet拉取镜像时能够自动加载secret认证到私有仓库上;docker-registry类型的secret可以被Pod通过下面两种方式去调用:
            1.~]# kubectl explain pod.spec.imagePullSecrets
                imagePullSecrets <[]Object>
                1.可以看到imagePullSecrets是一个secret对象列表,一个secret就是一组账号密码,如果有多个secret时,会按照列表中自上而下的顺序使用secret依次进行检索认证登录;
                2.但是这种调用是在具体的Pod模板中定义的,其他Pod模板也需要拉取镜像时还需要再定义一次,显然是不合理的,所以我们会使用第二种serviceAccountName的方式进行secret的调用;
            2.~]# kubectl explain pod.spec.serviceAccountName
                serviceAccountName <string>
                1.serviceAccount是指K8S的标准资源服务账号,因为由于K8S中的每个Pod都需要某个用户的身份运行其内部的应用程序（uid和gid决定）,同时每个Pod作为客户端都需要认证到外部私有服务时有一个用户名,所以才存在serviceAccount这类资源;
                2.此处只需要填上我们创建好的包含认证到私有镜像仓库的账号密码的SA的Name即可,它会自动根据serviceAccount内部的secrets字段去调用secret;
                3.这种调用方式的好处在于每个Pod模板都只需要写上serviceAccount的Name,即使需要修改认证的账号密码信息也只需要修改serviceAccount调用的我们可以使用serviceAccountName的serviceAccount内部调用资源即可;
            这两种调用方式无论是哪一种都需要用到docker-registry类型的secret,需要管理员先定义出来;
3.Pod加载Scret资源配置信息的方式通用也有env和存储卷两种：
    1.通过env环境变量引用
        ~]# kubectl explain pod.spec.containers.env
            env:
            - name                  #Pod镜像中的存在的变量名
              valueFrom:    #valueFrom有4种引用方式
                configMapKeyRef   #上面调用configmap资源的方式
                filedRef
                resourceFieldRef
                secretKeyRef:       #表示从secret类型资源引用键,此处用到的
                  name:             #引用的Secret资源的名称
                  key               #引用的secret资源中的哪个key
                  optional:         #secret资源或者引用的key不存在时报错
                    true|false
    2.通过存储卷引用Secret资源
        ~]# kubectl explain pod.spec.volumes.secret
        volumes:
        - name:                 #定义存储卷名称以便挂载时使用
          secret:
            secretName          #引用的secret资源名称
              items:            #引用的多个键对象
              - key             #引用的secret资源中的键名
                mode:    #文件权限，私密数据建议600权限，不指定则使用外部默认的
                path:    #映射到pod中叫什么文件名(可以和key一样，也可以随意更改)
          defaultMode           #引用资源的默认权限，0644
          optional
    注意：
        tls类型的证书和私钥一般要用存储卷方式进行引用，不然value的值太长的;
4.三种类型创建的基本语法：
    ~]# kubectl create generic -h      #generic创建语法
        kubectl create generic NAME [--type=string] [--from-file=[key=]source] [--from-literal=key1-value1] [--dry-run] [options]
    ~]# kubectl create tls -h           #tls类型语法
        kubectl create tls NAME --cert=path/to/cert/file --key=path/to/file [--dry-run] [options]
    ~]# kubectl create docker-registry -h    #docker-registry类型语法
        kubectl create docker-registry NAME --docker-username=user --docker-password=password --docker-email=email [--docker-server=string] [--from-literal-key1-value1] [--dry-run] [options]
5.1.generic类型使用env环境变量方式引用secret示例：mysql密码加密传输
    ~]# kubectl create generic mysql-root-passwd -n test-ns --from-literal=password=root123
        secret/mysql-root-passwd created
    ~]# kubectl get secret -n test-ns
        NAME                TYPE      DATA   AGE
        mysql-root-passwd   Opaque     1      30s
        注：Opaque是不透明的、模糊之意,所以generic类型也称为是通用类型的
    ~]# kubectl get mysql-root-passwd -n test-ns -o yaml #详细信息
        apiVersion: v1
        data:
          password: cm9vdDEyMwo=  #root123被base64编码后的结果
        kind: Secret
        metadata:
          creationTimestamp: "2019-03-02T21:10:30Z"
          name: mysql-root-passwd
          namespace: test-ns
          ...
        type: Opaque
        注：通过echo cm9vdDEyMwo=|base64 -d解码就是root123,所以它不是严格的加密
    ~]# vim mysql-test.yaml
        apiVersion: v1
        kind: Pod
        metadata:
          name: mysql
          namespace: test-ns
        spec:
          containers:
          - name: mysql
            image: mysql:5.6
            env:
            - name: MYSQL_ROOT_PASSWORD
              valueFrom: 
                secretKeyRef:      #引用的secret的信息
                  name: mysql-root-passwd    #调用的secret名称
                  key: password    #引用的secret中的键,上面创建的
    ~]# kubectl apply -f mysql-test.yaml
    ~]# kubectl exec -it mysql-test -n test-ns /bin/sh
        # 
        # mysql -proot123
        # printenv
        我们通过连接到此Pod中看到需要使用之前设置的密码才能够连接到mysql;使用printenv也可以看到我们传递进去的环境变量;
5.2.tls类型使用存储卷方式调用secret示例:给nginx配置tls证书
    ~]# kubectl create secret tls nginx-cert --cert=./nginx-server.cert --key=./nginx-server.key -n test-n
        secret/nginx-cert created
    ~]# kubectl get secrets -n test-ns
        NAME                TYPE            DATA   AGE
        mysql-root-passwd   Opaque          1      10m
        nginx-cert         kubernets.io/tls 2      30s
    ~]# kubectl get secret nginx-cert -n test-n -o yaml
        apiVersion: v1
        data:
          tls.crt: *************  #经过base64编码后的信息
          tls.key: *************  #经过base64编码后的信息
        kind: Secret
        metadata:
          creationTimestamp: "2019-03-02T21:20:30Z"
          name: nginx-cert
          namespace: test-ns
          ...
    ~]# vim nginx.yaml
        apiVersion: v1
            kind: Pod
            metadata:
              name: nginx
              namespace: test-ns
            spec:
              containers:
              - name: nginx
                image: nginx:1.14-alpine    
            volumeMounts:
            - name: tls
              mountPath: /etc/nginx/ssl
          volumes:
          - name: tls
            secret:
            - secretName: nginx-cert
              items:
              - key: nginx.crt              #原来的证书名
                path: nginx-server.crt #映射到pod中的证书名
              - key: nginx.key              #原来的私钥名
                path: nginx-server.key  #映射到pod中的私钥名    
                mode: 0600
